## DX â€” the developer experience (new this round)

This is the half of CritterWatch an operator never sees: standing it up. CritterMart is the first
production-shaped install of the console, so these are first-contact notes.

### DX-1. The version coupling is the headline developer pain
**Target:** beta Â· **Effort:** M (a preflight check) Â· **Evidence:** `Directory.Packages.props`, the held-at-6.12.0 comment block

- **What a developer hits:** there is exactly one `Wolverine.Marten` per process, shared by the console
  and every monitored service. So **every `WolverineFx.*` pin in the whole solution must equal the
  version CritterWatch was compiled against**, or the console throws a startup
  `TypeLoadException` in `EventSubscriptionAgentFamily.TryRebuildRegisteredProjectionAsync`. alpha.3 is
  built against WolverineFx 6.12.0; a developer on the happy path with a *current* Wolverine (6.13.x via
  Dependabot) gets a cryptic crash with no hint that a version skew caused it. We diagnosed this the
  hard way and now pin the entire Wolverine family to 6.12.0 with a paragraph of guard comments and a
  standing "do not merge the Dependabot bump" rule.
- **Why it matters:** this is the single biggest piece of friction in adopting CritterWatch, and it's
  invisible until runtime. The error names a Wolverine projection-distribution internal, not "you have
  the wrong Wolverine for this CritterWatch."
- **The question:** could the console run a **preflight version check** at startup and fail loud and
  legible â€” *"CritterWatch 1.0.0-alpha.3 was built against WolverineFx 6.12.0; this process loaded
  6.13.1. Pin WolverineFx.* to 6.12.0."*? Even a logged warning would turn an hour of stack-trace
  archaeology into a one-line fix. Bonus: publish the CritterWatchâ†’Wolverine compatibility row somewhere
  a human (or Dependabot) can read it, so the coupling is documented, not folklore.

### DX-2. Licensing: the dev-tier fallback silently masks a real key
**Target:** beta Â· **Effort:** S Â· **Evidence:** `deep/dx-settings-license.png`, console `Program.cs` lines 4-10

- **What a developer hits:** run the console host in `ASPNETCORE_ENVIRONMENT=Development` (the default
  for `dotnet run`) and it substitutes a **"Development" tier that never expires and never reads your
  key**. Your real trial license looks like it isn't working â€” or worse, looks like it *is* (it shows a
  valid-looking tier) while silently ignoring the key you set. We only get the real Trial tier by (a)
  forcing the console to `Production` in the AppHost **and** (b) calling
  `builder.Configuration.AddUserSecrets(...)` explicitly, because user secrets otherwise only auto-load
  in Development â€” the exact environment that triggers the fallback. That's a catch-22 a first-timer
  won't reason their way out of.
- **The question:** when a `JasperFx:LicenseKey` **is present** but the environment is Development, could
  the License Info panel say so out loud â€” *"A license key was found but the Development-tier fallback is
  active; run as Production to validate it"*? Right now the panel shows `Development / Never` with no clue
  that a real key sits unread three feet away. The silent-success case is more dangerous than a silent
  failure.

### DX-3. The install docs would re-introduce a bug we already fixed
**Target:** beta (docs) Â· **Effort:** S Â· **Evidence:** console `Program.cs` lines 22-35 vs the setup skill

- **What a developer hits:** the published setup guidance (`wolverine-integrations-critterwatch-setup`,
  and our local `critterwatch-install` REFERENCE) uniformly calls `.DisableDeadLetterQueueing()` on the
  console's RabbitMQ transport. Following that verbatim **breaks the console** in a multi-service setup:
  the monitored services declare the shared `critterwatch` queue with the default
  `x-dead-letter-exchange`, the console then re-declares the same queue with *different* args, and
  RabbitMQ rejects the inequivalent redeclare with `PRECONDITION_FAILED` (406). Whichever side starts
  second dies. CritterMart's working console deliberately does the **opposite** of the docs â€” it leaves
  DLQ at the Wolverine default ON, with a comment explaining why.
- **The question:** should the setup skill's single-node example *not* disable DLQ when the console
  shares a broker with monitored services â€” or at least call out that the `critterwatch` queue's
  dead-letter args must be identical on both sides? This is a documentation accuracy gap that actively
  steers new installs into a startup crash. (We're filing the same note upstream against the ai-skills
  repo.)

### DX-4. Every doc layer lags the binary it documents
**Target:** beta (docs cadence) Â· **Effort:** Sâ€“M Â· **Evidence:** the three skill snapshots vs installed alpha.3

- **What a developer hits:** the docs are honest but perpetually one version behind. The upstream setup
  skill snapshots `0.9.5 / Wolverine 6.10.0 / Marten 9.8.0`; our local install playbook is titled
  "0.9.x" and its REFERENCE is "verified against 0.9.1 / Wolverine 6.8.0"; the installed console is
  `1.0.0-alpha.3 / Wolverine 6.12.0`. Because of DX-1, "which Wolverine?" is not a footnote â€” it's the
  load-bearing fact, and it's stale in every doc. A developer can't tell whether a divergence between the
  skill and reality is *their* mistake or *doc drift*.
- **The question:** would a single published compatibility matrix â€” CritterWatch version â†’ exact
  WolverineFx/Marten/JasperFx versions it was built against â€” be the highest-leverage doc you could ship?
  It would anchor DX-1's preflight, DX-3's DLQ note, and every "what do I pin?" question at once.

### DX-5. The immutable-record stepper crash is a *developer* story too
**Target:** beta Â· **Effort:** M Â· **Evidence:** `deep/closure-stocklevel-stepper-crash.png` (107 identical errors)

- **What a developer hits:** the Projection Stepper rehydrates an aggregate by **dynamically
  constructing it**, which needs a parameterless constructor. CritterMart's `StockLevel` is a `sealed`,
  immutable, self-aggregating record â€” *the* modern Critter Stack aggregate idiom the team steers people
  toward (ADR 020). Stepping it throws `Cannot dynamically create an instance of type
  'CritterMart.Inventory.Stock.StockLevel'. Reason: No parameterless constructor defined.` â€” **once per
  event**, 107 times for this stream (re-confirmed live this round; round one saw ~502 on a longer
  stream â€” the spam scales with stream length). The tool fails on precisely the aggregate shape JasperFx
  recommends, and the developer who built the aggregate "correctly" is the one who meets the wall.
- **The question (DX framing):** can the stepper construct via the registered `Create`/serializer path
  the projection itself uses, rather than `Activator.CreateInstance`? Failing that, can it detect the
  no-parameterless-ctor case once and show a first-class *"this aggregate has no parameterless
  constructor and can't be stepped this way"* â€” instead of N copies of a CLR stack trace? (UI-side
  collapse is UX-7 below; this entry is about the construction strategy.)

### DX-6. The MCP surface is unexercised â€” opportunity, not defect
**Target:** v1.x (docs/sample) Â· **Effort:** S to assess Â· **Evidence:** no `CritterWatch.Mcp` / `Wolverine.CritterWatch.Http` reference in the solution

- **What we found:** CritterMart wires neither `CritterWatch.Mcp` (the cross-application MCP server, 15
  read + 21 action tools per the setup skill) nor `Wolverine.CritterWatch.Http`. So we can't review the
  MCP DX from use â€” only from the docs. The mounting story reads clean (`AddCritterWatchMcp()` +
  `MapCritterWatchMcp()` at `/api/mcp`, stateless transport mandatory for RBAC), and the tool catalog is
  thoughtfully capability-gated.
- **The question:** for an event-sourcing teaching app like CritterMart, the MCP server is a compelling
  "ask an agent about your event store" beat. Is there a minimal reference (one console + one service +
  the MCP package) that shows the smallest wiring that lights up `/api/mcp`? We'd happily be a second
  real-world MCP install if there's a 10-minute on-ramp. (Minor doc nit while here: the setup skill says
  `AddCritterWatchMcp()` "registers all ten tool types," but the catalog table lists 11 families / 36
  tools â€” the count reads inconsistent.)

## UI / UX â€” interaction-level (deeper than screenshots)

### UX-1. Accessibility: most of the WCAG gaps live in the shared shell â€” fix once, fix everywhere
**Target:** beta (the shell) / v1.1 (the rest) Â· **Effort:** M Â· **Evidence:** `deep/axe-report.json` (79 rule-hits across 13 surfaces)

- **Observation:** an axe-core WCAG 2.1 A/AA pass on all 13 routes found the **same six violations
  recurring on every surface**, because they live in the shared chrome, not the pages:
  - `aria-required-children` / `aria-required-parent` (critical) â€” the left-nav `el-menu` wraps its
    `role="menuitem"` items in an `el-menu-item-group > ul`, which breaks the ARIA menu role chain (15
    nodes/page).
  - `list` / `listitem` (serious) â€” same malformed list semantics (a `<ul>` whose direct children
    aren't `<li>`).
  - `color-contrast` (serious) â€” the nav group titles, the active item label, and the **`.el-tag--danger`
    / `.el-tag--dark` status chips** fail the 4.5:1 threshold. This spikes where chips are dense:
    Timeline 102 nodes, Services 51.
  - `label` (critical) â€” `el-select`s, switches, and search inputs render without an accessible name
    (Listeners alone: 25 unlabeled controls).
- **Why it matters:** because they're shell-level, a small number of fixes clears the bulk of 79
  rule-hits across *every* page at once. This isn't 13 problems; it's ~5 components.
- **The question:** could the nav be authored as a proper `role="menu" > role="menuitem"` (or native
  `<nav><ul><li>`) structure, the shared `el-select`/switch components get `aria-label`s, and the danger
  chip token be darkened to clear contrast? The last one double-scores â€” see UX-2.

### UX-2. The all-red error wall also fails color contrast
**Target:** beta Â· **Effort:** S Â· **Evidence:** `axe-report.json` (`.el-tag--danger` on nearly every surface)

- **The question:** when you collapse the repeated-error wall (round-one entry 2 / UX-7), can the
  surviving error treatment be chosen to clear 4.5:1 at the same time, so the two fixes land together?

### UX-3. The console doesn't adapt to narrow viewports
