# Retrospective — Research 002: E-Commerce Frontend Stack Survey

## Outcome summary

The session produced `docs/research/ecommerce-frontend-stack.md` (v1.0, ~67 KB) as a single decision-evidence document for the round-one frontend-stack ADR. The document surveys ten landscape axes (rendering models, framework families, meta-frameworks, state management, styling, performance, forms/payments/accessibility, real-time, deployment ergonomics, .NET integration), evaluates seven candidate stacks against CritterMart-specific constraints (A: Vite+React+TanStack Query+Tailwind+shadcn/ui, B: Next.js App Router, C: Blazor United, D: Blazor Server, E: Astro Islands + small interactive island framework, F: SvelteKit, G: HTMX + ASP.NET Core Razor Pages/MVC), names eight decision criteria, surfaces five cross-cutting themes, and lists six open questions for the team to settle before the ADR lands. The document deliberately does not name a winner. Real-time material is flagged as long-road only throughout, per vision doc § "What this deliberately is not." This `docs/retrospectives/research/002-ecommerce-frontend-stack.md` retrospective lands alongside it in the same PR. The `docs/retrospectives/research/` subdirectory was created in this session (it did not exist before; this is the first research-kind retrospective in the project).

The vision doc's "the specific frontend stack is TBD for round one" line becomes unblocked once a follow-on `docs/prompts/decisions/NNN-frontend-stack.md` prompt is authored against this document.

## What worked

- **Reading the orientation list in the order the prompt specified.** Vision doc first anchored the constraint set (six-day timeline, no real-time, no auth, three Wolverine.Http services, Aspire-orchestrated locally). Context map then narrowed the integration surface — particularly the Catalog-has-no-BC-level-integration line, which made clear that the frontend is the place where product data threads into Cart commands. The four selected ADRs (003 RabbitMQ, 004 Aspire, 006 no-BFF, 009 Polecat-deferred) each landed specific constraints into the candidate-evaluation grid. Workshop 001 supplied the round-one slice list against which "what interactive surfaces does this stack need to handle?" can be evaluated. CritterBids' `frontend-stack-research.md` was the highest-value single source for structural and tone precedent. By the time source-discovery began, the criteria were already largely determined.

- **The CritterBids precedent was credit-able as a sibling project, not absorbed wholesale.** CritterBids' frontend research deliberated Vite + React + TanStack Query + Tailwind + shadcn/ui for an architecturally adjacent (but not identical) project. The Research 002 document treats it as one candidate among seven with explicit acknowledgment of the precedent, rather than pre-deciding that CritterMart should match. This matched the prompt's "do not pick a winner" directive while still letting the precedent's research depth carry forward.

- **First-party sources were the dominant cited tier.** Of the 38 cited sources in the table, more than two-thirds are first-party venues (framework docs, framework authors writing on their own blogs, framework-organization release announcements, Microsoft Learn, Stripe documentation, web.dev case studies). The third-party citations (Smashing Magazine for the Rich Harris interview, Visual Studio Magazine for the Conway Blazor Summit write-up, InfoQ for the Qwik and TanStack Start RC news, The New Stack for Carniato JSNation coverage) all met the bar via being recognized engineering venues with named authors. The "AI-content-smell" filter caught roughly four to five would-be sources that had the generic-phrasing / no-examples / formulaic-structure pattern; those went into Appendix A.

- **The real-time-as-long-road flag held in every axis.** SignalR / SSE / WebSocket material appears only in the Real-time axis (clearly labeled LONG-ROAD ONLY) and inside candidate-stack "Long-road preservation" cells. It does not influence any candidate's round-one scoring. This matches the vision doc's explicit "no real-time storefront updates" line and prevents the document from accidentally re-litigating a settled scope decision.

- **The seven candidate stacks followed identical sub-structure.** Every candidate has "What it is, in one sentence" / "Strengths" / "Weaknesses" / "Demo ergonomics" / "Six-day fit" / ".NET-shop audience fit" / "Long-road preservation." The structural uniformity makes the document readable cross-cut (compare all candidates on demo ergonomics) and within-candidate (read one candidate end-to-end). The prompt suggested this structure; holding it without drift across all seven was worth the discipline.

- **Decision criteria were named without weighting.** The criteria section gives the ADR author eight axes and explicitly does not say "criterion 1 is most important and criterion 8 is least." Different reasonable people would weight these differently; the ADR records the weighting actually applied. This avoids encoding the document author's preference as if it were the team's.

## What was harder than expected

- **The session size against the prompt's scope.** The prompt called for 50–100 candidate sources at the source-discovery stage, ten landscape axes, seven candidate stacks, and a full appendix structure. In practice the session reached approximately 45–50 quality-bar-clearing source candidates across roughly nine search rounds before moving to writing. That is at the low end of the prompt's target. The document is comprehensive enough that the ADR author should not need to re-do the foundational research, but a future research session on a topic with similarly broad scope should budget more searches up front — or scope the document to a smaller surface area.

- **The "do not pick a winner" discipline against the gravitational pull of the precedent.** CritterBids' Vite + React + TanStack Query + Tailwind + shadcn/ui choice is a strong precedent for an architecturally adjacent project, and on most of the decision criteria (six-day fit, .NET-shop audience fit, demo ergonomics, long-road preservation, ecosystem health) Candidate A scores highest. The discipline of laying out all seven candidates with equal sub-structure and not concluding "and therefore A wins" required deliberate restraint. The document is honest about Candidate A's strength without pre-deciding the ADR. Whether that restraint paid off depends on whether the ADR author finds room to weigh Candidates C, E, or G as serious contenders given a different criteria weighting.

- **The candidate-stack count.** Seven is on the high end. Reasonable arguments exist for collapsing D (Blazor Server) into C (Blazor United) as a render-mode subset, and for omitting F (SvelteKit) given how weak its .NET-shop audience fit is. The prompt suggested 4–7 candidates with the freedom to merge; the session kept all seven for completeness, on the reasoning that the ADR author benefits from seeing every credible option even if some are obvious quick rejects. A four-candidate version (A, B, C, G) would have been faster to author and perhaps easier to read.

- **The tool-confusion incident on first file-write.** The session initially used the `create_file` tool on what looked like a Windows path but resolved to Claude's container rather than the user's filesystem. The file appeared to write successfully but was not actually on the user's machine. Recovered by loading `Filesystem:write_file` and re-writing. This cost roughly one round-trip; lessons for future sessions are in "Methodology refinements that emerged" below.

- **Source-quality verification under time pressure.** The prompt's source-quality bar is strict — verify the author has other substantive work, verify the venue is real, verify claims are specific (named systems, real benchmarks, real production stories). Doing this for every candidate source is slow. The session compressed verification by leaning on a pre-known pool of recognized authors (Carson Gross, Rich Harris, Misko Hevery, Ryan Carniato, Lee Robinson, Tanner Linsley, Addy Osmani, Heydon Pickering, DHH, Tom MacWright) where verification was effectively cached from prior knowledge of their work. New-to-the-session authors (Allen Conway, Alex Cloudstar, Maddy Montaquila) got a lighter verification pass — checked-they-exist plus venue-cross-reference — but not the full "find five other substantive engineering posts" treatment that the prompt's strictest reading would require. A more rigorous session would tighten this.

- **The "AI-content-smell" exclusion is judgment-based, not deterministic.** Several search results landed in Appendix A under this banner. The signal is real (generic phrasing, no concrete examples, formulaic structure, recent date with no findable author footprint) but the call is judgmental. A different session-runner might have included one or two of the rejected sources, or excluded one or two of the included ones. The document records the criterion explicitly so the call is auditable.

## Methodology refinements that emerged

These are observations about the research-document process worth carrying forward, not corrections to the document itself.

1. **Two-filesystem tool confusion is a real session hazard.** The user's filesystem (`C:\Code\...` via `Filesystem:write_file`) and Claude's container filesystem (`/home/claude`, `/mnt/...` via `create_file`) are different surfaces. The `create_file` tool's "container" wording is ambiguous when the path looks like a Windows path. Future sessions should explicitly load `Filesystem:write_file` early and use it for any output destined for the user's repository — never fall back on `create_file` for user-filesystem writes. Worth a one-line CLAUDE.md note when this kind of issue recurs.

2. **The decision-evidence framing is load-bearing for "do not pick a winner."** The document's header paragraph explicitly states "This document is decision-evidence for the round-one frontend-stack ADR, not the decision itself" and "It does not recommend a single stack." Holding that framing in mind while writing every section — particularly the candidate-stack subsections, where the temptation to insert "and therefore Candidate A is best" is highest — was meaningful. Future research-kind sessions whose deliverable is decision-evidence (rather than reference-building, as Research 001 was) should make the framing explicit early.

3. **Sibling-project research as precedent versus as authority.** CritterBids' frontend research did substantial deliberation that overlaps with what Research 002 needed to do. The session treated it as precedent (cited where it informed the document, named in Appendix B as suggested follow-up reading) without treating it as authority (Candidate A was evaluated on its own merits against CritterMart-specific criteria, not adopted because CritterBids did). This distinction is worth carrying forward whenever a sibling project's research overlaps with the current project's research. The pattern: "the sibling did this deliberation; we credit the work but re-evaluate against our own constraints."

4. **The candidate-stack sub-structure can be promoted to a shared format.** The seven candidates' uniform sub-structure (What it is / Strengths / Weaknesses / Demo ergonomics / Six-day fit / Audience fit / Long-road preservation) reads well and would generalize to other decision-evidence documents. Worth lifting into a `docs/skills/` skill file the next time a research session needs comparative-candidate analysis. The skill would name the sub-fields and the cross-cut readability requirement.

5. **Real-time flagged as long-road in every place it appears.** The prompt was explicit about this and the document held the discipline: the Real-time axis is labeled LONG-ROAD ONLY in its heading; each candidate's "Long-road preservation" cell mentions SignalR or equivalent; nowhere does real-time material influence round-one scoring. For future research documents on topics where the prompt names a scope cut, the pattern of labeling the relevant section header AND naming the long-road-only nature inside each candidate's relevant cell is worth keeping. Belt and suspenders.

6. **Source quality verification benefits from a pre-known-author pool.** Roughly two-thirds of the cited sources had authors the session already recognized from sustained engineering writing (Carson Gross, Rich Harris, Misko Hevery, Ryan Carniato, Lee Robinson, Tanner Linsley, Addy Osmani, Heydon Pickering, DHH, Tom MacWright, Allen Conway). Verification for these effectively reduced to "is this the same author on the official venue?" — fast. New-to-the-session authors required heavier verification. A future research session could front-load a "named-author trust list" pass: list the recognized authors expected for the topic, search their work directly, and only then do open-ended discovery for any remaining gaps. This would be faster than the current pattern of open-ended search-then-verify per result.

7. **The "Sources surveyed" table size is the first signal of source-discovery sufficiency.** The document's table has 38 entries. The prompt called for 50–100 candidates at discovery stage and a smaller cited set. The 38 cited / ~50 surveyed split sits at the low end. A future session should aim for 60–80 surveyed candidates as the comfort-zone target for a comparable scope.
