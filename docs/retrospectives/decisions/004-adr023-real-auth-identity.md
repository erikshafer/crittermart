# Retrospective — Decisions 004: ADR 023 — real authentication for Identity (ASP.NET Core Identity + self-validated JWT)

## Outcome summary

Authored the pre-code design layer for giving Identity real authentication, per the handoff. **ADR 023** records the decision: ASP.NET Core Identity for the user store and password handling, and — the architecturally load-bearing part — a **self-validated, asymmetrically-signed JWT** that Catalog/Inventory/Orders verify **offline** against Identity's config-distributed public key, so real auth adds **zero synchronous service-to-service HTTP** and honors ADR 001's no-sync-HTTP non-negotiable without bending it. The `X-Customer-Id` seam retires as the trust boundary in favor of the JWT `sub` claim. The two sub-questions the handoff flagged as open — cookie-vs-bearer and how sibling services trust auth — collapsed into one axis and were resolved with Erik via `AskUserQuestion` (self-validated JWT bearer; asymmetric key, public half distributed as config) before any drafting.

The decision propagated in one coherent PR: **Workshop 002 v1.3** adds slices 5.8–5.11 (register-with-credentials, log-in/issue-JWT, verify-token-at-resource-server, log-out) with an `*(auth)*` marker, an auth architect note deriving the trust boundary as the no-sync-HTTP analogue of the existing OHS/PL split, and GWT scenarios; the **context map** names the auth relationship (OHS + Published Language extended, resource servers Conformist, but validated offline so *no runtime edge*) and marks the Long-road item resolved; **`structural-constraints.md` v1.8** is the required paired rule-file update; **ADR 009** gains a third amendment superseding its auth-deferral stance only; the README index and a minimal, deliberate **vision.md** Long-road/​non-goal update close the set. No code, no `openspec` change this session — design-only, as the handoff scoped.

## What worked

- **Verifying the `MapIdentityApi`-is-not-a-JWT fact before presenting options changed the recommendation.** Per the global ctx7 rule, I fetched current ASP.NET Core docs rather than trusting training data — and the docs are explicit that Identity's built-in login issues *custom, non-JWT tokens validated via its own Data-Protection key ring*, not standard JWTs. That single fact is what makes cookie and opaque-bearer options unable to support offline sibling-service validation, and what forced the standard-JWT layer as a *separate* decision on top of "use ASP.NET Core Identity." Presenting options without that fact would have muddied the fork.
- **Recognizing the two flagged sub-questions were one axis.** The handoff framed cookie-vs-bearer and cross-BC-trust as two open questions; they are the same question seen from two angles (the credential must ride the browser's HTTP request, since RabbitMQ can't authenticate an inbound browser call, and each service must validate it offline). Naming that collapse up front made the `AskUserQuestion` previews concrete — trust topology diagrams, not abstract prose — and kept the decision to two clean picks.
- **The offline-validation reconciliation gave the workshop a real teaching payoff.** The auth architect note in Workshop 002 § 3 mirrors the existing "OHS-for-frontend / PL-for-backends" note exactly: the same no-sync-HTTP rule that keeps customer *data* off sync HTTP now keeps identity *trust* off it. That the increment adds *zero* new cross-BC runtime edges is the pedagogical point, not an incidental property.
- **The pre-existing amendment/marker conventions were ready-made.** ADR 009 already carried two blockquote amendments and the workshop already used "vN.M amendment" paragraphs and `*(marker)*` slice tags — extending those rather than inventing mechanics kept the append-only discipline honest with no design overhead.

## What was harder / notable

- **A directional slip ("below" vs "above") surfaced only in the post-authoring sweep.** The v1.3 amendment referenced the superseded v1.0 "not auth" line as "below" when it sits above the amendment. A grep sweep for stale absolute-negation phrasing caught it alongside two v1.0 flat-negation lines (§ 1 and § 2) that needed forward-markers. Lesson reinforced: after superseding prose in an append-only doc, sweep the whole file for every bald claim the supersession contradicts — the amendment paragraph alone doesn't neutralize a contradiction three paragraphs away.
- **Distinguishing frozen history from live contradiction.** The same sweep flagged Document-History rows ("Identity stays a data store, not auth" in the workshop v1.0 row; "still no authN/authZ" in structural-constraints v1.7 row). Those are correct *as history* and must not be rewritten — the discipline is to add a new version row recording the change, which the v1.3 / v1.8 entries do. Judgment call resolved by the append-only rule, not a fresh decision.
- **The required-paired-update chain is exactly the discipline `structural-constraints.md`'s own v1.7 entry shows fails silently when skipped.** ADR 022 and ADR 009's second amendment both shipped without their paired rule-file update and had to be reconciled later (v1.7). This session honored the pairing proactively — ADR 023 and structural-constraints v1.8 in the same PR — rather than repeating that gap.
- **vision.md scope was a deliberate-vs-opportunistic judgment.** The handoff DoD didn't name vision.md, but its Long-road line said "the auth mechanism is not yet chosen," which ADR 023 makes false, and a round-one non-goal flatly claimed "carries no authentication." Touching both is decision-driven (an ADR landing), not opportunistic scope-creep — and it parallels the `2f5cda4` precedent that corrected both Long Roads together. Kept the edits minimal and flagged them explicitly for Erik.

## Methodology refinements

- **Check whether flagged forks are genuinely independent before presenting them.** The handoff's two open sub-questions were one axis; had I presented them as two separate `AskUserQuestion` items, Erik would have answered a false dichotomy. Collapsing them first produced a cleaner, more honest decision surface.
- **For a library-API-shaped decision, fetch current docs before drafting the options, not after.** The ASP.NET Core Identity token-format fact was decisive, not incidental; getting it up front shaped which options were even viable.
- **An ADR that changes a `structural-constraints.md` line must land the rule-file update in the same PR — treat it as part of authoring the ADR, not a follow-up.** The file's own header rule says so, and its v1.7 entry is the standing evidence that "later" means "silently stale." Encode this as a reflex: authoring an ADR includes checking the constraints file for contradicted lines.
