# Retrospective — Decisions 005: ADR 024 — DCB-protected coupon redemption in Orders (Promotions definitions-only)

## Outcome summary

Ran the post-auth "pick the next direction" mission from the handoff. The open fork (six unranked long-road candidates, real auth having closed candidate #1 of the prior fork) was presented to Erik via `AskUserQuestion` with previews and a recommendation; he chose **Promotions via DCB**. Two further forks were then resolved the same way before any drafting: the coupon **invariant** (a **global redemption cap**) and the **DCB topology** (redemption + enforcement **inside Orders**, Promotions contributing coupon **definitions** only — not a fifth service).

**ADR 024** records the decision. Its spine is a modeling choice, not an infrastructure one, because the two facts that could have forced a heavier first artifact were both verified false: (1) **DCB does not require Polecat** — it is first-class in the pinned **Marten 9.11.0** (verified in `Marten.dll`: `FetchForWritingByTags`, `EventTagQuery`, `DcbConcurrencyException`), additive to stream-based sourcing, opt-in schema (`tags TEXT[]` + GIN index) — so no new engine and no Wolverine/Marten bump (which the CritterWatch 1.0.0-beta.1 coupling would have blocked); and (2) **DCB is store-scoped**, which is precisely why the cross-stream cap must live in one service's store, and — since every checkout flows through Orders — that store is Orders. The ADR argues DCB's value honestly on a **contention** axis (a popular coupon makes a single-stream `Coupon` aggregate a write hotspot exactly when the cap bites) rather than pretending a single-stream aggregate couldn't enforce a pure global cap.

The decision propagated in one coherent design-only PR (no code, no `openspec` change), mirroring ADR 023's decision-PR (#129): the **context map** resolves the Long-road entry and forward-marks the "no Promotions BC" stub; **`structural-constraints.md` v1.9** is the required paired rule-file update (a DCB persistence rule; "Promotions with DCB" retired from the deferred list); **vision.md § Long road** names the chosen-but-not-yet-built fourth increment; the README index and the retrospectives-README count close the set. The per-slice build is deferred to **Workshop 003 (Promotions)** — a new workshop, since Promotions is new territory with no existing workshop to amend.

## What worked

- **Verifying the DCB-engine question before recommending anything reshaped the whole session.** The installed skill titles DCB a *Polecat* pattern, and ADR 009 defers Polecat — so the naive reading is "DCB ⇒ SQL Server 2025 ⇒ a big engine ADR." Fetching current Marten docs *and then grepping the actual pinned `Marten.dll`* proved DCB ships in 9.11.0. That single fact turned a feared infrastructure-reversal into a lightweight modeling decision, and it is why the recommended direction (highest teaching novelty) was also the *lowest*-risk once verified. Per the global ctx7 rule: check the library fact before framing the options, not after.
- **Naming the store-scoping constraint turned a style question into a forced one.** "Where does the DCB live?" reads like taste until you see that `FetchForWritingByTags` queries one store's `mt_events` — then schema-per-service (ADR 002) *dictates* the answer. Presenting the topology fork with that physical constraint up front (not as prose but as the reason the two options exist) made Erik's pick a real engineering decision rather than a preference.
- **Writing the honest "why DCB over a single-stream aggregate" into the ADR rather than hiding it.** The global-cap invariant is the case where a plain `Coupon` aggregate is *most* competitive with DCB — so the ADR had to earn DCB on contention, and name the richer variants (one-per-customer, shared budget) as the more-dynamic-boundary extensions this is the entry point to. Surfacing the tension is stronger teaching material than a strawman.
- **The three-fork sequence (direction → invariant → topology) each fed the next.** The invariant chosen constrained what the topology options even were; the topology chosen is the ADR's Decision spine. Splitting them into three `AskUserQuestion` calls with previews kept each decision legible instead of bundling a false compound choice.
- **The ADR 023 decision-PR was a ready-made template.** Retro 004 documented the exact paired-cascade set (ADR + context map + `structural-constraints` version bump + vision + README index + retro, no separate prompt); reusing that shape kept the required-pairing discipline honest with no re-derivation.

## What was harder / notable

- **The recommended candidate carried a hidden dependency that had to be chased down before it could be recommended in good faith.** "Promotions via DCB" looked like the highest-payoff option in the abstract, but recommending it *responsibly* meant first ruling out the Polecat/engine reversal and the Marten-version-bump blocker. The recommendation in the fork preview was made *after* that verification, not before — the right order, but it meant the first tool work of the session was library recon, not authoring.
- **"Definitions-only inside Orders" bends the context-map framing, and that had to be written as a named cost, not smoothed over.** The Long-road entry always described Promotions as its own BC publishing coupon definitions Orders *consumes*. Realizing it as definitions-in-Orders this increment is a deliberate scale-down; the ADR, context map, and constraints file all say so explicitly and keep the standalone-service option alive as a forward question. Append-only honesty over a tidy story.
- **`structural-constraints.md` is titled "Round One" yet keeps absorbing post-round-one ADRs.** ADR 022, 023, and now 024 all amended it. The file is the living orientation surface, not a frozen round-one snapshot — treated it accordingly (new Persistence rule + deferral retirement + v1.9 history row) rather than hesitating over the title.
- **vision.md § Long road needed the chosen-vs-built distinction stated precisely.** The three prior increments are *shipped*; Promotions is *chosen but not yet built*. Wording that as a "fourth increment, design-only, ADR 024, Workshop 003 next" avoids implying code exists — the same care ADR 023's Long-road edit took.

## Methodology refinements

- **When a direction's headline pattern maps to a specific persistence engine in the skills library, verify the engine claim against the pinned assembly before recommending the direction.** The DCB-is-Polecat assumption would have mis-shaped the entire first artifact. Grepping `Marten.dll` for the API symbols was faster and more authoritative than docs alone, which only referenced `master`.
- **A physical constraint (store-scoped DCB) is a better fork framing than a preference.** Leading the topology question with *why the options exist* (schema-per-service + tag queries are store-local) produced a decision Erik could reason about, not just prefer. Encode: when presenting an architecture fork, lead with the constraint that makes it a fork.
- **Author the ADR, checkpoint its framing, then cascade.** Showing the ADR's decision + honest tradeoffs before rippling into five paired files gave a cheap correction window; the pairings are mechanically entailed once the ADR framing is agreed, so the checkpoint sits exactly at the ADR, not mid-cascade.
