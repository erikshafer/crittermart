# Retrospective — Implementations 033: Customer registry (slices 5.1 + 5.2), the EF-Core Identity service on `main`

## Outcome summary

Landed **CritterMart's 4th deployed service**, `CritterMart.Identity`, on `main` through the per-slice chain — re-righting the trace the spike (`spike/efcore-identity` @ `0ffe42e`) legitimately inverted. Identity is the **one non-event-sourced bounded context**: Wolverine over **EF Core / Npgsql**, schema-per-service `identity`, a mutable `Customer` row as the source of truth, and a single `CustomerRegistered` notification cascaded through the **EF-Core transactional outbox** — the persistence-agnostic-handler teaching beat (the same Wolverine shape and outbox guarantee over a `DbContext` instead of an `IDocumentSession`).

Slices **5.1 (Register)** and **5.2 (Resolve)** shipped in **one PR** per the consolidate-slice-PRs convention. The kept code re-applied the spike's 15 files verbatim and added the one thing the spike skipped — a **duplicate-email guard**, the session's one genuine fork:

- **Uniqueness key = normalized email** (trimmed + lowercased), so casing/whitespace can't defeat it (`Ada@Example.com` ≡ `ada@example.com`).
- **Enforced twice over** — a railway-style `ValidateAsync` → `409 CustomerAlreadyRegistered` (mirroring `PublishProduct.ValidateAsync`) for the friendly path, **plus** a DB **unique index** on the email column as the race backstop.

Both sub-decisions were **put to the owner** (AskUserQuestion with code previews) and confirmed: normalized + both-layers. The why is the heart of the design — Catalog gets DB-level uniqueness *for free* (a product's SKU **is** its Marten document id, so the PK enforces it), whereas Identity keys uniqueness on email, **not** the primary key, so the unique index is what makes Identity's guarantee a faithful *equal* of Catalog's rather than a weaker app-only check.

**The index's mechanism changed mid-session because the live boot caught a real gap.** It was first declared as an EF `e.HasIndex(x => x.Email).IsUnique()` — and the integration suite went green — but a live `pg_indexes` check found **only `pk_customers`**: the index was not in the database. Root cause (confirmed against the Wolverine EF-Core docs): `UseEntityFrameworkCoreWolverineManagedMigrations` drives Weasel, which migrates tables/columns/PKs/FKs but **not secondary indexes** — so the EF `HasIndex` was silently dropped, and the app-guard-only integration test passed regardless. Fix: apply the index as **idempotent startup DDL** (`CREATE UNIQUE INDEX IF NOT EXISTS ux_customers_email …`) from an `ApplicationStarted` lifetime hook in `Program.cs` (fires after Weasel's table creation, runs synchronously), plus a **new test that exercises the index directly** — a duplicate inserted through the `DbContext`, bypassing the app guard, must raise Postgres `23505`.

**Tests**: **Identity 6/6, 0 failures** (the 4 re-applied spike tests + `registering_a_duplicate_email_is_rejected_case_insensitively` [the app-guard path] + `the_email_unique_index_rejects_a_duplicate_inserted_directly` [the DB backstop, direct insert → `23505`]). `dotnet build` (full solution) clean — 0 errors, only the pre-existing NU1507. `openspec validate customer-registry --strict` green (Complete). **Live-verified on the full Aspire stack:** all 4 services healthy (Identity `:5105`); register `201` / resolve `200` / unknown `404` / duplicate `409` (case-insensitive — `ADA@example.COM` rejected as `ada@example.com`); 4 schemas coexisting; one row despite the duplicate; outbox drained to 0; and after the fix, `ux_customers_email` present in the DB and a direct duplicate insert rejected with `duplicate key value violates unique constraint "ux_customers_email"`.

**Spec movement**: new OpenSpec **`customer-registry`** capability with **3 ADDED requirements** (register + outbox; email uniqueness [the net-new content over the spike]; resolve + 404). **Narrative 006** (v1.0) is the human-readable sibling — the first narrative for the Identity BC. The four-step closure is complete bar the post-merge archive.

## What worked

- **Verifying the spike-vs-`main` divergence before re-applying turned a risky three-way merge into a pure additive replay.** Four of the spike's 15 files are *modifications* to shared build files (`Directory.Packages.props`, `CritterMart.slnx`, AppHost `Program.cs` + `.csproj`). A naive `git checkout 0ffe42e -- <path>` could have silently reverted whatever landed on `main` after the spike branched. Checking `git merge-base` + `git diff --stat $MB main -- <each shared file>` first proved `main` was only 2 docs-only commits (#81/#82) ahead and **none of the 4 shared files changed** — so the checkout was exact and safe for all 15 files. That one recon step is the difference between a clean replay and reverting two PRs' worth of edits.
- **The owner-fork was framed with concrete code previews, not abstract prose.** The two guard sub-decisions (email semantics, enforcement layers) went to the owner as AskUserQuestion options carrying the actual `Normalize(...)` snippet, the `HasIndex(...).IsUnique()` line, and the GWT each implies — so the choice was made against what the code would literally be, not a paraphrase. Both came back as the recommended option.
- **Mirroring an in-repo idiom beat reaching for docs.** The guard is `PublishProduct.ValidateAsync` (a proven, on-this-Wolverine-version pattern) expressed over a `DbContext` — `AnyAsync` instead of `CheckExistsAsync`, `ProblemDetails`/`WolverineContinue.NoProblems` identical. The duplicate-email *test* is the `PublishProductTests` shape (assert `409`, then verify idempotency from a fresh scope — one row, the row-store analogue of "no shadow event"). No new pattern was invented; the slice reads as a sibling of the Catalog guard.
- **The live boot earned its keep — it caught a gap a green test suite hid.** The integration suite passed with the index missing from the database, because the duplicate-email test only ever drove the HTTP path, where the `ValidateAsync` app-guard short-circuits first — the DB index was never exercised. A live `pg_indexes` check on the running stack is what exposed it. This is the live-verify norm paying off exactly as intended: *integration green ≠ the deployed schema is what you think it is*, especially for a constraint a higher layer also enforces. The fix added a test that probes the index **directly** (bypassing the guard), so the proof now lives in the suite too — but the live boot is what found it.

- **The discipline was the deliverable, not the code.** ~95% of the source is a verbatim spike replay; the only net-new code is the `Normalize` helper, the `ValidateAsync` method, one `HasIndex` line, and one test. The session's real work was the *trace* — proposal → spec delta → narrative → design (recording *why* the spike's choices are right, plus the guard fork) → prompt → retro — so a future contributor can answer "what is this service for, and is it correct?" without reading the spike commit. ## Methodology refinements

- **Before replaying a spike branch, prove the shared files didn't move on the trunk.** `merge-base` + `diff --stat $MB main -- <shared files>` is the cheap check that tells you whether `git checkout <spike> -- <path>` is a safe additive replay or a destructive overwrite. For spike-promotion sessions, run it first and record the result in `tasks.md` — it is what licenses the wholesale checkout.
- **When a workshop explicitly defers a decision "to the proposal," that is the AskUserQuestion trigger.** Workshop 002 § 8 parked the uniqueness key for slice 5.1's proposal. Rather than silently picking, the fork (and its orthogonal enforcement sub-fork) went to the owner with previews — exactly the kind of genuine, trade-off-bearing decision the collaborate norm reserves for the owner, distinct from the conventional defaults (capability name, `409` status, branch name) that were just settled with a one-line rationale.
- **A faithful mirror matches the *guarantee*, not the code shape.** "Mirror `PublishProduct`'s guard" naively means "app guard only, no index." But Catalog's guard rides on SKU-as-document-id, which is a DB-level uniqueness guarantee the app check merely fronts. The faithful Identity mirror therefore *adds* the unique index, because email is not the key — matching Catalog's actual guarantee level, not just its visible code. Worth stating because "mirror X" can mislead when X gets a guarantee for free that the new context doesn't.
- **When a higher layer also enforces a rule, test the lower layer in isolation — and verify the schema, don't infer it.** The EF `HasIndex` looked applied (it's in the model) and the suite was green, but neither implied the index existed: the app guard masked its absence, and Weasel doesn't migrate EF indexes. Two habits close this class of gap — (a) a test that exercises the backstop *alone* (here: a direct `DbContext` insert that must hit `23505`), and (b) a live schema assertion (`pg_indexes`) rather than trusting that "the model declares it" means "the database has it." Generalizes to any defense-in-depth constraint: prove each layer independently, because a passing end-to-end test can be carried entirely by the outer layer.
- **Know what your migration tool actually migrates.** `UseEntityFrameworkCoreWolverineManagedMigrations` migrates tables/columns/PKs/FKs but not secondary indexes — a real, documented boundary. Reaching for `ctx7`/the docs the moment the live DB contradicted the model turned a confusing "where did my index go?" into a clean root cause and the right fix (out-of-band startup DDL) in one pass.
