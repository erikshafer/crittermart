# Retrospective — Implementations 024: Stock read/write split (ADR 020 rollout to the StockLevel aggregate)

## Outcome summary

Rolled the **ADR 020 (domain write-models vs. `*View` read models)** pattern out to **Stock** — the third and **final** pilot after Cart (#59) and Order (#63). `StockLevelView` stopped doubling as the write target and the served read model and split into a domain-named write aggregate and a dedicated read projection; no behaviour, contract, or frontend changed. This closes the ADR 020 rollout — all three round-one event-sourced aggregates are now split.

- **`Stock/StockLevel.cs`** (new) — the `sealed record` write aggregate: `Available`/`Reserved`/`Committed`/`Reservations` (+ `Id`), self-aggregating `Snapshot<StockLevel>(SnapshotLifecycle.Inline)`, static `Create`/`Apply`. The `FetchForWriting` target on all four stock write paths; never served over HTTP. `Reservations` is the load-bearing field (the analogue of `Order.Status`): the reserve/release/commit idempotency guards read it.
- **`Stock/StockLevelView.cs`** — converted from a mutable `class` + `StockLevelViewProjection : SingleStreamProjection` to a `sealed record` self-aggregating `Snapshot<StockLevelView>(SnapshotLifecycle.Inline)` with the same folds, **wire shape `{ id, available, reserved, committed, reservations }` byte-for-byte preserved**.
- **Repointed**: `ReceiveStock`/`ReserveStock`/`ReleaseStock`/`CommitStock` → `FetchForWriting<StockLevel>` (the guards read `.Available`/`.Reservations` off `StockLevel`); the `GET /stock/{sku}` `LoadAsync<StockLevelView>` read path is unchanged. `Program.cs` swaps `Add<StockLevelViewProjection>` for `Snapshot<StockLevel>` + `Snapshot<StockLevelView>`.
- **No folder rename, no PMvH.** ADR 021 does not fire (`StockLevel ≠ …Stock`, no collision), and Inventory has no process manager — so this was the smallest of the three rollouts (the Order pilot did 020 + 021 + the PMvH-state move).

**Tests**: full solution green — **106 backend tests, 0 failures** (Catalog 9, Inventory **21**, Orders 73, CrossBc 3); `dotnet format` clean. The former `StockLevelViewProjectionTests` (2 tests of the deleted projection class) became `StockLevelProjectionTests` (7: five `StockLevel` aggregate folds incl. the `Available + Reserved + Committed = ΣReceived` invariant + two `StockLevelView` read-model consistency cases), mirroring `OrderProjectionTests`. The Inventory integration tests (`LoadAsync`/`ReadAsJson<StockLevelView>` over real Postgres) and the three CrossBc smoke tests (`place → ReserveStock → StockReserved → CommitStock` over the real RabbitMQ broker, asserting available 98 / committed 2) prove the wire shape and the cross-BC flow survived the split.

**Live verification** (real Aspire stack, ports 5101/5102/5103): seeded a receipt → `GET /stock/{sku}` returned the exact shape `{ id, available:100, reserved:0, committed:0, reservations:[] }` → a place→reserve→confirm cycle settled to `{ available:98, reserved:0, committed:2 }` with the order `confirmed`. Both handoff verification asks met live.

**Spec movement**: **no new ADR, no OpenSpec/workshop/narrative change** — a contract-preserving model refactor. ADR 020 gained an **"Applied — the Stock pilot"** subsection and its **Revisit note is now closed** (all three aggregates split, the rollout complete); `docs/rules/structural-constraints.md` → **v1.6** (rollout-status line closed; the frontmatter `version` also caught up from the v1.5 entry's table-only bump).

## What worked

- **The Order pilot was an even cleaner template than Cart was for Order.** Stock is structurally simpler — no folder rename, no namespace sweep, no PMvH state to relocate — so the hand-authored work was just `StockLevel.cs`, the `StockLevelView` conversion, four one-line `FetchForWriting` repoints, and the Program.cs registration. The production project compiled on the first build after the repoint.
- **The read model survived untouched, and that was provable, not hopeful.** Because `StockLevelView` keeps its record shape and stays an inline snapshot, the four handler tests and the three CrossBc smoke tests (which `LoadAsync`/`ReadAsJson<StockLevelView>`) compiled and passed with **zero edits** — the split is invisible to every reader of `GET /stock/{sku}`. The live Aspire probe confirmed the same shape byte-for-byte over real HTTP.
- **No fork to surface — confirmed by checking the authority first.** The instinct was to surface "shared fold helper (à la Cart's `CartLines`) vs. duplicate the folds in both records" as a fork. Re-running the Order retro's discipline — *check the ADR + precedents before drafting an `AskUserQuestion`* — settled it: Order duplicated its (trivial) folds; Cart extracted a helper only because its line-fold is one cohesive list operation. Stock's fold is multi-field arithmetic across two record types, and only the one-line reservation-drop is shareable, so it does not factor into a clean single-field helper. Duplicating the five folds (the Order mirror) with the shared `StockLevelProjectionTests` guarding drift was the prescribed, no-fork path.
- **The integration suite caught the one genuinely novel mechanic for free.** `StockReceived` is both genesis (`Create`) and repeatable (`Apply`) — the only structural departure from Cart/Order, whose genesis events fire once per stream. The pre-existing `receiving_additional_stock_accumulates_available` test (100 + 50 = 150) is the binding proof that Marten dispatches `Create` on the absent snapshot and `Apply` thereafter; it passed without modification, and the live restock-free probe corroborated it.

## What was harder / notable

- **`StockReceived`'s Create+Apply duality is the one place the record conversion isn't mechanical.** The old mutable `StockLevelViewProjection` had a single `Apply(StockReceived)` and leaned on Marten default-constructing the empty `class` on the genesis event. A `sealed record` has no construct-then-mutate path, so the aggregate needs an explicit `Create(StockReceived)` (opens the level at `Available = e.Quantity`) *and* an `Apply(StockReceived, …)` (accumulates). Every event carries `Sku` first, so `Create` reads `e.Sku` for the `Id` — no `IEvent<T>` wrapper needed (Stock tracks no activity timestamp, unlike Cart).
- **The live cross-BC probe surfaced (and explained) the carry-forward `AddToCart` null-snapshot 500.** The first probe attempt 500'd with an NRE in `CartLines.Add` (`CartLine.cs:19`, `added.Snapshot.Name`). Root cause: my hand-rolled JSON used `snapshot` but the `AddToCart` command field is `ProductSnapshot` → JSON `productSnapshot`; the mismatch bound null, and the Cart projection dereferenced it. This is exactly the carry-forward gap — a missing/misnamed `productSnapshot` yields a 500 instead of a 400 at the boundary. The CrossBc test never hits it because Alba serializes the typed object. Fixing the request (correct field name) made the full cycle work. The robustness gap is real but lives in Orders/Cart — out of scope for this Inventory-only PR; re-flagged below.
- **The `structural-constraints.md` frontmatter was stale.** The Order pilot (v1.5) bumped the Document History table but left the header `version: v1.4`. Bumping to `v1.6` for this change corrected the header as a side effect — noted in the v1.6 history row so the divergence is on the record, not silently swallowed.

## Methodology refinements

- **"ADR + pilot one aggregate, then roll out per-PR" delivered its full promise across three PRs.** Cart de-risked Order; Order de-risked Stock. Each rollout was smaller and lower-drama than the last because the template (`Snapshot<T>` registration, wire-preservation constraint, integration-test safety net) was proven. A standing decision realized incrementally, one aggregate per PR, kept every diff reviewable.
- **A closed "Revisit" is worth stating as a posture shift, not just a checkbox.** ADR 020's Revisit didn't just tick Stock → done; the rollout completing means the split is now the **default** for any new event-sourced aggregate (born split, not migrated). Recording that in the ADR + the rules file turns "we finished the migration" into "here is the standing rule," which is the more useful thing for the next contributor.
- **Re-deriving the no-fork judgment from the prior retro saved the round-trip again.** This is the second consecutive read/write-split pilot where the helper-vs-duplicate instinct looked like a fork and wasn't. The pattern is now clear enough to encode as a heuristic: *a fold that touches one cohesive collection wants a shared helper (Cart); a fold of independent scalar fields across two record types is cleaner duplicated (Order, Stock).*
