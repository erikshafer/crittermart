# Retrospective: Implementations 011 — Slice 4.7 Cancel on Payment Timeout (Bruun temporal automation)

**Prompt**: `docs/prompts/implementations/011-slice-4-7-cancel-on-payment-timeout.md`
**Outcome**: shipped — the project's first temporal automation, and the **last order-cancellation path**. Placing an order now also schedules an `OrderPaymentTimeout` self-message (Wolverine `DelayedFor`, config-backed `Orders:PaymentTimeout`, default 10 minutes, durable across restart). When it fires on a non-terminal order, `PaymentTimeoutHandler` appends `OrderCancelled { payment_timeout }` and cascades `ReleaseStock` to Inventory **unconditionally** — reusing slice 4.6's contract and `ReleaseStockHandler` with **zero Inventory changes**. A new inline `OrdersAwaitingPayment*` projection (conditional delete) is the observable Bruun todo-list at `GET /orders/awaiting-payment`. **The Order lifecycle is complete for round one**: every placed order reaches `confirmed` or `cancelled`. OpenSpec change `slice-4-7-cancel-on-payment-timeout` `--strict` valid; Narrative 004 → v1.5 (Moment 6). One consolidated PR.
**Tests**: full solution green — **51 total, 0 failures** (+9): Orders 32 (+3 pure folds, +5 timeout integration, +1 schedule assertion), Inventory 10 (unchanged — the reuse held), Catalog 7 (unchanged), CrossBc 2 (unchanged — `UseDurableLocalQueues` did not ripple).

## What shipped

- **Orders — the schedule (4.1's deferred clause).** `PlaceOrder` returns a third cascaded output: `DeliveryMessage<OrderPaymentTimeout>` via `.DelayedFor(deadline.Duration)`. The deadline rides the same Marten/outbox commit that places the order.
- **Orders — the cancel handler.** `Order/PaymentTimeoutHandler.cs`: `FetchForWriting<OrderStatusView>` → terminal/unknown → `null` (no append, no cascade); non-terminal → `OrderCancelled { payment_timeout }` + `ReleaseStock` from `Aggregate.Lines`. `CancelReason.PaymentTimeout` added.
- **Orders — the Bruun todo-list.** `Order/OrdersAwaitingPayment.cs`: `OrderAwaitingPayment` doc (orderId, customer, total, deadline) + `OrdersAwaitingPaymentProjection` — `Apply(IEvent<OrderPlaced>)` (deadline = event timestamp + configured timeout) and `ShouldDelete(OrderConfirmed)` / `ShouldDelete(OrderCancelled)` conditional deletes. Instance-registered inline with the configured timeout. `GET /orders/awaiting-payment` endpoint.
- **Configuration + durability.** `Orders:PaymentTimeout` in appsettings (default `00:10:00`), `PaymentDeadline` singleton, and `opts.Policies.UseDurableLocalQueues()` so scheduled deadlines survive a service restart.
- **OpenSpec** — `order-lifecycle` only (2 ADDED requirements: cancel on payment timeout; track orders awaiting payment) — the **first Orders slice since 4.1 with no Inventory spec delta**, the designed-in reuse made structurally visible.
- **Narrative 004 → v1.5** — Moment 6 (time as the trigger; the lifecycle complete).

## What worked

- **All four verify-before-wiring items confirmed first-try, and the build compiled clean on the first pass.** `DelayedFor` → `DeliveryMessage<T>` as a cascading tuple member, `opts.Projections.Add(instance, lifecycle)` for the constructor-injected projection, the `IEvent<T>` Apply metadata convention, and `ShouldDelete` — all exactly as the docs said. Lifting these from prompt prose into **tasks.md group 1 (numbered, checkable)** made the doc checks first-class deliverables instead of a vibe; worth keeping.
- **The guard idiom paid its compounding dividend.** `PaymentTimeoutHandler` is the fifth handler to use `FetchForWriting` + status guard → no-op, and it was the easiest yet — the terminal-state guard the workshop demands is *exactly* the idiom, with nothing added. The handler is 15 lines of logic.
- **The reuse promise held literally.** Zero Inventory files touched; Inventory's 10 tests pass byte-for-byte. 4.6's design Decision 3 (build the per-SKU no-op guard before anything needs it) is what made decision 2 here (always release) free — the unanswered-order test exercises the guard branch 4.6 built but could never reach.
- **"Invoke the timeout message directly" was the whole test strategy, and it was enough.** A scheduled message is just a message with deferred delivery — so `InvokeMessageAndWaitAsync(new OrderPaymentTimeout(orderId))` *is* the deadline passing. No clocks mocked, no `Task.Delay`, no flaky timers; the suite runs in the same ~9s it did before. The schedule itself is pinned separately by `tracked.Scheduled.SingleMessage<OrderPaymentTimeout>()`.
- **The openspec CLI's artifact workflow (`new change` → `status` → `instructions` per artifact) scaffolded the change cleanly** — first session to use it end-to-end (prior sessions hand-authored the four files). The dependency ordering (proposal → specs/design → tasks) matched how the artifacts naturally build on each other.

## What was harder / notable

- **The seed-helper had a chicken-and-egg I didn't see until writing tests.** The first cut took `params object[] eventsAfterPlaced` — but a caller can't construct `new StockReserved(orderId)` before the helper generates the orderId. Restructured to two explicit helpers (`SeedPlacedOrderAsync` / `SeedOrderAtPaymentGateAsync`), which also reads better. Small, but the kind of thing a prompt's test plan can't foresee.
- **Two new Marten patterns entered the codebase in one slice**: constructor-injected projection instances (`opts.Projections.Add(new …(timeout), Inline)`) and event-metadata folds (`Apply(IEvent<OrderPlaced>, view)` + `new Event<T>(data) { Timestamp = … }` in unit tests). Both are documented conventions and both worked, but they are *new* conventions here — candidates for a future skill note if a third projection needs either.
- **The durability answer (open question 2) is nuanced and worth recording precisely**: scheduled local messages survive restart **only if** the local queue is durable — Marten persistence (`IntegrateWithWolverine`) alone is necessary but not sufficient. `opts.Policies.UseDurableLocalQueues()` is the one-line fix. Without it, the demo's "restart the service, the deadline still fires" beat would silently not work.
- **Workshop § 4.7's GWT text still carries the pre-4.6 message wording** ("Orders publishes `OrderCancelled { orderId }` to Inventory"). The v1.2 workshop amendment fixed § 2.3 and § 4.6 but § 4.7's own scenarios were left as-is (the projection/timeout clauses were still future then). The post-merge archive tidy should extend the same amendment note to § 4.7 — same divergence, same rationale, no new decision needed.

## Methodology refinements

- **Verify-before-wiring as a numbered tasks.md group.** Promoting the doc checks from prompt prose (Orientation 9) into checkable tasks (group 1) meant they were *done and recorded* before any code existed, and the retro can point at them. Candidate convention for every slice that touches an unverified framework feature.
- **When a slice's central decision inverts a sibling slice's logic, write the test that would fail under the sibling's logic.** The unanswered-order test (`a_timeout_cancels_an_unanswered_order_and_still_releases`) asserts a release that 4.6's only-at-the-payment-gate logic would *not* send. It exists to make silently reverting decision 2 impossible. Generalization of retro 010's "spec-delta closure surfacing as test churn" observation, but in the forward direction.
