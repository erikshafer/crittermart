# Retrospective — Implementations 041: One redemption per customer (the composite-tag second DCB, slice 6.5)

## Outcome summary

Ran the full consolidated per-slice loop the `second-dcb-one-redemption-per-customer` handoff commissioned — modeled **and** built slice 6.5 in one session/PR: CritterMart's **second Dynamic Consistency Boundary** and its first **composite-tag** boundary (ADR 024 §38). A coupon defined `oneRedemptionPerCustomer` is now redeemable at most once per customer, enforced by a `(coupon × customer)` composite-tag DCB opened alongside the existing global cap in the same checkout transaction. The one genuine modeling fork — *does per-customer compose with the global cap, is it a universal law, or a distinct coupon kind?* — was put to Erik with three previews and settled on **option 1: an opt-in policy flag on the definition** (configuration-as-events extended), so FLASH20's clean global-cap demo is untouched and a new `FIRSTORDER` demonstrates the composite. Authored the OpenSpec change (2 MODIFIED + 1 ADDED `coupon-promotion` requirements) + prompt first, then implemented, then reconciled Workshop 003 (v1.3, new slice 6.5 + GWT) and Narrative 011 (v1.2, Moment 5). **127 Orders tests green** (+6 coupon integration, +2 pure folds); full solution builds with 0 warnings.

## What worked

- **The composite-tag API verification (handoff's flagged risk) fully de-risked the build — no spike needed.** Reflecting on the *resolved* `JasperFx.Events` v2.27.0.0 assembly (`EventTagQuery` has `For`/`Or`/`Or<TEvent,TTag>`/`AndEventsOfType` but **no two-different-tag `.And<>()`**, and `ITagTypeRegistration` stores a single scalar) proved *before writing any code* that the `(coupon × customer)` boundary is a **single-scalar `CouponCustomerTag("{couponId}|{customerId}")`** — structurally identical to the shipped `CouponId`, through the exact same `RegisterTagType`/`.Or<T>()`/`WithTag` path. Where slice 6.3 needed a throwaway spike to find the append mechanic, 6.5 needed none: the mechanic was already proven, and the reflection confirmed the *only* unverified delta (the composite shape) collapsed onto the known one. The 21 green coupon integration tests were the empirical confirmation.
- **The opt-in-flag model kept blast radius tiny.** Because `oneRedemptionPerCustomer` defaults `false` and old `CouponDefined` events fold as `false`, every one of the 15 existing coupon tests and 100+ order tests passed **unchanged** — the second DCB is purely additive. Defaulted record parameters (`CouponDefined`, `CouponRedeemed`, `DefineCoupon`, `Order`) made each event evolution non-breaking with no migration.
- **Two `FetchForWritingByTags` compose on one session out of the box.** The design's open question — can two boundary reads arm two assertions in one transaction? — worked on the first run: the per-customer existence read + the global-cap read both arm, and the doubly-tagged `CouponRedeemed` participates in both. The concurrent-different-customers test proved the composite tags correctly *isolate* per-customer boundaries (no false cross-conflict).
- **The release symmetry fell out of one folded flag.** `Order` gained a single `CouponPerCustomer` bool (folded from `CouponRedeemed.PerCustomer`); the three cancel sites pass it + `CustomerId` to `AppendCouponRelease`, which rebuilds the composite tag from the same `(couponId, customerId)` pair the redemption used. No lookup, no new query — the aggregate already carried the customer id.

## What was harder / notable

- **The reachable concurrency proof is NOT a same-customer self-race — the session's main model correction.** The spec first claimed "a customer's concurrent double-submit resolves to exactly one." Tracing the real path showed it isn't cleanly reachable: the **one-open-cart invariant** (a partial-unique index) already serializes a single customer's checkouts, so two concurrent `POST /orders` for one customer collide on the *cart* stream (a plain Marten `ConcurrencyException`, **not** `DcbConcurrencyException`) — nondeterministically a 500, not a clean 409. So the per-customer invariant's real, reachable guarantee is the **cross-order existence check** (a *later* order refused because an *earlier* one redeemed), with the DCB assertion the transactional backstop. Revised the spec (the "concurrent double-submit" scenario → "concurrent *different customers* all succeed", the composite-isolation proof), design.md decision 3, and Workshop 003 §6.5 to say what's true. This is retro 039's lesson applied *before* shipping the claim rather than after: verify what the invariant's concurrency test can actually exercise.
- **The per-customer boundary is genuinely an *existence* check, a different DCB shape than the first.** The first DCB counts (`>= cap`); this one asks "does any net redemption exist for the pair?" (`>= 1`). The composite tag pre-scopes the read to exactly one pair, so `CustomerCouponUsage` is a trivial counter — *simpler* than the Polecat skill's `(course × student)` example, which gathers via a multi-tag OR and disambiguates in the aggregate by event data. CritterMart's tag-per-pair approach avoids the data comparison entirely.
- **The installed DCB skill remains Polecat-framed and misleading (unchanged since 039).** `[BoundaryModel]`, `EventTagQuery.For(…).AndEventsOfType().Or()`, `Load()` methods — none is the Marten path. Its one use was confirming the *shape* (`(course × student)` existence) is the textbook composite DCB. Verifying against the assembly, not the skill, was again load-bearing (`docs/skills/DEBT.md` row 3 still stands).
- **The second id-less boundary aggregate inherits the same test-cleanup wrinkle.** `CustomerCouponUsage` is a second id-less `[BoundaryAggregate]`; the `OrdersAppFixture.ResetAllDataAsync` SQL-truncate reset was generalized from the literal `mt_event_tag_coupon` to `mt_event_tag_%` to catch its `mt_event_tag_couponcustomer` table. The Marten 9.15.1 `DeleteAllDocumentsAsync` rough edge (retro 039 footnote b) is unchanged upstream.

## Methodology refinements

- **When a handoff flags "verify API X before modeling," reflect on the resolved assembly first — it can collapse an entire spike.** The composite-tag verification (a 5-minute reflection pass) proved the mechanic was byte-identical to a shipped one, so the whole slice de-risked to "a new domain shape on a known primitive." The proposal shipped correct symbols on the first pass (contrast retro 039, whose proposal guessed `EnableDcb()` and needed in-session correction).
- **Interrogate whether a modeled concurrency scenario is *reachable through the real entry point* before asserting it.** A DCB assertion can be theoretically sound yet practically shadowed by an upstream invariant (here, one-open-cart). State the reachable guarantee; keep the assertion as the named backstop. Prevents a spec scenario that can't be honestly tested.
- **A composite consistency boundary is a single encoded tag when the store has no multi-tag AND.** Worth a project note / upstream-skill correction: on Marten, model an `(A × B)` boundary as one strong-typed tag whose scalar encodes the pair, not two tags — the `EventTagQuery` surface has no cross-tag conjunction.
