# Retrospective — Implementations 025: Enrich `OrderStatusView` (placedAt + cancellation reason)

## Outcome summary

Enriched the Orders `OrderStatusView` read model with two **additive** fields — **`placedAt`** and **`cancelReason`** — and bound them on the W4 tracking screen, closing the two backend gaps W4 (PR #64) had honestly logged. This is the **first round-two slice to change a read *contract*** (W1–W4 to date were screen-only binds of existing contracts), and the cleanest demonstration of the ADR 020 split yet: the read view surfaces a placement timestamp the write aggregate deliberately never stored.

- **`Ordering/OrderStatusView.cs`** — record gained `DateTimeOffset PlacedAt` + `string? CancelReason` (appended after `Total`, so the existing five fields stay a wire-shape prefix). `Create(OrderPlaced)` → `Create(IEvent<OrderPlaced>)` reading `e.Timestamp` for `PlacedAt` (event metadata, mirroring `CartView.cs:23`); `Apply(OrderCancelled …)` folds `CancelReason = e.Reason` — the reason `OrderCancelled` already carried and the view had been dropping. The three intermediate folds are untouched (the `with` semantics carry the new fields forward). **No change to `OrderPlaced`, `OrderCancelled`, any handler, or `Program.cs`.**
- **`OrderProjectionTests.cs`** — added the `At<T>` `Event<T>` helper (the `CartProjectionTests` pattern, so a pure fold test can hand `Create` a wrapped event with a controlled timestamp) + assertions: `PlacedAt` set at genesis and surviving folds; `CancelReason` null pre-cancel, carried across all three cancel routes (a `[Theory]`), null on a confirmed order.
- **Frontend (`client/src/orders/`)** — `orderSchema.ts` gained `CancelReasonSchema` (a closed `z.enum`, nullable on the view) + `placedAt: z.string()`; `orderStatusJourney.ts` gained `humanizeCancelReason` (an exhaustive `Record<CancelReason,string>`) + `formatPlacedAt` (a pure UTC `Intl.DateTimeFormat`) and threads the reason into `deriveJourney`'s `cancelled` branch; `OrderStatusPage.tsx` renders a `Placed …` line and the per-reason cancel copy (replacing the generic "Cancelled").

Frontend **96 tests, 0 failures** across 15 files; `tsc --noEmit` + `vite build` green. ## What worked

- **`CartView` + `CartProjectionTests` were an exact template — the backend was near-mechanical.** The "Marten's using-metadata convention" already existed in-repo (5 sites surfacing `IEvent<T>.Timestamp`), and `CartProjectionTests` already solved the pure-fold-test mechanic with its `At<T>` → `Event<T>` helper. Mirroring them de-risked the whole backend change; the production project compiled on the first build after the field addition.
- **Additive-superset = zero churn to existing consumers, and that was provable.** Because the five existing fields stayed a prefix and the two new ones were appended, the W3 confirmation read, the Orders integration tests, and the CrossBc smoke tests deserialized unchanged — green with no edits to *their* logic (only their fixtures gained the two fields). The split-the-difference between "new requirement" and "no behavioural change" held.
- **The data was already on the streams — a true read-only enrichment.** `cancelReason` folds the `OrderCancelled.Reason` the event already carried (the write aggregate ignores it); `placedAt` is the genesis event's append metadata. Neither needed an event-contract change, so there was no migration, no PMvH touch, no cross-BC ripple — the smallest possible surface for a visible feature.
- **The live boot earned its keep precisely on the one test-coverage gap.** The vitest fixtures used a clean `"…T14:02:00+00:00"`; the live System.Text.Json `DateTimeOffset` is `"…T04:26:57.5184103+00:00"` — **7-digit fractional seconds**. Seeing the real format validated the `placedAt: z.string()` decision (a strict `.datetime()` could have rejected that precision at the boundary) and confirmed `formatPlacedAt`'s `new Date(...)` parses it. Tests prove the logic; only a real boot proves the serializer.

## What was harder / notable

- **The schema-evolution trap: a forward-compat test quietly asserted the *absence* of a field I then added.** `orderSchema.test.ts` had a "drops a benign additive field rather than rejecting it" case that used **`placedAt`** as its example unknown field. Making `placedAt` a modeled field flipped that test red (the schema now *keeps* it). The fix was to switch the strip-test to a genuinely-unmodeled field (`serverNote`). Worth naming as a pattern: a `.strip()`-proving test should use a field you will *never* model, or it becomes a tripwire on the next schema addition.
- **A required-field addition propagates to every fixture.** Five `OrderStatusView`-shaped literals across four test files (`wireOrder` ×2, `placedOrder`, `orderAt`, `viewAt`) needed the two new fields, plus `placeOrderMutation.test.tsx` correctly needed *none* (its response is `{ orderId }`, not a view). Found them all by grepping `total:`/`customerId:` before they failed, rather than chasing red tests one compile at a time.
- **The `placedAt` source was a real decision settled by convention, not surfaced as a fork.** "Event metadata vs. a new `OrderPlaced` field" genuinely changes the code and the event contract — but the named `CartView`/`IEvent<T>.Timestamp` convention (and the fact that placement time *is* the event's occurrence time) made it a settled choice, not an owner's-call fork. Recording it as design.md Decision 1 (with the rejected alternative) keeps the reasoning durable without spending an `AskUserQuestion` round-trip.

## Methodology refinements

- **When recon turns a potential fork into a settled convention, act — don't fork-spam.** This is the second consecutive slice (after the Stock no-fork judgment, retro 024) where checking the in-repo authority *before* drafting an `AskUserQuestion` dissolved an apparent fork. The heuristic is firming up: *surface a fork only after confirming no ADR, named convention, or worked precedent already settles it.* The owner picked the slice at the genuine fork (the post-Stock open pick); the sub-decisions inside it were convention-bound.
- **Author the prompt + OpenSpec + narrative as siblings up front, then let the build follow them.** Because the spec delta and the narrative bump were written before the code (and the prompt named the exact files + the `IEvent<T>`/`At<T>` mechanic), the implementation was execution against a frozen plan, not discovery — the consolidated one-PR slice ([[feedback-consolidate-slice-prs]]) stayed coherent.
- **The narrative is the spec-record-of-truth; the workshop is allowed to lag by one tidy.** Closure landed in Narrative 005 v1.7 *within this PR*; the workshop § 5.1 "aspirational → shipped" flip is fenced to a post-merge tidy. § 5.1 v1.10 itself framed the workshop as "the lone straggler catching up" to the narrative, so this is consistent, not drift — but it is named, not silently dropped.
