# Retrospective: Implementations 010 — Slice 4.6 Cancel on Payment Decline (+ Inventory 2.3 release stock)

**Prompt**: `docs/prompts/implementations/010-slice-4-6-cancel-on-payment-decline.md`
**Outcome**: shipped — the project's first cancellation that crosses a BC boundary *back*. When `PaymentDecisionHandler` declines at the payment gate, it appends `PaymentAuthFailed` then `OrderCancelled { reason: "payment_declined" }` in one commit (status `cancelled`) and cascades a single `ReleaseStock { orderId, lines }` to Inventory over RabbitMQ. Inventory's new `ReleaseStockHandler` appends `StockReleased` per SKU the order still holds, and the `StockLevelView` fold restores available/reserved and drops the order from its reservations. Both sides idempotent (Orders: payment-gate guard; Inventory: per-SKU reservation presence). OpenSpec change `slice-4-6-cancel-on-payment-decline` `--strict` valid; Narrative 004 → v1.4 (Moment 5). One consolidated PR.
**Tests**: full solution green — **42 total, 0 failures**: Orders 23 (the 4.3 decline test was rewritten 1-for-1, net 0), Inventory 10 (+1 fold unit, +3 release integration), Catalog 7 (unchanged), CrossBc 2 (+1 decline→release smoke).

## What worked

- **Mirroring 4.5 (Orders) and 4.2 (Inventory) made both sides nearly mechanical.** The decline branch is the 4.5 shape (append failure commit, then `OrderCancelled`) plus one cascaded message; `ReleaseStockHandler` is `ReserveStockHandler` with the guard inverted (release where reserved, vs. refuse where short). Having both patterns in-repo meant the code was idiomatic on the first pass.
- **The collaborative message-shape fork paid off as a documented divergence, not silent drift.** The workshop literally says publish `OrderCancelled { orderId }`; the user chose `ReleaseStock { orderId, lines }` for symmetry + anti-corruption. Recording it as design.md Decision 1 (with the faithfulness note) keeps the workshop honest — the *behavior* matches, only the contract name differs, and the divergence is the same kind 4.2 already made for `ReserveStock`'s lines.
- **Verifying the cascade routing up front (again) removed the one real risk.** The docs confirmed two load-bearing facts before wiring: a `null` return suppresses the cascade, and a cascaded type with **no local handler** routes to the broker by conventional routing. Crucially, the local-vs-broker precedence trap that bit 4.3 (a *locally-handled* type) does **not** apply here precisely because `ReleaseStock` has no Orders-local handler — so the precedence rule works *for* us. No "message vanished" debugging.
- **The per-SKU `Reservations.Remove` fold is what makes idempotency fall out for free.** Because the fold drops the order id on release, a duplicate `ReleaseStock` finds no reservation and the handler's guard no-ops it — the same self-checking-the-stream idempotency the reserve path uses, with no transport-level dedup.

## What was harder / notable

- **The cross-BC decline smoke needed real isolation thought.** The shared `CrossBcFixture` boots one Orders host with the always-approve stub (the reserve smoke needs approval to reach `confirmed`). The decline smoke needs a *declining* provider, but: a second Orders host on the same broker would compete for the reply queue (and share the DB, so the wrong host could approve the order), and a second fixture with its own containers would race the **process-global** connection-string env vars against the existing fixture (different xUnit collections run in parallel). Resolution: one shared `ConfigurableStubPaymentProvider` whose `ShouldApprove` flag the decline smoke flips off and back, relying on the `crossbc` collection running serially. Keeps one Orders host, one broker, one DB, no env-var race, no competing consumers — and honors the 4.3 stub policy (the decision is test-controlled, not a magic value in the order payload).
- **Shouldly's `ShouldContain` binds to the `Expression<Func>` overload, which forbids `is` patterns.** `events.ShouldContain(e => e.Data is OrderCancelled o && …)` failed to compile (CS8122). Rewrote with `events.Select(e => e.Data).OfType<OrderCancelled>().ShouldHaveSingleItem()` then asserting the reason — clearer anyway.
- **The 4.3 decline test pinned a truth 4.6 retires.** `a_declined_payment_records_a_failure_and_does_not_confirm` asserted 3 events + `stock_reserved` (the honest deferred state). 4.6 makes it 4 events + `cancelled` + a `ReleaseStock` cascade, so it was rewritten 1-for-1 (same swapped-provider setup, new assertions + the cascade capture). In-bounds — the slice's behavior change directly invalidated it — and exactly the spec-delta closure loop surfacing as test churn that 009's retro flagged. (009's candidate habit held: this test was predictable from the design and named in the prompt's "files touched.")

## Methodology refinements

- **For a cross-BC smoke that needs a *different* decision than an existing smoke, prefer a runtime-configurable seam on the shared fixture over a second fixture.** A second container-backed fixture is the obvious move and the wrong one — process-global env vars + parallel collections = a flaky race. A toggle flag on a shared singleton, guarded by serial collection execution, is cheaper and safer. Candidate habit: when adding a smoke that contradicts an existing smoke's happy-path assumption, reach for a test-controlled seam on the existing fixture first.
- **"Document the divergence in the same artifact that records the decision" closes the workshop-drift gap.** The faithfulness note lives in the change's design.md (Decision 1) *and* is forward-referenced from the narrative and the contract's own comment. When the change archives, the workshop gets its amendment. Three places point at one decision, so neither the workshop nor a future reader silently assumes the code matches the original wording.
