# Retrospective — Implementations 021: W3 Place-Order / Order Confirmation (slice 4.1)

## Outcome summary

Built **Moment 4 of the storefront journey** — checkout. The W2 `CartPage` gained a `[ Place Order ]` button; a new `client/src/orders/` feature folder binds the **third** read surface and carries the storefront's **first non-optimistic mutation**; and a new **W3 Order Confirmation** screen reads the placed order back and shows its honest `awaiting_confirmation` status + total. The loop is now **browse → add → edit → checkout → confirmation**. This is the slice's headline: `POST /orders` is the **OpenTelemetry trace front-door**, and 4.1 is the one beat where the cart's optimistic-UI template is **deliberately refused**.

- **`client/src/orders/orderSchema.ts`** — `OrderStatusViewSchema`, the **third** per-read-model schema against the **third** service surface (Orders' `OrderStatusView`). `status` is a closed `z.enum` of the five `OrderStatus` constants (locked decision 3) — an unexpected status fails loud at the boundary. `OrderLineSchema` is defined locally (structurally identical to `CartLineSchema`, but a distinct contract — the backend mirrors this with a separate `OrderLine` record; sharing would couple two independent wire surfaces).
- **`client/src/orders/orderQueries.ts`** — `orderQueryOptions(orderId, ctx)` keyed by `orderId` (not customer — an order is resolved by its id; the contrast with `cartKeys.mine`). A `404` here is a **genuine error**, not the cart's empty-state mapping — an order the SPA just placed must exist.
- **`client/src/orders/placeOrderMutation.ts`** — `usePlaceOrder`, the **first non-optimistic mutation**: no `onMutate`, no pure-merge sibling — just `mutationFn` (`POST /orders { customerId }`) + `onSuccess` (invalidate `cartKeys.mine` → badge resets to `Cart (0)`, then navigate to W3). Identity is **body-keyed** here — the **third** transport shape beside the header-keyed read and route-keyed cart commands.
- **`client/src/orders/OrderConfirmationPage.tsx`** — a **pure** component taking `orderId` as a prop (the route reads the param and passes it in, keeping the page router-free and trivially testable). Renders the § 5.1 W3 wireframe: "Order placed" in a `role="status"` region, the order id, the status as **text** (humanized, not color-only), the server-computed total (read off the view, not recomputed), and a **disabled** `[ Track this order ]` (W4 deferred — an honest control, not a broken link).
- **`client/src/cart/CartPage.tsx`** — `[ Place Order ]` after the table (primary-token button), disabled while `isPending` ("Placing…"), an honest `role="alert"` place-failed message on a 409. Renders only in the populated-cart branch, so an empty cart offers no checkout (the workshop's `CartEmpty` guard made unreachable from the UI).
- **`client/src/router.tsx`** — the `/orders/$orderId/confirmation` route; a thin route component reads `route.useParams()` and hands `orderId` to the pure page.

**Tests**: client — **69 vitest tests, 0 failures** (13 new: 4 `orderSchema` [valid parse / `z.enum` rejects an unexpected status / strips an additive field / rejects `total`-as-string], 3 `orderQueries` [200 parses + header set / 404 propagates as NotFoundError / `orderKeys` by id], 2 `placeOrderMutation` [success POSTs `{ customerId }` + invalidates cart + navigates with orderId / 409 surfaces + no navigate], 3 `OrderConfirmationPage` [renders status+total / error state / disabled Track], 1 `CartPage` [`[ Place Order ]` POSTs `{ customerId }` to `/orders` + enters pending]). `npm run build` (tsc --noEmit + vite build, **241 modules**) green. **No backend code, no `client.ts`, no OpenSpec/workshop change.**

**Live wire verification (this session, not deferred)**: booted the Aspire stack and exercised the full flow against a seeded `customer-demo` cart (2× crit-001 @ 24.99 + 3× crit-002 @ 18.00). `POST /orders` → **201**, `Location: /orders/{guid}`, body keys = **`orderId` only** (the load-bearing contract correction confirmed at the wire — no `status`). `GET /orders/{orderId}` → **`status=awaiting_confirmation`, `total=103.98` (on the view), 2 lines** — exactly what W3 renders. `/carts/mine` after → **404** (the checked-out cart is gone → badge → `Cart (0)`). The **cross-BC fan-out is verified by status transition**, not just the HTTP front-door: the first order (Catalog seeded, **Inventory not**) converged `awaiting_confirmation → cancelled` (`ReserveStock` reached Inventory over RabbitMQ, failed for no stock, `OrderCancelled` folded back; the Bruun awaiting-payment list dropped to 0); after seeding Inventory stock, a throwaway happy-path order reached **`confirmed`** (the full chain `OrderPlaced → ReserveStock → StockReserved → PaymentAuthorized → OrderConfirmed`). The OTel span-chain **visual** in the Aspire dashboard is the owner's pass; the cross-service message flow it traces is confirmed here.

**Spec movement**: Narrative 005 → **v1.5** (Moment 4 built; **the `{ orderId, status }` → `{ orderId }` + follow-on-read reconciliation landed** — the load-bearing correction the handoff flagged); frontend skill → **v6** (Convention 2 gains the 3rd schema/3rd surface + the `z.enum` drift-guard; Convention 3 records its named exception realized in code with the "apply when knowable, refuse when not" lesson; Convention 4 gains the body-keyed third transport; the client-entry map gains `orders/`). No OpenSpec/workshop change — a screen-only bind of two existing contracts.

**Three forks resolved with previews before any code** (locked in the prompt): W3-alone scope (not W3+W4), the mutation's home (`orders/`, it produces an order), and `status` as `z.enum` (a domain enum used as a boundary guard).

## What worked

- **The non-optimistic mutation is cleaner than the optimistic three.** No `onMutate`, no rollback context, no pure-merge sibling — `usePlaceOrder` is `mutationFn` + `onSuccess`. The *absence* of the optimistic machinery is the teaching point, and it made the hook the simplest mutation in the codebase. The two-test coverage (success → posts/invalidates/navigates; 409 → errors/no-navigate) is complete because there is no optimistic state to assert.
- **Keeping `OrderConfirmationPage` pure (orderId as a prop) paid off in the test.** The route does the param-reading; the page is a plain component. Its test needs only the query + identity providers — no `RouterProvider`, no memory-history scaffolding — and the disabled `[ Track this order ]` (a button, not a `Link`) keeps it router-free entirely. The route's self-referencing component (`route.useParams()`) is the canonical code-based idiom and type-checked clean.
- **The contract correction held up exactly as the handoff predicted, and the wire proved it.** `POST /orders` returned a body with a single key, `orderId` — so W3 genuinely must read the order back. The narrative reconciliation isn't a guess; the live response has no `status` field. The follow-on `GET` returns `awaiting_confirmation` with the total on the view, so W3 renders the view directly (no integer-cents re-sum like CartPage — the server did the arithmetic).
- **Verifying the cross-BC flow by status transition, not just the front-door.** Placing once with no Inventory stock and watching the order converge to `cancelled` (then seeding stock and watching another reach `confirmed`) proves the `ReserveStock` round-trip over RabbitMQ actually executed — the span chain's message flow, confirmed headlessly. The two outcomes from the same front-door are the concrete case for why optimism stops at 4.1.
- **`z.enum` as a boundary guard read well in the test** — `OrderStatusViewSchema.parse({ …, status: "shipped" })` throws, which is the located, loud failure Convention 2 is about. The teaching value (status is a known domain set) is visible in one assertion.

## What was harder / notable

- **The prior session's stack didn't just fail to survive — it left orphans that blocked the build.** The handoff said the W2 stack "will be torn down." It wasn't: five `CritterMart.*` processes from 12:08 PM were still running, **locking the build output DLLs** (`MSB3026: file is locked by CritterMart.Inventory/Catalog/CritterWatch`), so the first `dotnet run` exited 1 mid-build — while the orphaned Orders still answered `/health` 200, masking the failure. Diagnosis: read the AppHost log (the lock warnings named the PIDs), `Get-Process -Name CritterMart.*` (five orphans, no Aspire containers — `docker ps` empty), `Stop-Process` all five, reboot clean. This sharpens retro 020's lesson: don't just re-verify the stack is up — **a half-dead prior stack can actively block your build**, and a leftover service answering `/health` is not the same as a working datastore (the containers were gone).
- **W3 reads once, so the demo needs Inventory stock or every order cancels.** ADR 015 R5 — no poll on W3 (that's W4). The screen shows whatever status is current at read time; right after placement that's `awaiting_confirmation`, but the cross-BC outcome lands shortly after. Without Inventory stock seeded, that outcome is `stock_unavailable → cancelled`. For the owner's happy-path browser/OTel demo, Inventory stock for the cart's SKUs must be seeded first — surfaced and handled here (the stack is left seeded).
- **Two structurally-identical schemas, deliberately not shared.** `OrderLineSchema` is byte-for-byte `CartLineSchema`. The pull to `compose-shared-schemas` (the zod skill) is real, but the backend keeps `CartLine` and `OrderLine` as separate records on separate aggregates, and Convention 2's premise is that each wire surface drifts independently. Defining it locally in `orders/` keeps the feature self-contained and the two contracts free to diverge — the right call, but worth naming since it looks like duplication.

## Methodology refinements

- **"Apply optimism when the outcome is knowable; refuse it when it isn't" is now a named lesson in the skill.** The three cart commands + this exception teach it together — Convention 3 records both halves. The non-optimistic shape (mutationFn + onSuccess navigate/invalidate, no onMutate) is templated for any future command whose outcome the SPA can't predict.
- **A third identity transport closes the picture — and sharpens the harmonization debt.** Round one now has header-keyed (read), route-keyed (cart commands), and body-keyed (place-order) identity, all off the one `useCurrentCustomer` seam. The future Polecat-promotion tidy reconciles all three; Convention 4 now names the full set. The debt is wider but better-described.
- **The route-component-reads-param → pure-page-takes-prop split is the pattern for any param route.** It keeps the page testable without router scaffolding. Worth carrying to W4 (`/orders/$orderId`) and any future detail screen.
- **Design-return cadence — now due.** 4.1 is the **third consecutive frontend implementation** since the #59 ADR design-return (#60 W1, #61 W2-edits, this). It self-interleaves Narrative 005 → v1.5 + skill → v6, so it's soft-OK — but per the cadence rule, the **next** PR should be a design-return, not a fourth straight implementation: candidates are W4 tracking (which is itself partly design, reusing this folder), the ADR 020/021 backend rollout to `Ordering/Order` + `StockLevel`, or a tidy sweep.
