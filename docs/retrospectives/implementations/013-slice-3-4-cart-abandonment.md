## What shipped

- **The fire-and-check automation.** `Cart/CartActivityTimeout.cs` (self-message + `CartActivityDeadline`), `Cart/CartAbandonmentHandler.cs` (terminal guard → activity check → reschedule or abandon), scheduled by `AddToCart` **only on cart creation** — exactly one timeout in flight per open cart. The 3.2/3.3 edit handlers were **not touched**: under fire-and-check, their event timestamps are the refresh (the deferred-twice "refresh `CartActivityTimeout`" clauses dissolved rather than landed — design.md faithfulness note 4).
- **The activity clock on the fold.** `CartView.LastActivityAt`, folded from `IEvent<T>` append timestamps (the codebase's third `IEvent<T>` metadata fold) — what the handler's decision reads, honoring 4.7's "the stream decides, not the todo-list" principle.
- **The todo-list.** `CartAwaitingActivity` + `CartsAwaitingActivityProjection` (instance-registered, inline, conditional deletes on both terminal events) + the GET endpoint — the cart-side mirror of `OrdersAwaitingPayment`.
- **The async teaser.** `CartAbandonmentDailyReport` + `CartAbandonmentReportProjection` — `MultiStreamProjection` keyed by abandonment day via `Identity<IEvent<CartAbandoned>>`, folding count + value + per-SKU tallies from the fat event. Registered `ProjectionLifecycle.Async`, no `AddAsyncDaemon`; the integration test proves empty-until-rebuilt then correct-after-rebuild.
- **Time as a dependency.** `TimeProvider` injection (design.md Decision 8, decided during implementation): the handler reads "now" from DI; the test fixture's settable clock makes both the abandon and reschedule paths deterministic with zero real-time waits.
- **OpenSpec + narrative + skills debt.** 1 MODIFIED (add-item gains schedule-on-creation) + 3 ADDED requirements; Narrative 004 v1.7 Moment 1B; `docs/skills/DEBT.md` created with the Marten-pattern third-use row (prompt locked decision 8).

## What worked

- **Resolving the forks before freezing the prompt.** All three user forks (scheduling policy, report shape, daemon scope) plus the spec-delta question were presented with previews and resolved *before* prompt 013 was authored — so the prompt froze decisions, not options, and nothing blocked mid-session. This inverts the 012 pattern (fork resolved mid-session) and is strictly better when the forks are known at session start.
- **ctx7 verification reshaped a fork instead of just confirming an API.** The scheduled-message-cancellation check didn't verify an implementation detail — it **invalidated one of the two named options** (cancel-and-reschedule is unimplementable; the workshop's "Wolverine supports both" is wrong). Verify-before-wiring caught a *design* error, not a wiring error. The fork presented to the user was honest because the verification ran first.
- **The 4.7 mirror carried ~70% of the session.** Message + deadline record, handler shape, todo-list projection, config + singleton, instance registration, direct-invocation test strategy — all mechanical transfers. The genuinely new work (multi-stream projection, rebuild-on-demand, TimeProvider) was exactly the part the prompt's verification group and the user forks had pre-cleared.
- **Marten's documentation matched our design choices verbatim.** Both verifications (daemon-less rebuild, date-keyed multi-stream identity) landed on patterns Marten documents as canonical examples — the rebuild integration test is a near-transcription of Marten's own projection-testing guide.

## What was harder / notable

- **The session's one failure: a missing `partial` keyword.** `CartAbandonmentReportProjection` was first declared without `partial`; the host refused to boot (`InvalidProjectionException: No source-generated dispatcher found… there is no runtime fallback`), failing all 38 integration tests at once. Marten 9 dispatches conventional `Apply` methods via a **compile-time source generator** that needs a `partial` class to extend. The fix was one word; the error message itself named the convention. The single-stream projections all had `partial` (copied from precedent); the multi-stream one was written fresh and missed it. **Recorded in DEBT.md row 1** as exactly the nuance a local skill note would have prevented.
- **The prompt's file list missed a required test-infrastructure change.** Deterministically testing both fire-and-check paths (abandon vs. reschedule) requires controlling the clock the handler reads — which required amending `OrdersAppFixture.cs` (a settable `TestTimeProvider` overriding Program.cs's `TimeProvider.System`). The prompt listed the fixture as read-only orientation. This is a **deliberate, recorded divergence** (design.md Decision 8), not an opportunistic edit: the change serves the prompt's named deliverable (integration tests with no real-time waits) and nothing else.
- **The fat event was decided before code but is bigger than the workshop models.** `CartAbandoned { reason }` → `{ reason, lines, totalValue }`. Without it the daily report would need a custom grouper with database lookups. The post-merge workshop amendment must record this divergence (faithfulness note 3) alongside the § 8 factual correction.

## Methodology refinements

- **Run the ctx7 verifications that shape a fork *before* presenting the fork.** A fork option that turns out to be unimplementable wastes the user's decision. (This session's scheduled-message-cancellation check; generalizes the verify-before-wiring rule from "before code" to "before forks".)
- **When a slice's tests need to manipulate time, inject `TimeProvider` and override it in the fixture** — never wait real time, never shrink config windows to milliseconds. Candidate for the local Marten/Wolverine skill note (DEBT row 1).
- **Convention-method Marten projections must be `partial` — put it in the registration comment.** The Program.cs registration comments now say it; the future skill note should lead with it.
