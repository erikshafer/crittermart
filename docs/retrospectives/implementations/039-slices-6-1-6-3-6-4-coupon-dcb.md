## What worked

- **The DCB spike paid for itself immediately.** ADR 024 and the workshop named the append mechanics as an open call (§8 item 1); design.md committed to *spike-first, then lock*. The throwaway `DcbCouponSpike` (deleted after) proved against real Postgres that `RegisterTagType<CouponId>("coupon").ForAggregate<CouponUsage>()` is the **entire** DCB opt-in (no `EnableDcb()` — the symbol the proposal had *guessed*), that `session.Events.Append(orderId, evt.WithTag(...))` lands the tag on the real order stream, and that the race throws `DcbConcurrencyException`. Guessing any of those in shipped code would have cost far more than the 30-line spike.
- **DEBT row 3's Polecat-symbol trap was real and avoided.** The installed DCB skills present `[BoundaryModel]` / `IEventBoundary<T>` / `EventTagQuery.For(…)`; the actual Marten 9.15.1 surface is `[BoundaryAggregate]` / `EventTagQuery().Or<T>()` / `FetchForWritingByTags`. Verifying against the resolved assembly + current docs (not the skill) was load-bearing.
- **The no-coupon path stayed byte-for-byte.** Keeping slice 4.1's injected-session + tuple + outbox pattern untouched and diverting only the coupon path to its own transaction meant all 116 existing Orders tests passed unchanged once the reset was fixed. The `OrderPlaced` 4-arg convenience constructor (`Subtotal == Total`, `Discount == 0`) kept every legacy call site correct.
- **6.4 was a fold + a shared 3-line helper, not a new handler** — exactly as the workshop modeled it *(system)*. `Order` gained `CouponId?`; `session.AppendCouponRelease(orderId, aggregate.CouponId)` rode each existing `OrderCancelled` append. At-most-one-release fell out of the terminal-once guard for free.

## What was harder / notable

- **DCB optimistic concurrency is cap-BLIND — the session's biggest correction.** The initial design (a boundary pre-check, mapping the rare commit-conflict to a 409) passed every test *except* the 6-racer burst: only **1 of 6** succeeded against cap 3. `FetchForWritingByTags` arms an assertion on the coupon's whole tag-set, so *any* concurrent redemption invalidates a commit even under the cap. The fix is Marten's own documented answer — **reload and retry** (`RedeemWithDcbAsync`, a fresh session per attempt) — which admits exactly `cap`. design.md decision 3, the proposal, and the spec race scenario were all revised from "no-retry + exception-map" to the retry loop. Lesson: the workshop's "the losing handler retries" language was not decorative; a bare pre-check is a real under-admission bug that only a concurrency test surfaces.
- **`AutoApplyTransactions` owns the commit, so the retry could not use the injected session.** Wolverine.Http's transaction middleware calls `SaveChangesAsync` *after* the handler returns, so a handler-local try/catch around a DCB commit is impossible on the injected session. The retry therefore uses fresh `LightweightSession`s and publishes cascades post-commit via `IMessageBus` (a documented, accepted divergence from the transactional outbox — the order is durably placed; `ReserveStock`/timeout are at-least-once safety nets). This is the one place the coupon path structurally diverges from slice 4.1.
- **Marten 9.15.1 `DeleteAllDocumentsAsync` trips over the id-less boundary aggregate.** Once `CouponUsage` (`[BoundaryAggregate]`, intentionally no `Id` per Marten's own guidance) is materialized, `DeleteAllDocumentsAsync` enumerates it to build a document schema and throws `InvalidDocumentException` **store-wide** — breaking *every* test class's reset in a full run (21 failures until fixed). Worked around with a boundary-aggregate-safe `OrdersAppFixture.ResetAllDataAsync()` that TRUNCATEs the `orders` schema tables directly (also clearing `mt_event_tag_coupon`, which `DeleteAllEventDataAsync` leaves behind). This is an **upstream Marten rough edge**, not a project bug — a candidate for a Marten issue (see Outstanding).
- **The optional coupon rides a query parameter, not a body.** Wolverine.Http 400s a bodyless `POST /orders` when a body record parameter is declared, which would break the existing no-body checkout contract (and the W3 screen). Binding `string? couponCode` as an optional `?couponCode=` query parameter preserved the contract; the narrative/proposal's "`PlaceOrder { couponCode }`" is the domain command, transport-agnostic.

## Methodology refinements

- **When an ADR/workshop names an "open mechanics call," spike it before the proposal ships symbols.** The proposal named `EnableDcb()`/`TagEvent` from stale docs; the spike replaced them with the verified `RegisterTagType`/`WithTag`, and the proposal/tasks/design were corrected in-session. Prefer "author the proposal with a *flagged* mechanic, spike at implementation start, then reconcile the spec" over guessing symbols into a spec that reads authoritative.
- **A concurrency invariant needs a concurrency test, not just a sequential one.** Every sequential test passed with the buggy pre-check; only `concurrent_redemptions_never_exceed_the_cap` (Task.WhenAll of N racers vs cap K) caught the under-admission. Any future DCB/optimistic-concurrency slice should ship a burst test as the invariant's real proof.
- **Boundary aggregates are a distinct Marten citizen — treat their test-cleanup separately.** The id-less `[BoundaryAggregate]` breaks `Clean.DeleteAllDocumentsAsync`; a SQL-truncate reset is the portable workaround. Worth an upstream-skill note alongside DEBT row 3's availability correction.
