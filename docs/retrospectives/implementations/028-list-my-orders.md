# Retrospective — Implementations 028: List my orders (a customer-keyed order list)

## Outcome summary

Shipped the **"My Orders" list**, closing **Gap #3** — the last named round-two storefront deferral. A customer-keyed read across the customer's orders (`GET /orders/mine`) plus the list screen that binds it, extending the journey from "track this order" (W4) to "see all my orders." The defining property: it is **not new modeling** — a filtered query over the existing inline `OrderStatusView` documents, **no new event, command, projection, or aggregate**.

- **`Features/ListMyOrders.cs`** — new `GET /orders/mine`: `[FromHeader(Name="X-Customer-Id")] string? customerId` + `IQuerySession`; blank header → `Results.BadRequest` (400); else `Query<OrderStatusView>().Where(v => v.CustomerId == customerId).OrderByDescending(v => v.PlacedAt).ToListAsync()` → `Results.Ok`. A faithful sibling of `ViewMyCart` (same transport, same inline guard, same `IResult` shape).
- **`Program.cs`** — one line: `Schema.For<OrderStatusView>().Index(x => x.CustomerId)`, a **non-unique** read-model index mirroring the `CartView` one (`Program.cs:100`); not unique, because a customer has many orders (the contrast with the cart's one-open-cart write-side invariant).
- **`ListMyOrdersTests.cs`** — 5 Alba tests: newest-first; strict customer scoping; empty `200 []`; missing-header `400`; and a **seeded** confirmed+cancelled pair (terminal states + `cancelReason`) — seeded directly via `StartStream<Order>` because the terminal states aren't reachable through the Orders-only host (the cross-BC saga is absent).
- **Frontend (`client/src/orders/`)** — `OrderListSchema = z.array(OrderStatusViewSchema)` (the order contract reused wholesale); `orderKeys.mine(customerId)` + `fetchMyOrders` + `myOrdersQueryOptions` (mirroring `cartKeys.mine`); `MyOrdersPage.tsx` (loading/error/empty/list states, each row a typed `<Link>` into the W4 track screen, a cancelled row naming its reason via the reused `humanizeCancelReason`); the `/orders` route + an AppShell "My Orders" nav link.

**The one genuine fork — put to the owner.** Route/identity transport: **header-keyed `/orders/mine`** vs query-string `?customerId=`. This was an outward API contract that *contradicts* the workshop Gap #3 text and nudges the parked cart-identity-harmonization candidate, so it cleared the no-fork bar. The owner chose `/orders/mine` (mirrors `/carts/mine`, rides the ADR 009 `useCurrentCustomer` seam, keeps identity out of the URL). The sub-decisions inside the slice (reuse `OrderStatusView` vs. a new projection/DTO; the index; the `IResult`+guard shape) were **settled by precedent, not forked** — `BrowseProducts` (list) + `ViewMyCart` (customer-keyed read).

## What worked

- **Recon dissolved the fancier design before a line was written.** The preview I floated at the open-pick sketched a multi-stream `CustomerOrdersView` projection. Reading `BrowseProducts` (a list = `Query<Product>()`) and `ViewMyCart` (a customer-keyed read = `Query<CartView>().Where(customerId)`) showed the "My Orders" list is exactly their composition over the *existing* `OrderStatusView` — no new projection earns its keep. The no-fork heuristic fired correctly: a worked precedent beat the clever option, and the slice shrank to one endpoint + one index.
- **Contract reuse, end to end.** `OrderStatusView` was already the order read shape (no `id→sku`-style rename to force a DTO), so the backend returns `OrderStatusView[]` and the frontend reuses `OrderStatusViewSchema` (wrapped `z.array`) and the `orderStatusJourney` formatters wholesale. Net-new code was a query, a key, and a list layout — the list row and the W4 detail screen bind one contract, so the row `<Link>` is trivially consistent.
- **The live boot proved what the Alba host structurally can't.** The Orders-only test host can't drive a real `confirmed` order (the cross-BC saga goes to an absent Inventory in solo mode), so the integration tests cover the reachable paths + a *seeded* terminal case. The live boot is the only place orders that walked the real saga to `confirmed` appear in the list — and they did, newest-first, scoped, with the real camelCase wire shape. Tests prove the query; only a boot proves the whole stack (real index creation on startup, real serializer, real saga).
- **The fork was genuinely the owner's and genuinely narrow.** Header vs query-string is an outward contract that pre-commits the cart-identity-harmonization direction — exactly the kind of decision the collaborate-on-decisions feedback reserves for the owner. Presenting it with concrete endpoint previews (not an abstract preference) made it a one-tap confirm.

## What was harder / notable

- **Terminal-state coverage needed a different tool than the happy paths.** `confirmed`/`cancelled` aren't HTTP-reachable in the single-service Alba host, so asserting "the list includes terminal orders with their `cancelReason`" required seeding events directly (`store.LightweightSession()` + `StartStream<Order>(id, OrderPlaced, …, OrderConfirmed)` / `…, OrderCancelled(reason)`), relying on the inline projection materializing the view on `SaveChangesAsync`. Worth naming: when the cross-BC saga is out of the host's reach, seed the stream and let the inline projection do its job — don't contort the HTTP surface to fake a terminal state.
- **A page with TanStack Router `<Link>`s needs a memory-router test harness, not a bare render.** Unlike the pure `OrderStatusPage` (orderId as a prop), `MyOrdersPage` holds `<Link>`s (rows → W4, empty → browse), so its test mirrors `OrderConfirmationPage.test.tsx`: a throwaway `createRouter` + `createMemoryHistory` hosting the page at `/orders` with stub target routes, so links resolve and a click navigates. The in-repo precedent made this mechanical, but it's a real "this component is routed" tax that the prop-pure pages avoid.
- **An AppShell edit briefly duplicated its closing tags.** Wrapping the nav + badge in a `<nav>` while keeping the existing `<CartBadge />` produced a duplicated `</header><main><Outlet/>` block on the first edit; the `vite build` would have caught it, but a re-read caught it first. A reminder that structural JSX edits around existing siblings want a re-read, not just a green build assumption.

## Methodology refinements

- **The no-fork heuristic now cuts *both* ways in one slice — and that's the mature shape.** This session surfaced exactly one fork (the route shape, which has an ADR-adjacent contract consequence) and suppressed two non-forks (the projection-reuse and the index, both settled by worked precedent). Prior slices demonstrated suppression (Stock retro 024, enrich retro 025); this one shows the discipline isn't "never ask" — it's "ask only where no authority settles it, and there ask well." The fork that *was* surfaced was the one where the canonical spec (the workshop) and the in-service convention disagreed — spec-vs-convention disagreement is itself the signal that a decision is genuinely the owner's.
- **Fold the design-return content into the slice when the supersession is load-bearing to the contract.** Slice 025 fenced its workshop flip to a post-merge tidy (the narrative was the record-of-truth, the workshop could lag). This slice authored the workshop § 5.1 v1.12 amendment **in-PR**, because the `?customerId=`→`/orders/mine` supersession is part of the contract this PR ships — leaving the workshop saying `?customerId=` while the code ships `/orders/mine` would be active drift, not a one-tidy lag. The rule firming up: a workshop note that *records a superseded sketch* rides the PR that supersedes it; a workshop note that merely *flips aspirational→shipped* can lag by a tidy.
- **The cadence interleave is now DUE — and named, not silently spent.** This is the 3rd implementation since the #68 interleave. The narrative + workshop edits here are this slice's *own* design-return content, **not** the cadence interleave; the next PR must be the interleave proper (the still-fenced §6.1 flip + `openspec archive harden-add-to-cart-snapshot`, or a new narrative).
