**Prompt**: `docs/prompts/implementations/015-slice-3-5-view-open-cart.md`
**Outcome**: shipped — the **first frontend implementation slice**. `GET /carts/mine` resolves the customer's single open `CartView` **by identity** (`X-Customer-Id` header, behind the ADR 009 `useCurrentCustomer` seam) over the partial-unique open-cart index that already existed — `200` with the cart, `404` for no open cart (cold start / checked-out / abandoned), `400` for no identity. **No new event, projection, or index**: it exposes the same open-cart resolution every cart *command* already runs (`AddToCart.cs:31`) as a read, closing the pre-frontend audit's blocking **Gap #1** — the cart-review screen (W2) can now render on a cold load. The deferred **`docs/skills/frontend/SKILL.md`** landed as a v1 seed (the first non-`core` skill cluster). **Tests**: full solution green — **91 total, 0 failures** (+4, all Orders): the new `ViewMyCartTests` suite — resolved-by-identity happy path (a second customer's cart present, to prove it is keyed not arbitrary), no-open-cart `404`, checked-out-cart `404`, missing-identity `400`. ## What shipped

- **The endpoint.** `Features/ViewMyCart.cs` — `[WolverineGet("/carts/mine")]`, `[FromHeader(Name = "X-Customer-Id")] string?` → blank guard (`400`) → `session.Query<CartView>().Where(v => v.CustomerId == id && v.IsOpen).FirstOrDefaultAsync()` → `404`/`200`. `IResult` return, matching the sibling cart-read endpoints. **`AddToCart.cs` and `Program.cs` untouched** (new feature file; no opportunistic edits).
- **The integration suite.** `ViewMyCartTests.cs` — four Alba scenarios, the happy path deliberately seeding a *second* customer's open cart so a passing test proves customer-keyed resolution rather than "return any open cart." Checkout (the reachable terminal) drives the `404`-after-terminal case; abandonment (3.4) is noted as behaving identically.
- **The deferred frontend skill.** `docs/skills/frontend/SKILL.md` — six CritterMart-specific conventions (version pins; Zod at the three wire boundaries; optimistic-UI + rollback; the `useCurrentCustomer` → `X-Customer-Id` seam; no-BFF/CORS-dev=prod/no-proxy; TanStack Router + presentation-state guardrail), each deferring library mechanics to the installed per-library skills. Authored as an explicit **v1 seed** with `[planned]` markers, since `client/` does not yet exist. README index row added (new `frontend` cluster).
- **OpenSpec + narrative.** `shopping-cart` +1 ADDED requirement (*Read the Customer's open cart*, 3 scenarios); Narrative 005 → v1.1, "Forthcoming" moved from *modeled, not yet built* → *partly built (keystone read shipped, screen pending)*.

## What worked

- **Resolving the one fork before writing the endpoint.** The identity-transport choice (header vs. query param) was presented with rich previews — endpoint signature, the frontend seam's call site, and the *Polecat promotion path* for each — and resolved to the header before code. The previews made the deciding factor (which transport makes the eventual claim-swap a localized change vs. a call-site sweep) visible, not buried in prose.
- **The skill surfaced a real design correction, not just API syntax.** `wolverine-http-marten-integration` made it explicit that `[ReadAggregate]`/`[Aggregate]` resolve by *stream id* — the exact thing a cold load lacks. That turned "use the read-aggregate attribute" into "this must be a customer-keyed query," which is the whole reason Gap #1 exists. Loading the named skill first paid for itself.
- **ctx7-before-wiring ahead of a slow test run.** Both library unknowns — Wolverine's `[FromHeader]`/query binding and Alba's `WithRequestHeader` — were verified against current docs *before* the first Testcontainers run, so the full build+test cycle ran once and green rather than round-tripping on a wrong method name.
- **The slice was as clean as the audit predicted.** Pure additive: one endpoint file, one test file, zero edits to existing code. The near-zero regression surface is exactly why 3.5 was chosen as the first frontend slice.

## What was harder / notable

- **Authoring a frontend skill before frontend code exists.** The skill was *deferred* precisely so it would document real conventions, not guesses — but this slice is backend-only, so there is no `client/` to document from. Resolved by writing it as a **v1 seed**: grounded in the locked ADR conventions (which are real, not speculative) plus the one concrete integration this slice establishes (the `GET /carts/mine` consumption), with `[planned]` markers on the code snippets that await their first precedent. The alternative — writing throwaway client code just to justify the skill, or deferring the skill a second time — would have been worse on both counts. Worth watching: when the bootstrap PR lands, the `[planned]` markers must be replaced with real `client/` file references, or the skill drifts.
- **A guard beyond the workshop's GWTs.** The workshop modeled two paths (happy + no-open-cart); header binding defaults a missing header to null, so a third path — missing identity → `400` — was needed to keep the `404` ("no open cart") semantically crisp. Added as a third spec scenario and recorded as design.md faithfulness note 2 for the post-merge workshop amendment.
- **A scope fork the kickoff implied but did not state.** The handoff said the slice was "the natural moment to also stand up `client/`" *or* a dedicated bootstrap PR. Reading the explicit deliverable list (OpenSpec + endpoint + skill) against the disciplines (one-slice-one-PR, the *owed* Aspire-integration decision deserving its own session) settled it: defer `client/`. Acted on the obvious option and flagged it rather than asking — but it is named loudly here and in the proposal/narrative so it cannot be lost.

## Methodology refinements

- **A frontend slice whose first increment is backend-only still bumps the journey narrative — to "partly built," not "built."** The two-lens narrative split (002/004 behavior, 005 screen) lets the spec-delta closure loop record "the server half of Moment 3 shipped; the screen is pending" honestly, rather than waiting for the full screen to bump anything.
- **Defer a skill until the conventions are real, but don't defer it past the first slice that can ground *one* convention.** A "v1 seed" with `[planned]` markers, grounded in the ADRs + the single integration the slice establishes, beats both speculative-skill and deferred-again. The deferral's purpose (avoid guessing) is served by the markers, not by absence.
- **Run the previews on the *consequence* of a fork, not just its surface.** The identity fork's previews compared the Polecat-promotion path of each option, which is where the decision actually lives — surface mechanics (both are one-liners) would not have distinguished them.
