## What worked

- **The three-fork AskUserQuestion up front.** Layout (flat vs monorepo), CORS wiring (AppHost-injected vs appsettings), and PR structure were presented with concrete previews and resolved before any code — so the whole session ran without a mid-stream interrupt. The flat-app + AppHost-injection choices made the wiring notably smaller than the sibling's monorepo+proxy shape.
- **Mirroring the sibling's exact pins.** Copying CritterBids' version pins (same registry, same week) meant `npm install` resolved 146 packages with **0 vulnerabilities on the first try** and a clean lockfile — no version-floor guessing. The only added dep, `@types/node`, surfaced naturally from the build.
- **ctx7 verify-before-wiring paid off again.** The Aspire docs confirmed the exact `AddViteApp(...).WithEnvironment("VITE_*", svc.GetEndpoint("http"))` + `import.meta.env.VITE_*` pattern — i.e. the official no-proxy idiom is precisely ADR 018's posture — and the Alba docs confirmed `Header(...).SingleValueShouldEqual` / `.ShouldNotBeWritten`, so both the AppHost wiring and the CORS tests compiled first try.
- **Stacking PR #52 on #51.** Both touch the README counts; stacking kept each diff conflict-free and let the tidy stand on its own as a reviewable `tidy:`.

## What was harder / notable

- **TS6 strictness bit twice.** The current Vite TS template defaults tripped the build: `baseUrl` is deprecated (use bare `paths`), and `erasableSyntaxOnly` forbids constructor *parameter properties* (the `readonly status` shorthand) — rewritten to an explicit field. Both are worth remembering for the next frontend session.
- **Alba has no `OPTIONS` verb helper.** A literal preflight (OPTIONS + `Access-Control-Request-Method`) isn't a first-class Alba scenario. The CORS test instead issues a cross-origin GET with an `Origin` header and asserts `Access-Control-Allow-Origin` — the *same allowlist* a browser preflight validates (no-credentials, AllowAnyMethod/Header policy), with a negative case proving the filter actually filters. Honest substitute, named as such.
- **The apparent CORS reference cycle.** SPA needs the service URLs; services need the SPA origin. Resolved by declaration order + a one-directional `WaitFor` (storefront waits for services, never the reverse) + a pinned port so the origin is knowable regardless of resolution order. Documented inline in `Program.cs` so the next reader doesn't "fix" it into a deadlock.

## Methodology refinements

- **A skill authored before code must be converged the moment code exists, in the same PR.** The frontend skill's own rule ("code wins, skill gets a DEBT row") had no code to lose to until now. Converging the `[planned]` markers here — rather than deferring — is what keeps a pre-code seed from ossifying into fiction. Pattern: any `[planned]`/seed artifact names its convergence trigger, and the PR that hits the trigger converges it.
- **Headless verification ceiling, restated and partly lifted.** Retro 011 flagged that per-service CORS couldn't be headless-verified without the dashboard. The lift: assert it *in-process* via Alba with the origin injected through the production config key — no dashboard, no live browser, and it tests the real config path. What still can't be headless-verified is the *browser* round trip (the actual cross-origin fetch from the running SPA) — that needs the Aspire dashboard + a browser, deferred to the W2 screen's manual verification.
- **Mirror a known-good sibling's pins verbatim rather than guess floors.** For a fast-moving JS stack, copying a sibling project's exact caret ranges (same registry, same timeframe) eliminates the "no matching version" install-failure class that guessing floors invites.
