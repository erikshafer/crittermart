# Retrospective — Implementations 036: Slices 5.5–5.7 — Identity email-change saga (CritterMart's second convention `Wolverine.Saga`)

## Outcome summary

Landed CritterMart's **second** convention `Wolverine.Saga`: a `CustomerId`-keyed `EmailChange` saga, EF-Core-backed, that opens on `RequestEmailChange` (5.5), applies or rejects on `ConfirmEmailChange` within the window (5.6), and drops the pending change on `EmailChangeTimeout` (5.7). Unlike Saga #1 (`Replenishment`, Marten-backed, system-triggered only), every trigger except the timeout is customer-facing HTTP — genuinely new ground with no in-repo precedent, and the session's central lesson is that this ground needed real execution to map, not just documentation.

The session ran the full triangle in one sitting: resolved an explicit sequencing fork with Erik (consolidate into one PR vs. mirror Saga #1's two-PR split — consolidation won), amended Workshop 002 to v1.1, closed a `structural-constraints.md` pairing gap an independent design review (Fable 5) surfaced, authored the full OpenSpec change (proposal + `customer-registry` spec delta + design.md + tasks.md, `openspec validate --strict` green throughout), wrote Narrative 009, wrote this session's own prompt, implemented the saga, and live-verified against the real Aspire stack. All 161 repo-wide tests green (9 Catalog + 31 Inventory + 19 Identity + 99 Orders + 3 CrossBC), zero regressions.

## What worked

- **An independent design review before any code, not after.** Spawning a fresh-context Fable 5 review of the workshop amendment and the full remaining plan caught a genuine state-machine bug (the re-request path claimed to reschedule `EmailChangeTimeout`, which is impossible — Wolverine has no scheduled-message cancellation, so the original timeout would still fire early against the "reset" window) and a missing existence guard, *before* the OpenSpec proposal was drafted from the flawed design. Fixing a workshop paragraph is cheap; fixing a shipped saga's timeout semantics after the fact would not have been.
- **Actually running the host, not just `dotnet build`.** Two real defects surfaced only by executing the Alba integration tests against a live Postgres (Testcontainers): (1) `[WolverinePost]` on two different message-type methods sharing one saga class made Wolverine's HTTP chain builder throw `UnResolvableVariableException`, conflating the two chains' dependencies; (2) the base `Wolverine.Saga` class's inherited `Version` property (undiscovered until reflected on) needed an explicit lowercase-column EF mapping, the same casing class of bug `Customer` already guards against but for a property not visible in `EmailChange`'s own source. Neither was findable from ctx7 docs or Saga #1's precedent (which has no HTTP-driven saga to compare against) — both needed the runtime to actually disagree.
- **`ctx7` mid-implementation, not just at design time.** When the first fix attempt needed to know whether `IMessageBus.InvokeAsync` runs synchronously in-process, a targeted ctx7 query confirmed it before committing to the redesign, rather than guessing from the "reservation" HTTP-saga example alone.
- **The pure-unit / integration split, with an honest asymmetry named rather than papered over.** `StartOrHandle` and `Handle(EmailChangeTimeout)` are pure and unit-tested with no host; `Handle(ConfirmEmailChange, IdentityDbContext)` genuinely needs a database, so unlike `Replenishment`'s fully-pure saga surface, confirm's behavior is integration-only. Recording the asymmetry in the test file's own comment (rather than forcing an artificial pure test) kept the test suite honest about what it actually covers.
- **Live-verifying against the real stack, not just the test suite.** Booting the full Aspire stack (with a 25s demo knob mirroring `Inventory:ReplenishTimeout`) and driving both the confirm-and-apply path and the timeout-and-drop path via curl against the live Identity service caught nothing new (the tests had already found the real bugs), but it did confirm the `ProblemDetails` shapes (`404 CustomerNotFound`, `409 EmailAlreadyRegistered`) render correctly end-to-end, not just inside Alba's in-memory harness.

## What was harder / notable

- **`[WolverinePost]` does not partition cleanly by message type within one class.** The original design (grounded in two *real*, confirmed ctx7 doc patterns — a handler method doubling as an HTTP endpoint, and starting a saga from HTTP) reasoned that putting `[WolverinePost]` directly on `EmailChange.StartOrHandle` and `EmailChange.Handle(ConfirmEmailChange)` would work, since each pattern works in isolation. It doesn't compose: the moment a second `[WolverinePost]`-annotated method existed on the *same* saga class, Wolverine's HTTP chain builder pulled in a dependency belonging to the *other* chain and threw `JasperFx.CodeGeneration.UnResolvableVariableException` at first request. The fix moved the HTTP surface to two separate static endpoint classes dispatching into the (now HTTP-attribute-free) saga via `IMessageBus.InvokeAsync` — confirmed synchronous and in-process, so no reply-timing gap opens up.
- **`Wolverine.Saga`'s inherited `Version` property is a silent EF-Core mapping trap.** Reflecting on the assembly (`Wolverine.dll` 6.16.0) confirmed the base class unconditionally declares `public int Version { get; set; }`, used for `IRevisioned`-style optimistic concurrency even when the derived saga doesn't opt into `IRevisioned`. EF Core's default convention maps it regardless, and Wolverine's own generated saga-loading query references it — so a saga entity that maps only its own declared properties (as `Customer`'s pattern would suggest) is missing a column its *base class* needs. Any future EF-Core-backed saga in this repo should map `Version` explicitly from the start.
- **The `NotFound`-to-HTTP-status question resolved more simply than design.md's fallback anticipated.** Design.md's plan B (an explicit existence pre-check for a controllable 404) turned out to be unnecessary: a void `Post` dispatching via `InvokeAsync` returns `204` regardless of whether the saga was found, making a stale confirm and a successful confirm indistinguishable at the HTTP layer — arguably the *more* faithful "silent no-op" than a manufactured 404 would have been. The fallback was designed but never built, once the simpler default proved sufficient.
- **CritterWatch's saga-lifecycle surface (Workshop 002 § 8, item 10) is unverified in this session.** No browser automation was available in this environment, so the visual confirmation that `EmailChange` appears on the CritterWatch console — the way `Replenishment` does — was not performed. The HTTP-level behavior is thoroughly verified; the CritterWatch dashboard view is not. Flagged, not silently skipped.

## Methodology refinements

- **"Verify empirically" earned its keep as more than a hedge phrase.** design.md flagged the `NotFound`→HTTP-status question as something to verify rather than assume; the *actual* blocking defect (the codegen conflict) wasn't even the flagged risk — it was a different composition failure the design hadn't anticipated at all. Both were only findable by running the host. A future saga design pass should budget real implementation time for exactly this, not treat "verify empirically" as a formality satisfied by writing the sentence.
- **An independent, fresh-context review before drafting a spec catches state-machine bugs a self-review misses.** The re-request/timeout defect was a genuine logic error, not a typo — and it existed because the author (in the same continuous reasoning context as the rest of the design) didn't independently re-derive whether "reschedule the deadline" was actually implementable given "no scheduled-message cancellation," a fact stated *elsewhere in the same document*. A reviewer with no stake in the prior reasoning caught the contradiction immediately.
- **Record a design's actual, as-shipped shape in the design doc — including where it diverged from the plan — rather than leaving the doc describing a disproven approach.** design.md was rewritten in place (not left as a historical curiosity) once the `[WolverinePost]`-on-saga plan failed, so a future reader trusts it as ground truth rather than having to cross-reference the code to find out which parts of the doc are still accurate.
