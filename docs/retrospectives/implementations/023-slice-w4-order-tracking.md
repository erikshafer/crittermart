## What worked

- **Extracting the poll/stepper "brain" into a pure module killed the flaky-test problem before it appeared.** `refetchInterval`-driven React-Query polling is notoriously hard to test with fake timers (testing-library's `waitFor` and vitest fake timers fight). By making `pollIntervalFor(data)` and `deriveJourney(status)` pure functions, "stops on terminal" became a deterministic one-line assertion (`pollIntervalFor(viewAt("confirmed")) === false`) and the stepper got exhaustive per-status coverage — no timers, no races. The React layer is then a thin shell whose test only needs render + a click. This is the same extract-the-hard-part-as-a-pure-sibling discipline the cart merges use, now applied to convergence instead of optimistic reconciliation.
- **The handoff's two "honest gaps" were real, and confirming them in the source (not just the schema) was worth the read.** `OrderStatusView` carries no placed-at timestamp **and** no cancellation reason — `OrderCancelled` folds to a bare `Status = "cancelled"` (`OrderStatusView.cs:44`), so `stock_unavailable`/`payment_declined`/`payment_timeout` are not on the wire. Reading the backend confirmed W4 cannot show the reason or the failure step without a projection change — so the cancelled branch renders an honest generic terminal treatment, and both gaps are logged for a future "enrich `OrderStatusView`" slice rather than faked.
- **Wholesale reuse held exactly.** `OrderStatusViewSchema` already modeled `lines`/`status`; `orderQueryOptions` already keyed by `orderId`. W4 added a `refetchInterval` and `refetchIntervalInBackground:false` at the `useQuery` call site and nothing else in those files — the reused `queryOptions` carried its `OrderStatusView` data type through the spread, so the `refetchInterval: (q) => pollIntervalFor(q.state.data)` callback type-checked with no annotations.
- **The stepper honors the real enum, not the wireframe's simplification.** The § 5.1 wireframe shows `awaiting → stock reserved → confirmed` (3 steps), but the enum has `payment_authorized` between the last two. The screen must render all five statuses correctly anyway, so a 4-step stepper (cancelled as a branch) is both more honest and no more code — and the per-status test proves each lands on the right waypoint.
- **The live wire walked `awaiting_confirmation → confirmed` on the first placement.** With stock seeded, the cross-BC chain settled inside ~1.2 s — the convergence target the poll chases. Verifying it headlessly (the same endpoint W3 verified in #62, now consumed by-id with a poll) re-confirms the contract W4 binds is exactly what the backend serves.

## What was harder / notable

- **A long-lived process can't be launched with `Start-Job` from the PowerShell tool.** The first AppHost-boot attempt used `Start-Job`; the job died the instant its creating shell ended (the PowerShell tool starts a fresh session per call, and shell state — including background jobs — does not persist), leaving no log and a couple of orphaned `dotnet` build processes. The fix was the tool's own `run_in_background: true`, which detaches the process so it survives across turns. Lesson for the next session-runner doing a live verify: **boot the AppHost with the tool-level background flag, redirect to a log file, and poll the file** — not `Start-Job`.
- **The fast happy path makes the "live" convergence window brief.** With stock seeded, `awaiting_confirmation → confirmed` is ~1 s, so opening the W4 screen and *watching* the stepper advance is a blink. That's inherent to the seeded happy path, not a W4 bug; the poll-stops-at-terminal behavior is what keeps the settled screen quiet. For a demo that lingers on the convergence, leaving one SKU unstocked (so it settles `cancelled`) or watching the dashboard span chain reads slower than the status flip.
- **Introducing a `<Link>` into the previously-pure W3 page changed its test harness.** `OrderConfirmationPage` was router-free (the disabled control was a `<button>`); a TanStack `<Link>` needs router context. Its test now mounts the page in a throwaway memory-history router that also registers `/orders/$orderId` so the link resolves and a click navigates — exactly the helper shape `CartPage.test.tsx` already used and whose comment pre-named "the pattern W1/**W4** screen tests reuse." A small, anticipated cost; `OrderStatusPage` itself holds no `<Link>`, so its own test stays router-free.

- **"Extract the timing-sensitive rule as a pure function" is now the convention for any poll/interval screen**, the way "extract the optimistic merge as a pure sibling" is for mutations. Convention 3 now records both halves: the cart's pure merges and W4's pure `pollIntervalFor`/`deriveJourney`. A future polling screen should copy this rather than reach for fake-timer React-Query tests.
- **A screen-only slice can legitimately *surface* backend gaps without fixing them — log them, don't fake them.** W4 surfaced two (`OrderStatusView` has no placed-at, no cancel reason). The fence (no backend change in a screen slice) held; both are named in the narrative v1.6 + the prompt's out-of-scope as a future "enrich `OrderStatusView`" slice. This keeps the screen honest (a generic "Cancelled", no invented reason) without scope-creeping the slice into a projection change.
