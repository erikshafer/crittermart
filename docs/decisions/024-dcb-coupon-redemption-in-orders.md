# ADR 024: DCB-Protected Coupon Redemption in Orders; Promotions Contributes Coupon Definitions

**Status**: Accepted

## Context

Round one's modeled implementation set is complete, and three post-round-one increments have shipped: the Inventory `Replenishment` saga, the Identity `EmailChange` saga ([ADR 022](022-convention-sagas-additive-to-pmvh.md)), and real authentication for Identity ([ADR 023](023-real-authentication-for-identity.md)). Both prior top long-road candidates — a saga pair and real auth — are now built, so the strongest remaining teaching payoff is a **genuinely new event-sourcing pattern** for the talk. [`docs/vision.md`](../vision.md) § Long road and [the context map](../context-map/README.md) § Long road both name **"Promotions with DCB-protected coupon redemption"** as an unchosen candidate. This ADR chooses it, and settles the three architecturally load-bearing questions the choice raises. It is CritterMart's first use of Marten's **Dynamic Consistency Boundary (DCB)**.

The direction (a promotions increment showcasing DCB) and the modeled invariant (a **global redemption cap** — a coupon may be redeemed at most *N* times across all orders) were settled by owner decision. Two facts had to be verified before the decision could be recorded, because either could have forced a different first artifact:

1. **Does DCB require Polecat?** The installed skill names DCB a Polecat (SQL Server 2025) feature, and [ADR 009](009-polecat-deferred-for-round-one.md) defers Polecat. Verified against current Marten docs **and the pinned assembly**: DCB is first-class in **Marten 9.11.0** — the exact version resolved transitively through `WolverineFx.Marten` 6.16.0 (`FetchForWritingByTags`, `EventTagQuery`, `DcbConcurrencyException` are all present in `Marten.dll`). It is **additive** to stream-based event sourcing ("DCB events always belong to a stream") and its schema change is **opt-in** — a `tags TEXT[]` column and a GIN index on `mt_events`. So DCB runs on CritterMart's existing Marten/Postgres stack: no new engine, and no Wolverine/Marten version bump (which the CritterWatch 1.0.0-beta.1 coupling would otherwise block — [ADR 017](017-critterwatch-integrated.md)).
2. **DCB is store-scoped.** `FetchForWritingByTags` queries one store's `mt_events`, tenant-scoped by default. Under schema-per-service ([ADR 002](002-shared-postgres-schema-per-service.md)) each service owns its own store, so a cross-stream consistency boundary can be enforced atomically only **inside a single service's event store** — never across the RabbitMQ boundary between two services.

## Decision

Introduce **coupon redemption at Orders checkout**, enforcing a **global per-coupon redemption cap**, with the cap enforced by **Marten DCB inside the Orders event store**.

- A redemption is recorded as a **tagged event in the Orders store**, tagged by a strong-typed `CouponId`, on the order/checkout stream it naturally belongs to. The cap is enforced at checkout via `FetchForWritingByTags` over `EventTagQuery().Or<CouponId>(id)`: Marten aggregates the redemption count across **all** order streams carrying that tag and throws `DcbConcurrencyException` at `SaveChangesAsync` if a concurrent redemption would push the count past *N*. The consistency boundary aligns with **no single aggregate** — the textbook DCB shape.
- **Promotions is not a separate service this increment.** It contributes **coupon definitions** only — the code, discount, and cap *N* — modeled as a Published-Language contract Orders consumes at checkout, realized for this round as a seed/local read model in the Orders store. Redemption and cap enforcement live entirely in Orders. A standalone Promotions BC (its own schema, Aspire wiring, a Customer-Supplier redemption gate mirroring Inventory) is **explicitly deferred** to a later increment.
- The Orders store **opts into** the DCB schema (`tags TEXT[]` + GIN index on its `mt_events`); no other service's store is touched.

This is design-only. The per-slice loop (workshop → OpenSpec proposal → narrative → implement) builds it; the next artifact is the **Promotions event-modeling pass**. Its home is itself a next-session modeling call: the coupon-definition concept is new territory (arguing a standalone **Workshop 003**), but redemption executes on **Orders** streams, which are Workshop 001's territory — so that pass will add slices to the Orders swim lane whether it lands as Workshop 003 or a Workshop 001 amendment.

## Consequences

**Why DCB and not a single-stream `Coupon` aggregate.** A *pure* global cap could be a single `Coupon` stream whose redemption count `FetchForWriting` guards with ordinary optimistic concurrency. DCB earns its place on **contention**: the coupons whose cap actually bites are popular ones — a flash sale — and a single `Coupon` stream turns every concurrent checkout into a serialized write on one hot stream, retrying under load exactly when load is highest. DCB tags let redemptions append to their own order streams and conflict only when they genuinely race the cap near *N*. It is also the more honest "the consistency boundary does not align with an aggregate" demonstration — which is the pedagogical point of the increment, not an incidental property.

**Why in Orders and not a fifth Promotions service.** Because DCB is store-scoped, the cap must be enforced where every redemption event lives in one store — and every checkout already flows through Orders, so Orders is the one store that sees them all. Redemption-in-Orders gives the **purest** DCB demo (real order streams, no synthetic redemption streams, no RabbitMQ hop burying the boundary inside a supplier) and the **lightest** build, honoring the "pipeline scales down" ethos for a single teaching increment. The cost is named, not hidden: Promotions does **not** become its own BC/service this round; the context-map "Promotions BC / Published-Language coupon definitions Orders consumes" framing is realized as **definitions-only**, with the standalone service deferred.

**Honors the non-negotiables, does not bend them.** No new service (topology unchanged — four services, [ADR 001](001-separate-services-topology.md)); no Polecat ([ADR 009](009-polecat-deferred-for-round-one.md) reaffirmed); no Wolverine/Marten version bump (the CritterWatch coupling holds); no new cross-service **runtime** edge — redemption is intra-Orders, and coupon definitions cross as **Published Language** (like `CustomerRegistered`), not synchronous HTTP ([ADR 001](001-separate-services-topology.md)/[ADR 003](003-wolverine-rabbitmq-transport.md)). The only new surface is the opt-in DCB schema in the Orders store — the same additive, low-blast-radius shape ([ADR 002](002-shared-postgres-schema-per-service.md)'s per-schema isolation gives it.

**Rejected alternatives, folded in.** *Single-stream `Coupon` aggregate* — simpler for a pure cap, but a write hotspot for popular coupons and it hides the DCB teaching point. *A saga / process manager for redemption* — eventual, so it cannot enforce a hard cap synchronously at redeem time; it would let redemptions slip past *N* under concurrency (oversell), the opposite of the invariant. *Promotions as a fifth service* — architecturally consistent (a Customer-Supplier redemption gate mirroring Orders↔Inventory, a nice "two checkout supplier-gates, two consistency mechanisms" contrast), but it buries the DCB behind a service hop, makes the redemption streams synthetic, and is a heavy build (project + schema + Aspire + full test surface) for one teaching increment — deferred, not discarded. *Polecat for DCB* — unnecessary (DCB is in Marten 9.11) and would reverse ADR 009 and force a SQL-Server-2025 engine for one pattern.

**Forward questions this ADR names but leaves to the workshop and per-slice loop** (not re-litigated here):

- **Event and tag shape.** The redemption event name; the strong-typed `CouponId` tag; and the precise stream placement of the tagged redemption event (on the existing order stream vs. a dedicated redemption stream) — a Marten-DCB mechanics detail for the slice, which the "real order streams" framing above is the intent for.
- **Where the cap *N* and the coupon definition live.** A Promotions Published-Language contract vs. a round-one config/seed in the Orders store; whether *N* is a coupon-definition field or separate config.
- **The richer DCB variants this global cap is the entry point to** — *one redemption per customer* (a composite `(coupon × customer)` tag, the more dynamic boundary) and a *shared discount budget* (a summed value aggregated across streams, not a bare count). Modeled as natural extensions, not built now.
- **Whether and when Promotions graduates to its own service** — the deferred fifth-service option above, if a later round wants the Customer-Supplier gate.
