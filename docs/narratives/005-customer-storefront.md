# Narrative 005 ‚Äî The Customer Shops the Storefront

This is the same Customer as [Narrative 002](002-customer-browse-catalog.md) (browsing) and [Narrative 004](004-customer-purchase.md) (buying) ‚Äî but told through a different lens. Those two narratives are the **behavior** of the journey: the event streams, the read models, the cross-bounded-context hops, the cancellations. This one is the **screen**: what the Customer actually sees and touches as they move from the storefront's product listing, into a cart, through checkout, to a tracked order. It is the realization of [ADR 016](../decisions/016-frontend-full-pipeline-ui-first-class.md) ‚Äî the UI made first-class ‚Äî written as the journey *through* the wireframes the workshop now carries in [¬ß 5.1](../workshops/001-crittermart-event-model.md) (**W1** Browse, **W2** Cart Review, **W3** Order Confirmation, **W4** Order Status).

The storefront is a **single client-side React SPA** that calls the three Wolverine.Http services **directly** ‚Äî there is no BFF ([ADR 006](../decisions/006-wolverine-http-per-service-no-bff.md)); the SPA *is* the cross-service orchestrator, and that real cross-network boundary is the OpenTelemetry-trace beat the talk depends on ([ADR 015](../decisions/015-vite-react-frontend-stack.md), [ADR 018](../decisions/018-frontend-three-services-cors-posture.md)). The stack is locked and is not re-decided here; this narrative threads its load-bearing stances as established context: every wire response is **Zod-parsed at the boundary** before the app trusts it (three independently-deployed services means three contract surfaces; ADR 015 R3); cart mutations feel instant through **optimistic UI with rollback** (`onMutate` applies the guess, `onError` rolls back, `onSettled` refetches the authoritative read model; ADR 015 R4); and there is **no real-time push** round one (status converges by refetch, not a socket; ADR 015 R5). The Customer's identity is the round-one stub ‚Äî a hardcoded customer id behind the `useCurrentCustomer` seam ([ADR 009](../decisions/009-polecat-deferred-for-round-one.md)) ‚Äî so there is no login screen.

## Journey scope

The screen journey threads the wireframe-bearing slices into one browse ‚Üí cart ‚Üí checkout ‚Üí track arc:

- **Slice 1.2 ‚Äî Browse and view products.** Moment 1: landing on the storefront (**W1**).
- **Slice 3.1 ‚Äî Add item to cart.** Moment 2: adding from the listing, optimistically (**W1** ‚Üí cart badge).
- **Slice 3.5 ‚Äî View my open cart** *(net-new view slice, modeled in this session's workshop amendment).* Moment 3: returning to the cart on a cold load (**W2**). This is the keystone ‚Äî the read that makes the cart-review screen renderable.
- **Slices 3.2 / 3.3 ‚Äî Remove item / change quantity.** Also Moment 3: editing the cart in place (**W2**), optimistically.
- **Slice 4.1 ‚Äî Place order from cart.** Moment 4: checkout (**W2** ‚Üí **W3**).
- **The order lifecycle, read as status** (slices 4.2‚Äì4.7, surfaced through `OrderStatusView`). Moment 5: tracking the order (**W4**).

**What is *not* in this narrative.** The system internals ‚Äî the Klefter local commits, the broker hops, the Bruun timers, the cancellations ‚Äî are Narrative 004's material, not repeated here; this narrative points at them as the *status* the Customer sees settle. And per ADR 016's guardrail, **pure presentation state** ‚Äî a modal opening, pagination, a theme toggle, the cart-badge tween ‚Äî is deliberately *not* modeled as events; it lives in frontend code and, where it matters to the journey, in this prose. The line this narrative walks is: a screen *reads* a domain fact (a view slice) or *produces* one (a command slice); anything that does neither is not on a stream.

## Moment 1 ‚Äî Landing on the storefront

**Context.** The Customer opens CritterMart for the first time. The SPA is served locally through the Aspire AppHost (a managed resource in the dashboard alongside the three services, RabbitMQ, and PostgreSQL; ADR 015). The Seller has been at work ‚Äî the catalog holds the *Cosmic Critter Plush* (`crit-001`, `$24.99`) and the *Nebula Newt* (`crit-002`, `$18.00`), each published through slice 1.1.

**Interaction.** The Customer arrives at the product listing ‚Äî wireframe **W1**. They do nothing yet but look.

**System response.** The SPA reads the catalog **directly from the Catalog service** ‚Äî `GET /products` over HTTP, no aggregating gateway in front of it ‚Äî and **Zod-parses the response** before rendering anything from it (ADR 015 R3: the first of three contract surfaces the SPA validates). The listing shows both products, each with name, current price, and description ‚Äî the full `ProductCatalogView`. Product *detail* is rendered from that same list payload; there is no separate `GET /products/{sku}` (Gap #2 from the endpoint audit ‚Äî low, deferred), so a click for "more" expands what the SPA already holds rather than fetching a single product. The stubbed customer id rides along with the request but gates nothing: the catalog is public, and the Customer sees it without signing in. Whether anything is actually *in stock* is not shown here ‚Äî stock lives in Inventory, which the storefront does not consult to render a listing (Narrative 002); a shortfall only ever surfaces later, as a cancelled order.

## Moment 2 ‚Äî Adding to the cart, instantly

**Context.** The Customer has settled on the *Cosmic Critter Plush*. They have no cart yet ‚Äî this is their first step toward a purchase, taken from the listing (**W1**).

**Interaction.** They tap **`[ Add to cart ]`** on the plush. The cart badge in the header (`Cart (0)`) ticks to `Cart (1)` the instant they tap ‚Äî before the server has answered.

**System response.** When it rendered the listing the SPA already held the product's fields, so it **snapshots** name and price into the command ‚Äî `AddToCart { customerId, sku: "crit-001", quantity: 1, productSnapshot: { name, price: 24.99 } }` ‚Äî and sends it to Orders (this is the presentation-layer composition the context map names: product data reaches the Cart only through the frontend's snapshot, never a Catalog‚ÜîOrders integration). The instant badge bump is **optimistic UI** (ADR 015 R4): `onMutate` applies the guess locally; if the command fails, `onError` rolls the badge back; either way `onSettled` refetches the authoritative `CartView`, so the displayed cart converges on what the server actually recorded ‚Äî the read model, never the optimistic guess, is the truth. Behind that snappy badge, the behavior is Narrative 004's Moment 1: Orders creates the Cart stream (`CartCreated` + `CartItemAdded`). If the Customer adds the *Nebula Newt* too, the badge reaches `Cart (2)` and the same optimistic-then-reconcile cycle runs again.

## Moment 3 ‚Äî Returning to the cart, and editing it

**Context.** The Customer comes back to look at what they've gathered ‚Äî perhaps in the same session, perhaps on a **cold load**: a fresh browser session the next day, holding nothing but the stubbed customer id. They open the cart ‚Äî wireframe **W2**.

**Interaction.** They navigate to the cart-review screen.

**System response ‚Äî and the gap that makes it possible.** Rendering this screen needs the cart's contents, and here the storefront hits the mismatch the [pre-frontend endpoint audit](../research/pre-frontend-endpoint-audit.md) flagged as **Gap #1, the one blocking gap**. Every cart *command* is **customer-keyed** ‚Äî the Customer never sends a cart id; "which cart is theirs" is the server's business (Narrative 004, Moment 1A). But the only round-one cart *read* is `GET /carts/{cartId}`, and on a cold load the SPA has **no cart id** ‚Äî it learned one only from an earlier `AddToCart` response that this fresh session never saw. Without a customer-keyed read, the cart-review screen simply cannot render.

**Slice 3.5 ‚Äî "View my open cart"** closes that. The SPA asks for *its* cart by identity ‚Äî `GET /carts/mine`, the customer carried behind the `useCurrentCustomer` seam (ADR 009) ‚Äî and Orders resolves the Customer's single open cart through the partial-unique open-cart index that already exists (`Orders/Program.cs:74`), returning that one `CartView` (Zod-parsed, like every wire response). It adds **no new event** and exposes **no new projection** ‚Äî it is the read counterpart to the customer-keyed write side every cart command already uses, the missing mirror. If there is **no open cart** ‚Äî the Customer never started one, or their last cart was checked out or abandoned ‚Äî the read resolves to "no open cart" (not an error), and the screen renders empty, ready for the next add.

With the cart on screen, the two edits are in reach. **`[-]` / `[+]`** issue `ChangeCartItemQuantity` (slice 3.3); **`[x]`** issues `RemoveCartItem` (slice 3.2) ‚Äî each optimistic in exactly the way Moment 2's add was: the line updates immediately, then reconciles against the refetched `CartView`. Changing a quantity never re-prices ‚Äî the snapshot price from Moment 2 stays authoritative (Narrative 004, Moment 1A). A SKU appears as exactly one line (adding it again merges, quantities summed), which is what makes "set `crit-001` to 2" and "remove `crit-001`" unambiguous on screen. And if the Customer empties every line, the cart does not vanish ‚Äî it stays their open cart, just empty; the only thing an empty cart cannot do is check out (`PlaceOrder` is refused with `CartEmpty`).

## Moment 4 ‚Äî Placing the order

**Context.** The Customer is on the cart-review screen (**W2**) with two lines ‚Äî the plush (`crit-001`, quantity 2 at `$24.99`) and the newt (`crit-002`, quantity 3 at `$18.00`) ‚Äî and decides to buy.

**Interaction.** They tap **`[ Place Order ]`**.

**System response.** The SPA sends `PlaceOrder { customerId }` ‚Äî and *nothing else*: the order's contents are not re-sent from the browser. What the Customer is buying is whatever their open cart already holds, server-side (the same open-cart resolution slice 3.5 reads, now driving a write). The `POST /orders` response carries **only the new `orderId`** ‚Äî *not* a status (the placement has only just kicked off the cross-BC outcome; the server has nothing settled to report at the moment it answers). So the SPA navigates to the confirmation route keyed by that `orderId`, and **W3 reads the order back** ‚Äî `GET /orders/{orderId}`, the `OrderStatusView`, Zod-parsed at the boundary like every wire response (the very read Moment 5's W4 uses, entered here post-placement rather than by-id) ‚Äî to show the status and total. This is the one place a screen *reads its way to* the thing it just *wrote*; it is also where the optimism that carried Moments 2 and 3 deliberately **stops**. A cart edit could be faked locally because its outcome was knowable; placing an order cannot, because it sets off the cross-bounded-context process ‚Äî reserve stock in Inventory over the broker, then authorize (stubbed) payment ‚Äî whose outcome the SPA does not yet know (Narrative 004, Moments 3‚Äì4). So **W3** shows an honest `awaiting confirmation` status and the total (`$103.98`) read straight from the `OrderStatusView` (the view carries the server-computed total, so the screen renders it rather than re-summing the lines) ‚Äî not a faked "confirmed." The cart badge resets to `Cart (0)`: the checked-out cart is no longer the Customer's open cart, and their next add starts a fresh one.

This `POST /orders` is also the **front door of the trace beat**: it is the real HTTP hop from the SPA into Orders that the OpenTelemetry trace follows before it fans out across RabbitMQ to Inventory and back (ADR 015 ¬ß Consequences; the cross-network span chain is a hard success criterion, not a nice-to-have). The Customer sees a status; the talk's audience sees the span light up in the Aspire dashboard.

## Moment 5 ‚Äî Tracking the order

**Context.** The order is placed and sits at `awaiting_confirmation` (**W3**). From here the Customer takes no further action ‚Äî they watch the status settle on the order-status screen, wireframe **W4**.

**Interaction.** They tap **`[ Track this order ]`** (or return to it later). The screen reads the order and shows where it is.

**System response.** The SPA reads `OrderStatusView` ‚Äî `GET /orders/{orderId}`, which exists today (the audit's ‚úÖ row; single-order tracking needs no new slice) ‚Äî Zod-parsed like every response. The status walks the path the backend drives behind the scenes: `awaiting_confirmation ‚Üí stock_reserved ‚Üí confirmed`, or it settles on `cancelled` for one of three reasons ‚Äî `stock_unavailable`, `payment_declined`, or `payment_timeout` (the three failure routes of Narrative 004, Moments 3 / 5 / 6). Because there is **no live push** round one (no SignalR ‚Äî ADR 015, explicitly unlike the CritterBids sibling), the screen does not update itself the instant the server changes; it converges by **TanStack Query refetch** ‚Äî a poll or a manual refresh ‚Äî so the optimistic-UI rule from the cart ("re-query, don't render a guess") applies here too, just to a read instead of a mutation. "Confirmed" is as far as the order travels: CritterMart models no picking, packing, or shipping, so the terminal of a successful purchase is a *confirmed order*, not a delivered parcel (Narrative 004). A Customer who wants to see *all* their orders at once finds no such screen yet ‚Äî the "My Orders" list (`GET /orders?customerId=`) is **Gap #3**, named and deferred; round two tracks one order at a time.

## What the Customer does *not* yet see (on screen)

These absences are deliberate, and naming them keeps the screen journey honest:

- **No login screen.** Identity is the round-one stub behind the `useCurrentCustomer` seam (ADR 009); the Customer is "signed in" as a hardcoded id from the first render. A real Identity service is a long-road item.
- **No live stock on the listing (W1).** The storefront does not consult Inventory to render the catalog; a shortfall surfaces only later, as a cancelled order (Narrative 002, Narrative 004).
- **No deep-linkable product page (W1).** Detail renders from the list payload ‚Äî there is no `GET /products/{sku}` (Gap #2) ‚Äî and the SPA has no SEO/prerender story (ADR 015 R5, explicitly *unlike* MmoReconnect): an authenticated demo storefront has neither crawler-visibility nor launch-wave-resilience as a driver.
- **No real-time status (W4).** No socket; status converges by refetch. The "instant" feel lives only where it can honestly be faked ‚Äî cart mutations (W2) ‚Äî never on the order's cross-BC outcome.
- **No "My Orders" history (W4).** Gap #3, deferred; single-order tracking is the round-two surface.
- **No modeled presentation state anywhere.** Modals, pagination, the cart-badge animation, route transitions ‚Äî none of it reaches an event stream (ADR 016 guardrail). It is real, and it lives in the frontend code, but it is not domain behavior.

## Forthcoming

This narrative is **mostly built** ‚Äî the storefront is now a loop through checkout, and **the Customer can place an order**. The backend reads are all shipped (slice **3.5 `GET /carts/mine`** closed the blocking Gap #1); the **frontend-bootstrap PR** stood up the `client/` Vite SPA + Aspire `AddViteApp` wiring + CORS-origin injection; the **W2 cart-review screen** renders the cold-load cart (`client/src/cart/`, the first `CartViewSchema` + `cartQueryOptions`) **and lets the Customer edit it in place**; the **W1 browse screen + add-to-cart** closes the entry: `client/src/catalog/` binds `GET /products` through the second schema (`ProductCatalogViewSchema`, a `z.array`) into the product grid, and `[ Add to cart ]` issues the storefront's **first optimistic mutation** (`client/src/cart/cartMutations.ts` ‚Äî `useAddToCart`, snapshotting name + price into `AddToCart`; `onMutate` bumps the cart badge, `onError` rolls back, `onSettled` reconciles against the refetched `CartView`). The **W2 edits** (Moment 3) then realized Convention 3's 2nd + 3rd consumers: the `[-] N [+]` stepper issues `ChangeCartItemQuantity` (slice 3.3) and the `[x]` issues `RemoveCartItem` (slice 3.2), each the same optimistic-then-reconcile cycle (`useChangeCartItemQuantity` + `useRemoveCartItem`, each a thin hook over a pure merge sibling), and the command layer gained the cart's first DELETE and its first 204-response (no-body) commands. The **W3 checkout** (Moment 4) then added `[ Place Order ]` on the cart and the **Order Confirmation** screen: the new `client/src/orders/` folder binds the **third** read surface through `OrderStatusViewSchema` (the third per-read-model schema; `status` a closed `z.enum`) and `orderQueryOptions` keyed by `orderId`, and `usePlaceOrder` (in `orders/`, mirroring backend `Orders/Features/PlaceOrder`) is the storefront's **first non-optimistic mutation** ‚Äî Convention 3's deliberate exception: `POST /orders { customerId }` (body-keyed ‚Äî a third identity-transport shape), then on success `invalidate(cartKeys.mine)` (the badge resets to `Cart (0)`) and `navigate` to W3, which reads the order back for its honest `awaiting confirmation` status + total. This `POST /orders` is the **OpenTelemetry trace front-door** ‚Äî the first SPA-initiated cross-service hop, fanning out over RabbitMQ to Inventory. The browse ‚Üí add ‚Üí edit ‚Üí **checkout ‚Üí confirmation** loop is demonstrable. What remains is screen-pending:

- **Moment 5 (W4 Track)** ‚Äî the `OrderStatusView` single-order status read, polling toward the terminal state. It **reuses the `client/src/orders/` folder this slice introduced** (`OrderStatusViewSchema` + `orderQueryOptions`), differing only by entry (by-id rather than post-placement) and refresh (a `refetchInterval` poll, ADR 015 R5); the disabled `[ Track this order ]` on W3 becomes its live entry point.

As each screen is built, this narrative grows ‚Äî each bumping its version and appending to the history below. Nothing in this journey is half-finished silently: every screen named here maps to a modeled slice and a wireframe; the build order is the audit's (Gap #1 first), then W2's read, then W1's browse + the first mutation, then W2's in-place edits, now W3's checkout ‚Äî leaving only W4 tracking, the named deferred follow-on.
