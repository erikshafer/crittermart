## Framing

Slices 3.1–3.3 let the Customer fill and edit a cart; slices 4.1–4.7 took a checked-out cart through the full order lifecycle, ending with 4.7's payment-timeout — the project's first time-as-trigger automation. Slice 3.4 mirrors that automation on the cart side: a cart left inactive past a configured window is **abandoned** — the Cart stream's second terminal event (after `CartCheckedOut`). The slice also lands the round-one **async projection teaser** (ADR 008): `CartAbandonmentReport`, a daily analytics rollup that exists to be *rebuilt on demand* in the talk, never depended on live. With 3.4 shipped, every Cart and Order stream reaches a terminal state and round one's modeled implementation set is complete.

## Goal

A cart created by `AddToCart` schedules a `CartActivityTimeout` self-message for the configured inactivity window (`Orders:CartActivityTimeout`, default 2 hours). When the timeout fires: a closed cart (checked out or already abandoned) is a silent no-op; a cart with activity newer than the window reschedules the timeout from that activity (fire-and-check); a cart with no activity within the window appends `CartAbandoned { reason: "inactivity_timeout", lines, totalValue }` and closes (`IsOpen` → false). The `CartsAwaitingActivity` inline projection (one row per open cart, deadline visible, conditional-deleted on either terminal event) is readable at `GET /carts/awaiting-activity`. The `CartAbandonmentReport` async multi-stream projection (daily rollup: count, total value, SKU breakdown) is registered with `ProjectionLifecycle.Async`, **no daemon**, and is materialized by an on-demand rebuild — proven by an integration test. `openspec validate --strict` passes; full solution green.

## Locked decisions (forks resolved with the user at session start, 2026-06-02)

1. **Scheduling policy: single rolling timeout (fire-and-check).** ONE scheduled `CartActivityTimeout` per open cart, scheduled only when `AddToCart` creates the cart. Cart edits append events but schedule nothing — their timestamps are the activity record. The firing handler checks `CartView.LastActivityAt`: activity within the window → reschedule for `lastActivity + window`; otherwise → abandon. Grounded by ctx7 verification: Wolverine cannot cancel a scheduled message, so the originally named cancel-and-reschedule policy is not implementable; fire-and-check matches the workshop's GWT failure path exactly and keeps exactly one message in flight per cart.
2. **Report shape: daily rollup.** `CartAbandonmentReport` is a `MultiStreamProjection` keyed by calendar day (UTC) of abandonment: `abandonedCartCount`, `totalValueAbandoned`, `abandonedSkus` (SKU → count). The codebase's first multi-stream projection — paired with its async lifecycle, the two new teaching artifacts land in one class.
3. **Daemon scope: rebuild-only, no daemon.** `ProjectionLifecycle.Async` registration, **no** `AddAsyncDaemon(...)` in Program.cs — the strictest reading of ADR 008 ("No Daemon for Round One" is the title). The report stays empty until an on-demand rebuild materializes it; that rebuild is the talk's teaching beat and this slice's integration-test proof.
4. **Spec deltas: minimal MODIFIED.** Only *Add an item to the cart* is MODIFIED (it gains real behavior — the schedule-on-creation cascade). Remove/change-quantity/checkout keep their existing wording; their "open cart" language already absorbs abandoned carts via the `NoOpenCart` rejection.
5. **Fat `CartAbandoned` event** *(session-runner decision from analysis, recorded in design.md)*: the event carries the abandoned cart's `lines` + `totalValue`, snapshotted by the handler from the folded `CartView` at append time. Event-sourcing idiom: record the decision with the data it was made on; this is what makes the multi-stream report a pure fold.
6. **`CartView.LastActivityAt` via `IEvent<T>` folds** *(session-runner decision from 4.7's Decision-3 principle)*: the fire-and-check question ("did activity intervene?") is answered by the aggregate folded from the stream — not by the todo-list view, not by a raw-events query. The `CartViewProjection` activity Apply methods take `IEvent<T>` wrappers for timestamps — the codebase's third `IEvent<T>` metadata fold, triggering the skills decision (locked decision 8).
7. **Default window 2 hours**, bound from `Orders:CartActivityTimeout` (mirroring `Orders:PaymentTimeout`), registered as a `CartActivityDeadline` singleton feeding both the AddToCart schedule and the projection's visible deadline. Workshop § 7's rebuild story (">2h as abandoned") implies the default; the Aspire demo can shorten it without recompiling.
8. **Skills decision: DEBT row, not a skill file.** The Marten-pattern third use (instance-registered projection + `IEvent<T>` fold + conditional delete, all now used 3×) is recorded as a `docs/skills/DEBT.md` row naming the candidate skill and its three precedent sites. Authoring the skill itself is a future `tidy: skills` session — this slice's budget goes to the implementation.
