## Framing

W1 (#60) shipped the storefront entry + the first optimistic mutation; W2 (#58, #61) shipped the cart-review screen and made it editable — the browse → add → edit → W2 loop is demonstrable. The cart's `[ Place Order ]` was deliberately absent in every prior slice (the `CartPage` comment + the narrative said so). This slice adds it and builds the screen it leads to: tap `[ Place Order ]` → `POST /orders` → navigate to **W3** → the confirmation screen reads the placed order and shows `awaiting confirmation` + total.

This is the slice where the optimistic-UI template the three cart commands share **does not apply**. A cart edit could be faked locally because its outcome was knowable; placing an order cannot, because it kicks off a cross-bounded-context process (reserve stock in Inventory over RabbitMQ, then authorize stubbed payment) whose outcome the SPA does not yet know. So `usePlaceOrder` is a **non-optimistic** mutation — `mutationFn` + `onSuccess` (invalidate the cart so the badge resets, then navigate), **no `onMutate` guess** — and W3 shows an honest pending status read from the server, never a faked "confirmed."

It is also the **OpenTelemetry trace front-door**: `POST /orders` is the first real HTTP hop the SPA makes into Orders that fans out across the broker to Inventory and schedules the payment-timeout. The talk's audience watches that span chain light up in the Aspire dashboard; this slice is what initiates it from the browser.

## Goal

On the cart-review screen (`/cart`, non-empty), a `[ Place Order ]` button issues `PlaceOrder { customerId }` (body-keyed; the `X-Customer-Id` header rides along but the server reads the body). On success the cart key is invalidated (the badge refetches → `GET /carts/mine` 404 → `Cart (0)`) and the SPA navigates to `/orders/$orderId/confirmation`. The W3 screen reads `orderQueryOptions(orderId)` → `GET /orders/{orderId}` (`OrderStatusView`, Zod-parsed at the boundary, the 3rd contract surface) and renders the wireframe: "Order placed.", the order id, the status (`awaiting confirmation`, as text not color-only), the total (read directly off the view — the server computed it), and a `[ Track this order ]` control (W4, deferred — rendered disabled, not a broken link). Honest loading/error states. A `409` (`NoOpenCart`/`CartEmpty`) surfaces as an honest place-failed message on the cart, with no navigate. `npm run build` + `npm run test` green; the place + the follow-on read + the cross-BC span chain confirmed live in the browser + Aspire dashboard.

## Spec delta

**No OpenSpec / workshop change** — this slice *binds* two already-shipped contracts (`POST /orders` + `GET /orders/{orderId}`); it adds no new behavioral SHALL, no new event, no new read model (ADR 016 + the #58/#60/#61 precedent: a screen that binds existing contracts is not a new modeled contract). The canonical-spec movement is **Narrative 005 → v1.5**: Moment 4 goes from *screen-pending* → *built*, and the narrative's `{ orderId, status }` wording is **corrected** to the shipped `{ orderId }`-only response + the follow-on `GET /orders/{orderId}` read (the load-bearing reconciliation). `## Forthcoming` and `## Document History` record it. The frontend skill advances v5 → **v6** if earned: Convention 2 gains the 3rd schema/3rd read surface, Convention 3 records its named exception (4.1 = optimism stops) realized in code, Convention 4 gains the body-keyed `{ customerId }` transport (the 3rd identity-transport shape).

## Locked decisions (forks resolved with the user at session start, 2026-06-16)

1. **W3 alone — not W3 + W4 bundled.** Ship place-order + the confirmation screen now; W4 order-tracking (`GET /orders/{orderId}` with polling) becomes a thin follow-on PR reusing the new `client/src/orders/` folder (`OrderStatusViewSchema` + `orderQueries`). Keeps the OTel trace-front-door slice focused and reviewable. (Rejected: bundle W3 + W4 — defensible since W4 is mostly the same read + a route + a poll, but larger and dilutes the headline beat.)
2. **The place-order mutation lives in `client/src/orders/`** — not `client/src/cart/`. It *produces* an Order (mirroring backend `Orders/Features/PlaceOrder`), so it belongs with the order read it kicks off; the cart-badge reset is just an `invalidate(cartKeys.mine)` the mutation does on success. Parallel to `useAddToCart` living in `cart/` because *that* targets the cart. `orders/` must exist anyway for the W3 read schema. (Rejected: `cart/` — it's the button on the cart screen and ends the cart's life, but it creates an order, not a `CartView` mutation.)
3. **`OrderStatusViewSchema.status` is a `z.enum` of the 5 known values** — not a loose `z.string()`. Catches an unexpected/typo'd status loud-and-located at the boundary (the Convention 2 spirit; the Zod skill's `schema-use-enums` for a fixed set; teaching value — status is a known domain enum). It couples the schema to today's `OrderStatus` set, but a new backend status → a schema bump is the convention anyway (schemas are hand-written, not generated). (Rejected: `z.string()` — forward-compatible to a 6th status with no bump, but lets an unknown status slip past the boundary to the UI.)
