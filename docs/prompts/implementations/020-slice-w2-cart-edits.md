## Framing

W2 (#58) shipped the cart-review screen as **read-only**; W1 (#60) shipped the first optimistic mutation (`useAddToCart`) and the browse → add → badge → W2 loop. The wireframe's `[-] N [+]` steppers and `[x]` remove were deliberately left absent — "the edit/checkout controls deliberately absent rather than stubbed" (CartPage.tsx). This slice makes the W2 rows editable: each control fires an optimistic mutation that reconciles against the refetched `CartView`, and the cart badge + Total update for free (both derive from the same cart query).

It bundles two workshop slices because they share one screen and one file — 3.2 and 3.3 both edit the same `CartPage` rows and add hooks to the same `cartMutations.ts`. The backend shipped them together in PR #39; the consolidate-slice-PRs stance carries to the frontend. This is also the slice where the command layer learns to handle a **`204 No Content`** response — both edit commands return 204 (no body), which W1's always-parses-a-body `postCommand` cannot consume — and the SPA issues its **first DELETE**.

## Goal

On the cart-review screen (`/cart`), each line row carries a `[-] N [+]` quantity stepper and an `[x]` remove. `[+]` sends `ChangeCartItemQuantity { newQuantity: N+1 }`; `[-]` sends `N-1` (disabled at N=1 — removal is only ever `[x]`, locked decision 2); `[x]` sends `RemoveCartItem`. Each is **optimistic**: `onMutate` applies the pure merge against `cartKeys.mine`, `onError` rolls back, `onSettled` invalidates so the guess reconciles against the refetched `CartView`. Per-row `isPending` disables that row's controls (mirror W1's per-card pattern) without freezing the table. The integer-cents Total recomputes from `lines` for free. `npm run build` + `npm run test` green. The Aspire stack boots, the SPA loads at `:5173`, and editing a line in the browser reconciles while the `DELETE` / `POST …/quantity` spans appear **SPA → Orders** in the dashboard.

## Locked decisions (forks resolved with the user at session start, 2026-06-16)

1. **Optional-schema `postCommand` + a sibling `deleteCommand` — additive, the W1 call site stays untouched.** `postCommand`'s `schema` parameter becomes **optional** (overloaded: schema provided → returns the parsed `T`; omitted → returns `void` and does **not** call `.json()`, which a 204 would throw on). A new `deleteCommand(url, ctx)` issues the SPA's first DELETE (no body either way), sets the `X-Customer-Id` header, throws `ApiError` on non-2xx. The shipped W1 `useAddToCart` call site is **unchanged** (still passes `AddToCartResponseSchema`). (Rejected: option (b), a unified `sendCommand(method, url, ctx, { body?, schema? })` — covers all three shapes in one helper but churns the shipped W1 call site and reads less verb-first at the call site.)
2. **`[-]` disabled at quantity 1 — one control = one command.** The backend rejects `newQuantity <= 0` ("use remove-item for zero"). At qty 1 the `[-]` button is **disabled**; removal is only ever the explicit `[x]`. This keeps the teaching mapping clean (each control maps to exactly one command) and means the SPA never sends a non-positive quantity — there is no `400 NonPositiveQuantity` path to handle in the happy UI. (Rejected: option (b), `[-]` at 1 issues `RemoveCartItem` — friendlier/fewer-clicks decrement-to-zero, but couples two commands into one control and blurs the mapping.)
3. **Bundle 3.2 + 3.3 in one PR.** Siblings — same screen rows, same `cartMutations.ts`; the backend shipped them together (PR #39); the consolidate-slice-PRs stance. (Confirmed by the kickstart, not a real fork.)
