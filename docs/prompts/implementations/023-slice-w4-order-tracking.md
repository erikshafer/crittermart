## Framing

W3 (#62) shipped checkout → the Order Confirmation screen and introduced the `client/src/orders/` folder — `OrderStatusViewSchema` (the 3rd read surface, already modeling `lines`), `orderQueries.ts` (`orderQueryOptions(orderId)`), and the pure-`orderId`-prop page pattern. It deliberately left `[ Track this order ]` **disabled** ("Order tracking arrives in the next slice (W4)"). This slice is that next slice: it reuses the schema + query **wholesale** and adds only three genuinely-new things — a `/orders/$orderId` route, a `refetchInterval` poll that **stops itself at a terminal status**, and the fuller receipt render the W4 wireframe (§ 5.1) calls for.

W4 is the beat where the journey *completes without the customer acting*: they placed the order at W3 and now **watch** the status settle. There is no push round one (ADR 015 R5, explicitly unlike the CritterBids SignalR sibling), so the screen converges the way the cart's optimistic rule already taught — re-query, don't render a guess — except here it polls a read instead of reconciling a mutation. The status walks `awaiting_confirmation → stock_reserved → payment_authorized → confirmed`, or settles on `cancelled`. The poll stops the instant the status is terminal (`confirmed`/`cancelled`) — the order will not change further, so idle network churn is pointless.

## Goal

Tapping `[ Track this order ]` on W3 navigates to `/orders/$orderId`. `OrderStatusPage` reads `orderQueryOptions(orderId)` → `GET /orders/{orderId}` (`OrderStatusView`, Zod-parsed at the boundary) with `refetchInterval` set to **~2.5 s while the status is non-terminal** and **`false` once terminal**, `refetchIntervalInBackground: false`, plus a manual `[ Refresh ]` button. It renders the § 5.1 W4 receipt: the order id, a status **stepper** derived client-side from the current status (`awaiting → stock reserved → payment authorized → confirmed`, the real 5-state lifecycle; `cancelled` as a terminal treatment), per-line rows (name · sku · ×qty · line total), and the server-computed Total (read directly off the view — not re-summed). Honest loading/error states (a 404 here is a genuine error, *not* the cart's empty-state mapping). A11y: status conveyed as text not color-only, `aria-live="polite"` on the converging status, no layout shift between polls. `npm run build` + `npm run test` green; the place → poll-convergence (`awaiting_confirmation → stock_reserved → confirmed`) + the cross-BC span chain confirmed live in the browser + Aspire dashboard (the wire `GET /orders/{orderId}` progression is curl-confirmable).

## Locked decisions (forks resolved with the user at session start, 2026-06-16)

1. **Build W4 — not the Stock pilot.** Complete the storefront spine (browse→cart→checkout→**track**) now; the Stock read/write split (the last ADR 020/021 rollout) remains an equally-teed-up backend tidy for a later session. (Rejected: Stock pilot — well-teed-up and smaller than the Order pilot, but it's a backend tidy that can follow; W4 finishes the demo's customer-visible journey.)
2. **Poll ~2.5 s, stop at terminal, + a manual `[ Refresh ]` button.** `refetchInterval` is a function returning `2500` while non-terminal and `false` once `confirmed`/`cancelled`; `refetchIntervalInBackground: false`; plus a manual Refresh (belt-and-suspenders + a clean interaction/test hook). (Rejected: no Refresh button — cleaner card but loses the "show me now" affordance; ~5 s cadence — calmer for span-watching but slower to feel "live".)
3. **Full per-line receipt; drop the Placed timestamp.** Render the full receipt (per-line rows + server Total) — richer than W3. **Drop** the wireframe's `Placed … UTC` line: `OrderStatusView` carries **no timestamp field**, and adding one is a backend change fenced out of this screen-only slice. Logged as a **backend gap**. (Rejected: a client-side "Last updated" time — honest about liveness but not the order's placed-at, extra chrome for little value; adding `placedAt` to the backend — breaks screen-only scope + needs an OpenSpec touch.)
4. **Client-side stepper over the real 5-state lifecycle.** A presentation-only stepper derived from `status`: `awaiting → stock reserved → payment authorized → confirmed` (the real enum, not the wireframe's collapsed 3-step), with `cancelled` shown as a **terminal treatment**. No new contract — pure derivation. (Rejected: current-status-as-text-only — leanest but loses the "where in the journey" story; text + one-line path — middle ground, but a stepper reads clearer.)

> **Two honest gaps surfaced by decisions 3 + 4 — both logged, neither built (screen-only fence):**
> - **No placed-at timestamp** on `OrderStatusView` (decision 3).
> - **No cancellation *reason*** on `OrderStatusView` — `OrderCancelled` folds to a bare `Status = "cancelled"` (`src/CritterMart.Orders/Ordering/OrderStatusView.cs:44`), so the three reasons (`stock_unavailable`/`payment_declined`/`payment_timeout`, known to Narrative 004) are **not on the wire**. W4 shows a generic "Cancelled" — it cannot show the reason or the step it failed at without a backend projection change. Both gaps belong on a future "enrich `OrderStatusView`" slice, not here.

## Working pattern

Author the pure derivation first (`orderStatusJourney.ts`: `isTerminalStatus(status)`, `pollIntervalFor(data) → number | false`, `deriveJourney(status) → progress steps | cancelled`) with its unit test (all 5 statuses → the right step states; terminal → `false` poll) — the hard part (the stop-on-terminal rule + the stepper derivation) made testable without React-Query timing. Then `OrderStatusPage.tsx` (a pure `orderId`-prop component: `useQuery({ ...orderQueryOptions(orderId, ctx), refetchInterval: (q) => pollIntervalFor(q.state.data), refetchIntervalInBackground: false })`; loading / error / success; the receipt table + the stepper + the `[ Refresh ]` button; `aria-live` on the status) with its test (render each status → the right display; Refresh triggers a refetch). Then flip W3's button → a `<Link to="/orders/$orderId" params={{ orderId }}>` and update its test to wrap in a memory router and assert the link. Then `router.tsx` (the `/orders/$orderId` route — a thin component reading `route.useParams()` → `<OrderStatusPage orderId={…} />`; update the routes-comment). `npm run build`/`test` green → boot Aspire (kill orphan `CritterMart.*` first), seed products + stock + an open cart, place an order, open `/orders/$orderId`, watch the poll-convergence + the SPA → Orders → (RabbitMQ) → Inventory span chain → converge the skill (v7) + bump Narrative 005 → v1.6 → README counts → retro. One PR on `feat/w4-order-tracking`; the user merges.
