## Framing

The Cart pilot (#59) split `CartView`-the-mutable-aggregate into `Cart` (write) + `CartView` (read) and moved `Cart/` → `Shopping/`. Order was deliberately deferred: Program.cs still reads *"OrderStatusView still doubles as aggregate + read model — the Order/Stock read/write split is the follow-up to this Cart pilot (ADR 020), not done here."* This PR is that follow-up.

`OrderStatusView` today plays **three** roles — a sharper conflation than Cart's: (1) the `StartStream<OrderStatusView>` write target in `PlaceOrder`; (2) the **PMvH decision state** — every cross-BC outcome handler (`StockReservedHandler`, `PaymentDecisionHandler`, `PaymentTimeoutHandler`) does `FetchForWriting<OrderStatusView>` and reads `.Status`/`.Total`/`.Lines` to guard idempotently and decide the next cascade; (3) the public read model served over `GET /orders/{orderId}`. ADR 020 says these are separate concerns. So `Order` becomes the protected write aggregate *and* the process-manager state the handlers fold over, and `OrderStatusView` becomes a pure read projection.

This is the design-return for the Orders BC after three consecutive frontend implementations (W1/W2-edits/W3) — closing the gap between the implementation runs and the backend design, exactly as the cadence rule intends. It is contract-preserving: the `OrderStatusView` wire shape is unchanged, so the W3 confirmation screen (and the future W4) are untouched.

## Goal

`PlaceOrder` starts the order stream as `StartStream<Order>`; the three handler families fetch and fold `Order`; `OrderStatusView` is projected independently and still serializes byte-for-byte as `{ id, customerId, status, lines: [{ sku, quantity, name, price }], total }`. The Order slice's files live in `Ordering/` (namespace `CritterMart.Orders.Ordering`), the aggregate keeping its canonical name `Order` (a verb namespace never collides with the noun type — ADR 021). The `Order` aggregate carries the write-side decision state the PMvH needs (`Status`, `Total`, `Lines`, plus `Id`/`CustomerId`) and is never serialized over HTTP. Full solution **build + test green**, `dotnet format` clean, and the cross-BC flow (`PlaceOrder → ReserveStock → StockReserved → AuthorizePayment → OrderConfirmed`, and the cancel/timeout branches) still drives end-to-end.

**No OpenSpec / workshop / narrative change** — a contract-preserving model refactor (the `OrderStatusView` wire shape is unchanged, so no spec the SPA or another service binds moves).
