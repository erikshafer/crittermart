## Framing

W4 tracks **one** order by id (`GET /orders/{orderId}`), but a Customer has no way to see all their orders. Narrative 005 names this twice (Moment 5's closing line; the "No 'My Orders' history" bullet) and the audit logged it as **Gap #3** — the last named round-two deferral. This slice closes it.

It is **not** new modeling. Every placed order already has an inline `OrderStatusView` document carrying `customerId`, `status`, `total`, and (slice 025) `placedAt`. The list is a **filtered query over those documents** — the composition of two in-repo precedents: `GET /products` (a list read = `Query<Product>().ToListAsync()`, `BrowseProducts.cs`) and `GET /carts/mine` (a customer-keyed read = `Query<CartView>().Where(v => v.CustomerId == id)`, `ViewMyCart.cs`). The "My Orders" list is their composition (`FirstOrDefault` → `ToList`), served by a non-unique `OrderStatusView.CustomerId` index (the mirror of the `CartView` read-model index, `Program.cs:100`).

## Goal

`GET /orders/mine` returns the requesting customer's `OrderStatusView`s (resolved by the `X-Customer-Id` header), ordered newest-first by `placedAt`, every lifecycle state included, each cancelled order carrying its `cancelReason`; a customer with no orders gets `200 []`; a request with no identity header gets `400`. The `MyOrdersPage` lists them, each row a typed `<Link>` into the W4 track screen, a cancelled row naming its reason. Full backend suite green (`dotnet test`), `dotnet format` clean; `client/` build + vitest green; a live boot confirms the list returns the right orders newest-first, scoped per customer, empty for a fresh customer, and `400` without a header — against the real Aspire stack with the cross-BC saga reaching `confirmed`.
