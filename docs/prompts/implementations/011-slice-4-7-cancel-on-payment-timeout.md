## Framing

Slice 4.6 closed the *decline* path: the provider answers "no" and the order cancels itself. Slice 4.7 closes the *silence* path: nobody ever answers. An order that never reaches a terminal state — because Inventory's reply was lost, or the payment provider never decided — must not wait forever; a deadline set when the order was placed cancels it. This is the Bruun temporal-automation pattern (Workshop § 5 slice 4.7, § 6 GWT): a **scheduled `OrderPaymentTimeout` self-message** carries the deadline, an inline **`OrdersAwaitingPayment*`** projection makes the waiting visible as a todo-list read model, and a **terminal-state guard** makes the timer harmless when the order settled first. Every prior slice reacted to a message that arrived; this one reacts to a message that Orders sent *to its own future self*.

## Goal

When an order is placed, `PlaceOrder` additionally cascades a **scheduled** `OrderPaymentTimeout { orderId }` self-message with a config-driven delay (`Orders:PaymentTimeout`, default 10 minutes). When it fires, the new `PaymentTimeoutHandler` guards on the Order stream: terminal (`confirmed`/`cancelled`) → silent no-op; otherwise it appends `OrderCancelled { reason: "payment_timeout" }` (status `cancelled`) and **always cascades `ReleaseStock { orderId, lines }`** to Inventory — reusing slice 4.6's contract and `ReleaseStockHandler` **unchanged**. A new inline `OrdersAwaitingPayment*` projection (row created on `OrderPlaced` with the deadline, **deleted** on terminal events — Marten conditional delete) is the Bruun todo-list, exposed at `GET /orders/awaiting-payment`. Proven without real-time waiting: tracked sessions assert the scheduled envelope at placement; the handler is tested by direct invocation. `openspec validate --strict` passes; full solution green.

## Spec delta

A new OpenSpec change `slice-4-7-cancel-on-payment-timeout` with **one** capability delta (no new capability, no Inventory delta — the release side reuses 4.6's shipped requirement):
- **`order-lifecycle`** (two ADDED requirements):
  1. *Cancel an order on payment timeout* — placing an order schedules `OrderPaymentTimeout` for a configurable deadline; when it fires on a non-terminal order, append `OrderCancelled { payment_timeout }` and send `ReleaseStock` with the order's lines; when it fires on a terminal order (confirmed or already cancelled), append nothing and send nothing; idempotent under duplicate delivery.
  2. *Track orders awaiting payment* — an inline `OrdersAwaitingPayment*` read model holds a row per non-terminal order (order id, customer, total, deadline), created at placement and removed when the order reaches a terminal state; queryable as the Bruun todo-list.

## Locked decisions (collaborative forks, resolved with the user this session)

1. **Schedule at order placement, not at stock reservation.** Workshop-faithful (slice 4.1's writes-to names the schedule). One deadline covers the whole journey to terminal — it also catches an Inventory reply that never arrives (order stuck at `awaiting_confirmation`), not just a silent provider. The reason stays `payment_timeout` (workshop vocabulary) regardless of which gate the order was stuck at.
2. **Always cascade `ReleaseStock` on timeout-cancel — even when Orders never recorded the stock grant.** This is what survives the Workshop § 4.7 delayed-grant race: Inventory granted, the reply was lost, Orders cancels at `awaiting_confirmation`, and Inventory is still holding stock. Inventory's per-SKU reservation guard (4.6) makes the release a safe no-op when nothing is held. (4.6 releases only at the payment gate because there the guard *proves* the reservation; 4.7 cannot prove either way, so it releases unconditionally and lets Inventory decide.)
3. **`OrdersAwaitingPayment*` is a todo-list read model, not load-bearing for the handler.** Inline single-stream projection with a **conditional delete**: doc created on `OrderPlaced` (orderId, customerId, total, deadline), deleted when a terminal event folds. The timeout handler guards on the Order stream (single source of truth); the projection is the observable Bruun todo-list + a `GET /orders/awaiting-payment` demo endpoint. New teaching beat: Marten conditional deletes in projections.
4. **Timeout duration is config-backed with a default** — `Orders:PaymentTimeout` (default `00:10:00`), read in `Program.cs`. Tests never wait for real time; the Aspire demo can set it short (e.g. 30s) to show an order cancel itself live.
5. **Test strategy: no real-time waiting, no new cross-BC smoke** *(session-runner recommendation, confirmed at the prompt review gate)*. The scheduled envelope is asserted via tracked session at placement; `PaymentTimeoutHandler` behavior (cancel + release cascade, terminal no-op, duplicate no-op) is tested by invoking `OrderPaymentTimeout` directly; the projection fold + conditional delete get pure unit tests. The cross-BC `ReleaseStock` path over the real broker is already proven by 4.6's decline→release smoke and the contract + Inventory handler are reused byte-for-byte — a timeout smoke would re-prove the same wire.
