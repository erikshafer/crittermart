## Framing

Every placed order cascades a durable `OrderPaymentTimeout` self-message `DelayedFor(Orders:PaymentTimeout)` (default 10 min; `PlaceOrder.cs:83`); every created cart cascades a `CartActivityTimeout` `DelayedFor(Orders:CartActivityTimeout)` (default 2 h; `AddToCart.cs:92`). These are the two Bruun temporal automations (Workshop 001 slices 4.7 + 3.4; ADR 007). Wolverine stamps the scheduling request's W3C trace context onto every envelope it sends, **including delayed ones**, so by default the fired timeout's execution span parents back **into** the original placement / add-to-cart trace. OpenTelemetry holds the trace "open" until its last span closes — and that last span is the timeout firing minutes-to-hours later — so the trace's Duration inflates to the entire timeout window even though the request itself completed in ~50 ms. A placement trace that reads ten minutes is a lie about the request's latency, and it is the one wart on the demo's centerpiece visual. (Documented as a known finding in `docs/demo-runbook.md` Step 6, with a forward-reference to "a future slice"; this is that slice.)

## Goal

A `POST /orders` (and `POST /carts/{id}/items`) trace's waterfall reads tens of milliseconds **no matter when it is opened** — the deferred deadline never re-joins it. The fired timeout runs in its **own root trace**, related to the originating request by a span **link** (the OpenTelemetry idiom for asynchronous follow-up work), so it stays fully observable and navigable back to the request that armed it. Domain behavior is byte-for-byte unchanged (the same cancel + always-release on payment timeout, the same no-op / reschedule / abandon on cart inactivity); the full Orders suite stays green and gains a deterministic guard on the new rooting. Confirmed against a live Aspire boot (services healthy with the changed handlers; happy + decline saga routes drive correctly; no telemetry export errors). The trace *visual* is owner-captured on the Aspire dashboard (chore/002 precedent — the Blazor/SignalR surface does not render headless).

## Orientation

1. **CLAUDE.md** — one-prompt-one-PR, no-opportunistic-edits, `{type}/{slug}` branch (`feat/deferred-timeout-linked-traces`, created). The seam + the two handler edits + the tests + the runbook callout land in **one** PR; the owner merges.
2. **`wolverine-observability-opentelemetry-setup`** skill — confirms the documented lever (`[WolverineLogging(telemetryEnabled:false)]`) and that Wolverine propagates trace context across message boundaries; there is **no** public `DeliveryOptions` flag to emit a new root, which is what forces the linked-root-by-hand approach.
3. **The wire points**: `PaymentTimeoutHandler.Handle` and `CartAbandonmentHandler.Handle` (add the attribute + `Envelope` param + wrap the body); the new `TemporalAutomationTracing` seam; `ServiceDefaults/Extensions.cs:60` (`AddSource(builder.Environment.ApplicationName)` — already collects an `ActivitySource("CritterMart.Orders")`, so no ServiceDefaults edit).
4. **Two facts verified against the pinned 6.8 assembly before the fork was posed**: `Envelope.ParentId` *is* the W3C traceparent (the Wolverine↔MassTransit interop maps it straight to/from `MassTransitHeaders.ActivityId`), parseable by `ActivityContext.TryParse`; `WolverineLoggingAttribute` lives in `Wolverine.Attributes` with defaulted ctor params, so `[WolverineLogging(telemetryEnabled:false)]` compiles standalone.
5. **The .NET rooting gotcha**: `ActivitySource.StartActivity(name, kind, parentContext: default)` silently re-parents onto `Activity.Current` (an invalid `ActivityContext` is treated as "no override"). Forcing a true root requires clearing `Activity.Current` first — caught by the seam test, not by reading docs.

## Working pattern

**Seam first**: author `TemporalAutomationTracing` (static `ActivitySource("CritterMart.Orders")` + `StartLinkedRoot(name, envelope)` that parses `envelope.ParentId` into an `ActivityLink` and starts a new root). → wrap both handlers: `[WolverineLogging(telemetryEnabled:false)]`, inject `Envelope`, `using var activity = StartLinkedRoot(...)`, tag the outcome on each path. → `dotnet build` (Orders). → **write the seam test with an ambient placement Activity set as Current** and assert the timeout roots away from it + links back — let the test *prove* the rooting (it caught the `parentContext: default` re-parent). → `dotnet test` Orders (89 existing must stay green; +2 new). → **Live boot**: kill orphans, boot the AppHost (background + log), poll `/health` on `:5101/:5102/:5103`, confirm auto-seed; drive the happy + decline saga routes; confirm domain intact + zero log errors; tear down. → update the runbook callout. → README counts; retro. One PR; **the owner merges**.

## Out of scope

- **No conversion of the timeout handlers from cascade-return to inline `IMessageBus` publish** — the cancel path keeps cascading `ReleaseStock` (the project's cascading-over-PMvH preference). Consequence, documented in the retro: the cascaded `ReleaseStock` sends *after* the handler's activity disposes, so it lands as its own clean root rather than nested under the timeout trace. Nesting it would mean inline-publishing within the activity scope — a deliberate non-choice here (timeout-fired cancel is a ~never-demoed prod path).
- **No ServiceDefaults / AppHost edits** — `AddSource(ApplicationName)` already collects the source; no short-timeout demo affordance is added to the AppHost (the owner can shorten `Orders:PaymentTimeout` ad hoc to watch the fix live).
- **No ADR** — Orders-local, self-documented in the seam; an ADR 005 cross-reference is flagged in the retro as optional, not folded in.
- **No embedded screenshots** — the trace visual is owner-captured (chore/002 precedent); the runbook callout is the navigation note.
- **No change to `Orders:PaymentTimeout` / `Orders:CartActivityTimeout` defaults** — the durable scheduling behavior (ADR 007, slices 4.7/3.4) is untouched; only the *trace topology* of the fired message changes.
