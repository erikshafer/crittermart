## Framing

The Cart pilot (#59) split `CartView`-the-mutable-aggregate into `Cart` (write) + `CartView` (read); the Order pilot (#63) did the same for `OrderStatusView` → `Order` + `OrderStatusView`. `StockLevelView` is the last of the three round-one event-sourced aggregates still playing two roles at once: (1) the `FetchForWriting<StockLevelView>` write target on all four stock write paths (`ReceiveStock`, `ReserveStock`, `ReleaseStock`, `CommitStock`); (2) the public read model served over `GET /stock/{sku}`. ADR 020 says these are separate concerns. This PR is that split, applied to Inventory.

Stock's conflation is the **simplest** of the three — no PMvH (Inventory has no process manager; the cross-BC handlers are one-shot reactions to `Contracts` messages, not a saga over the Stock stream), and the `Stock/` folder already needs no rename because `StockLevel ≠ …Stock` (the type↔namespace collision that forced `Shopping/`/`Ordering/` does not arise). So this is ADR 020 **only**: introduce the `StockLevel` write aggregate, convert `StockLevelView` to a dedicated read projection, repoint the four write paths, and register both as inline snapshots. The read shape is byte-for-byte preserved, so nothing that binds `GET /stock/{sku}` (tests, the demo) changes.

One mechanic is genuinely new versus Cart/Order: **`StockReceived` is both the genesis event and a repeatable one** (a SKU's first receipt starts the stream; later receipts accumulate `Available`). The old mutable projection leaned on Marten auto-constructing the empty class on the genesis event and applying every `StockReceived` uniformly. A `sealed record` has no construct-then-mutate path, so the aggregate needs **both** a `Create(StockReceived)` (genesis → opens the level) **and** an `Apply(StockReceived, StockLevel)` (subsequent → accumulates). Marten dispatches `Create` when the snapshot is absent and `Apply` when it exists — the existing `receiving_additional_stock_accumulates_available` integration test (100 + 50 = 150) is the binding check that this dispatch is correct.

## Goal

The four write handlers `FetchForWriting<StockLevel>`; `StockLevel` carries the write-side decision state the idempotency guards read (`Available` for the reserve gate, `Reservations` for the per-SKU release/commit no-op guards) and is never serialized over HTTP. `StockLevelView` is projected independently and still serializes byte-for-byte as `{ id, available, reserved, committed, reservations }` over `GET /stock/{sku}`. Both are self-aggregating `Snapshot<T>(SnapshotLifecycle.Inline)`. The `Stock/` folder and the `CritterMart.Inventory.Stock` namespace are unchanged (no ADR 021 rename). Full solution **build + test green**, `dotnet format` clean, and a live `place → ReserveStock → CommitStock` cycle still decrements `Available` / increments `Committed` with the read shape unchanged.

## Spec delta

**No OpenSpec / workshop / narrative change** — a contract-preserving model refactor (the `StockLevelView` wire shape is unchanged, so no spec the SPA or another service binds moves; the `stock-management` capability's SHALLs are behaviour, all preserved).
