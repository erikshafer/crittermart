## Why

Slice 6.5 shipped CritterMart's second DCB ([PR #149](../../../docs/retrospectives/implementations/041-slice-6-5-per-customer-redemption-dcb.md)): a composite `(coupon × customer)` boundary enforcing one-redemption-per-customer at checkout. It deliberately parked two storefront-UX follow-ons ([Workshop 003 §8 item 6](../../../docs/workshops/003-promotions-event-model.md)) — a **per-customer advisory preview** and **tailored copy** for the `CouponAlreadyRedeemedByCustomer` refusal. This change models and specifies both as **slice 6.6** (Workshop 003 v1.4, Narrative 011 v1.3 Moment 6).

The gap is a customer-experience one, not a correctness one. The rule works; it just announces itself at the worst possible moment. A Customer applies `FIRSTORDER`, watches their total drop 15%, taps **Place Order**, and only then learns they used the offer months ago — in a sentence that reads like a database constraint. Slice 6.6 moves that news to cart review and rewrites it in the storefront's own voice. **Nothing about the invariant changes**: the composite DCB append at checkout remains the sole authority, and the preview is advisory even when it is personally accurate.

Two forks were settled with the owner at session start (`AskUserQuestion`):

1. **One slice, not two.** The preview and the copy are one promise ("this coupon speaks to *you*, not just to the crowd") told at two moments of one screen. The copy half alone carries no command, no event, and no view — it is not a vertical slice on its own.
2. **Enrich the existing route, don't fork it.** `GET /coupons/{code}/validate` becomes **optionally authenticated** rather than gaining an `[Authorize]`-gated sibling or being hard-`[Authorize]`'d. An anonymous caller's answer is unchanged byte-for-byte (no storefront breakage, no regression to a shipped anonymous contract); an authenticated caller's answer gains one status.

**The modeling finding that shaped the slice** (surfaced by reading the code, not the docs). Through slice 6.5 the `(coupon × customer)` pair existed **only** as a DCB tag — no *event member* named the customer. That suffices for a write-time boundary, which is opened by tag query. It does **not** suffice for a persisted read model: a Marten `MultiStreamProjection` routes events by an **event member** (`Identity<CouponRedeemed>(e => e.CouponId)` — the convention the shipped `CouponUsageView` already uses), and a tag is a write-side query mechanism, not a grouping key. The per-customer view a preview needs therefore cannot be projected from the events as they stand. Both redemption events must carry `customerId`.

## What Changes

Everything executes in the **Orders** bounded context. No new service, no new contract, no broker topology, **no new DCB, no new tag type, and no write of any kind** — slice 6.6 is the only slice in Workshop 003 that appends nothing.

- **Both redemption events gain `customerId`.** `CouponRedeemed { …, customerId }` and `CouponRedemptionReleased { …, customerId }`, as optional defaulted record fields — the identical non-breaking event-evolution move `oneRedemptionPerCustomer` (6.5) and `perCustomer` (6.5) already made. Old serialized events fold with the field absent.
- **A new persisted advisory view.** `CustomerCouponUsageView` — an **inline** `MultiStreamProjection` keyed by the composite identity `"{couponId}|{customerId}"` built from the events' **members**, folding `CouponRedeemed` `+1` and `CouponRedemptionReleased` `−1`. Inline for the reason `CouponUsageView` is inline: no async daemon runs this round (Workshop 003 §8 item 2). It is the first *persisted* per-customer coupon state, and it is **distinct from** the never-persisted `CustomerCouponUsage` DCB boundary state — same arithmetic, different existence, and only the boundary is ever the authority.
- **The validate query becomes optionally authenticated and gains a fourth status.** When the caller presents a bearer token **and** the resolved definition carries `oneRedemptionPerCustomer = true`, the query consults `CustomerCouponUsageView` for the `(coupon × caller)` pair and answers `already_redeemed` when that pair's net count is `≥ 1`. Precedence mirrors checkout exactly — `invalid` → `already_redeemed` → `exhausted` → `valid` — so the preview and the authority agree about **why**, not merely **whether**. An anonymous caller, or a global-cap-only coupon, gets today's answer unchanged; `already_redeemed` is unreachable in either case.
- **The refusal copy is rewritten.** The `409 CouponAlreadyRedeemedByCustomer` **detail** moves from *"Coupon 'X' may be redeemed only once per customer, and you have already redeemed it."* to **"You've already used this coupon — remove it to continue, or try another."** — parallel in shape to the sibling `CouponExhausted` copy. The status code and the `CouponAlreadyRedeemedByCustomer` title token are **unchanged**, so no machine-readable contract moves.

**Forward-only, and honestly so.** Redemptions appended before the `customerId` field exists cannot be attributed, so the preview may **under-warn** (a Customer whose only redemption predates the field sees `valid` and is refused at checkout exactly as today) and can never **over-warn** (it will not wrongly accuse someone entitled to the discount). That one-sided error is tolerable *precisely because* the read is advisory — the same license `CouponUsageView` already has to lag — and it is specified as its own scenario rather than left to be discovered.

## Capabilities

### Modified Capabilities

- `coupon-promotion`: **Validate and price a coupon at cart review** is MODIFIED — the query becomes optionally authenticated and gains the `already_redeemed` status with its checkout-mirroring precedence rule, while the anonymous contract is pinned unchanged. **Enforce one redemption per customer for a per-customer coupon at checkout** is MODIFIED for the refusal-copy delta only — the `409`, the title token, the boundary mechanic, and every existing scenario are untouched. One requirement is **ADDED** — *Track advisory per-customer coupon usage* — covering the `customerId` event fields, the inline `CustomerCouponUsageView`, its distinction from the DCB boundary state, and its forward-only limitation. The global per-coupon cap and release requirements are unchanged.

## Impact

- **Orders — new.** `Promotions/CustomerCouponUsageView.cs` (the document + its `MultiStreamProjection`; `partial` is load-bearing per `docs/skills/marten-projection-conventions`).
- **Orders — modified.** `Promotions/{CouponRedeemed,CouponRedemptionReleased}.cs` (the `customerId` field); `Features/PlaceOrder.cs` (populate `customerId` on both events at the append sites; reword the `CouponAlreadyRedeemedByCustomer` detail); `Ordering/{StockReservationOutcomeHandlers,PaymentHandlers,PaymentTimeoutHandler}.cs` or `Promotions/CouponRelease.cs` (carry `customerId` onto the release — the customer id is already threaded to these sites for the composite tag); `Features/ValidateCoupon.cs` (optional auth, the fourth status, the precedence ladder); `Program.cs` (register the projection inline).

- **Tests.** New `CouponTests`: authenticated already-redeemed → `already_redeemed`; authenticated not-redeemed → `valid`; anonymous caller of a spent per-customer coupon → `valid` (the unchanged contract); a global-cap-only coupon never answers `already_redeemed`; precedence (already-redeemed **and** globally exhausted → `already_redeemed`); a cancelled redemption restores the preview to `valid`. Pure-fold unit tests for the new projection.
- **Downstream (the implementation session).** `design.md` + `tasks.md` in this change; prompt + retro `docs/{prompts,retrospectives}/implementations/NNN-…`; demo-runbook / demo-traffic beat if the preview is demoed; READMEs.
