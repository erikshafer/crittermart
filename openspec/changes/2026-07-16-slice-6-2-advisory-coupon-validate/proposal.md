## Why

The DCB core shipped ([PR #144](../../../docs/retrospectives/implementations/039-slices-6-1-6-3-6-4-coupon-dcb.md), slices 6.1/6.3/6.4): a coupon can be defined, redeemed under a global cap, and released on cancellation, and the checkout append is the authority on whether a code applies. What the storefront customer cannot yet do is **see the discount before committing** — type a code on cart review, watch the total drop, and get told inline when a code is not valid or no longer available. [Workshop 003](../../../docs/workshops/003-promotions-event-model.md) modeled that as slice **6.2**, the P1 storefront-UX half the DCB PR deliberately deferred, and [Narrative 011](../../../docs/narratives/011-customer-redeems-coupon.md) named it forthcoming ("What the Customer does *not* yet see").

This change implements slice 6.2: a **read-only advisory validation query** (`GET /coupons/{code}/validate`) plus the **W2 cart-review coupon field** and the **W3 order-confirmation discount line** that bind it. The query is advisory **by design** (Workshop 003 §3, §6.2): it reads `CouponView` (the definition) and `CouponUsageView` (the advisory net count) and **writes nothing** — the checkout DCB append (slice 6.3) remains the sole authority. The advisory answer may lag or a slot may free by cancellation between the check and checkout; that gap is the deliberate advisory-vs-authoritative teaching contrast, not a bug to close. A stale-comment tidy rides along: ~10 `client/src` files still describe the identity transport as the retired `X-Customer-Id` header (it is `Authorization: Bearer` since the ADR 023 hard cutover) — comment-only, and the W2 frontend session is its natural host.

## What Changes

Everything executes in the **Orders** bounded context and the **storefront SPA**. No new backend write, no schema change, no new contract — the query reads two read models the DCB core already maintains, and W3 binds fields the API already returns.

- **6.2 backend — the advisory validate query.** A read-only `GET /coupons/{code}/validate` resolves the code against `CouponView` and answers a **discriminated status**: `valid` (with `discountPercent`), `invalid` (no definition resolves), or `exhausted` (a definition resolves but `CouponUsageView`'s net count has reached `cap`). It returns `200` in every case — checking a code is not an error — and writes nothing. The dollar discount is **not** computed here: the client already holds the cart total (it sums lines in integer cents for the displayed Total) and prices `discountPercent` against it, so the advisory query stays uncoupled from cart money and does not duplicate `PlaceOrder`'s pricing math (see `design.md`, the endpoint-shape decision Erik settled at session start).
- **6.2 frontend W2 — cart-review coupon field.** `CartPage.tsx` gains a coupon input + **Apply** that fires the advisory query, shows the discounted-total preview (`Subtotal / Discount (CODE) / Total`), and renders inline errors: "This code isn't valid." (`invalid`) / "This coupon is no longer available." (`exhausted`). The applied code is held in **UI state** — a reload forgets it (accepted round-one behavior, Workshop 003 §8 item 11) — and rides checkout as the existing **`POST /orders?couponCode=`** query parameter (slice 6.3, already built). An empty field is slice 4.1's unchanged checkout.
- **6.2 frontend W3 — order-confirmation discount line.** `OrderConfirmationPage.tsx` renders `Subtotal / Discount (CODE) / Total` when a discount is present, unchanged otherwise. Pure binding: `EnrichedOrderView` already returns `subtotal` / `discount` / `couponCode` (shipped in #144); this change adds those three fields to the SPA's `OrderStatusViewSchema` (they were not yet parsed) and binds them. No backend change.
- **Tidy — stale header-transport comments.** ~10 `client/src` files (`api/client.ts` historical note excepted — it is already correct) describe the customer-identity transport as `X-Customer-Id`; the ADR 023 hard cutover retired that header for `Authorization: Bearer` (the `sub` claim is the trust boundary). Comment-only corrections in `cart/`, `orders/`, `catalog/` and their tests.

## Capabilities

### Modified Capabilities

- `coupon-promotion`: gains one requirement — the advisory cart-review validation query. It is a **read** over the capability's own `CouponView` and `CouponUsageView` read models; it adds no event, no write, and no new invariant (the cap invariant is wholly owned by the existing "Redeem a coupon under a global cap at checkout" requirement). The W2/W3 UI binding is journey behavior [Narrative 011](../../../docs/narratives/011-customer-redeems-coupon.md) carries, not a new SHALL.

## Impact

- **Orders — new.** `Features/ValidateCoupon.cs`: a read-only `GET /coupons/{code}/validate` endpoint (Wolverine.Http, `IQuerySession`) returning a `CouponValidation { code, status, discountPercent? }` response. No write, no new event, no aggregate.
- **Orders — unchanged.** No `Program.cs` change (both read models already registered inline by #144); no schema change; no new contract or broker topology.
- **Storefront SPA — modified.** `client/src/cart/CartPage.tsx` (coupon field + preview + inline errors), `client/src/cart/couponQueries.ts` + `couponSchema.ts` (new — the advisory query's TanStack layer + Zod boundary parse of the discriminated response), `client/src/orders/placeOrderMutation.ts` (carry the optional `couponCode` as `?couponCode=`), `client/src/orders/orderSchema.ts` (add `subtotal`/`discount`/`couponCode` to `OrderStatusViewSchema`), `client/src/orders/OrderConfirmationPage.tsx` (discount line).
- **Tidy.** Comment-only edits to `client/src/cart/{cartMutations,cartQueries,CartPage}.ts(x)`, `client/src/catalog/catalogQueries.ts`, `client/src/orders/{orderQueries,placeOrderMutation,MyOrdersPage}.ts(x)` + affected test descriptions.
