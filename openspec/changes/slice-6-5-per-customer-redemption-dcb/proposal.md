## Why

The first DCB shipped ([PR #144](../../../docs/retrospectives/implementations/039-slices-6-1-6-3-6-4-coupon-dcb.md), slices 6.1/6.3/6.4; advisory UI 6.2, PR #146): a coupon enforces a **global per-coupon redemption cap** through a single-tag Marten DCB (`CouponUsage` folded from every event tagged with a strong-typed `CouponId`). [ADR 024 §38](../../../docs/decisions/024-dcb-coupon-redemption-in-orders.md) and [Workshop 003 §8 item 6](../../../docs/workshops/003-promotions-event-model.md) both name the natural next variant: **one redemption per customer** — the "more dynamic boundary," a composite `(coupon × customer)` tag. This change builds it as CritterMart's **second DCB** and its first **composite-tag** boundary.

The owner settled the one genuine modeling fork at session start (`AskUserQuestion` with previews): per-customer is an **opt-in policy on the definition**, not a universal law and not a distinct coupon kind. `CouponDefined` gains a `oneRedemptionPerCustomer` bit — **configuration-as-events extended**: the same event that already carries the cap now carries the per-customer policy, and checkout picks the boundary shape from it. This preserves FLASH20's pure global-cap demo (per-customer defaults off), keeps the two invariants cleanly separable, still teaches the composite tag, and subsumes a standalone once-per-customer coupon as the special case *(per-customer on + a high cap)*.

**Verified before modeling (handoff-flagged risk, resolved).** The resolved `JasperFx.Events` v2.27.0.0 (Marten 9.15.1) `EventTagQuery` exposes `For<T>` / `Or<T>` / `Or<TEvent,T>` / `AndEventsOfType<…>` but **no two-different-tag `.And<>()` conjunction** (`AndEventsOfType` filters event *types*, not tags). So the composite is realized as a **single strong-typed tag whose value encodes the pair** — `CouponCustomerTag("{couponId}|{customerId}")`, structurally identical to the shipped `CouponId(string Value)` and queried through the same `.Or<T>()` path the first DCB already proved. No new Marten API, no spike, no version bump (the pin holds — ADR 024, [[critterwatch-wolverine-version-coupling]]).

## What Changes

Everything executes in the **Orders** bounded context; no new service, no new contract, no broker topology. The Orders store opts a **second tag type** into its DCB schema (`mt_event_tag_couponcustomer` + GIN on `mt_events`) — the same additive, opt-in shape the first tag took.

- **Definition gains a per-customer policy (configuration-as-events).** `DefineCoupon { code, discountPercent, cap, oneRedemptionPerCustomer? }` → `CouponDefined { …, oneRedemptionPerCustomer }`; `CouponView` resolves it. Default `false` — every existing definition, seed, and test is unchanged, and old serialized `CouponDefined` events fold as `false`.
- **Checkout composes a second boundary when the flag is set.** When the resolved definition has `oneRedemptionPerCustomer = true`, `PlaceOrder.RedeemWithDcbAsync` ALSO opens the composite boundary `FetchForWritingByTags<CustomerCouponUsage>(new EventTagQuery().Or<CouponCustomerTag>(pair))` alongside the existing global-cap boundary, and — while that boundary's net count for the `(coupon, customer)` pair is `≥ 1` — rejects the placement with `409 CouponAlreadyRedeemedByCustomer`, creating no order stream. On success the tagged `CouponRedeemed` carries **both** tags (`CouponId` + `CouponCustomerTag`), arming both boundaries' assertions in the one `SaveChangesAsync`; the existing reload-and-retry loop covers a customer's self-race (double-submit) as well as the global-cap race.
- **Release returns the customer's slot too.** For a per-customer redemption, the compensating `CouponRedemptionReleased` carries the composite tag as well, so a cancelled per-customer order lets that customer redeem again — the reserve/release symmetry, now decrementing both boundaries.
- **Advisory preview unchanged (still checkout-authoritative-only).** `GET /coupons/{code}/validate` stays global-cap-only and anonymous; it does **not** preview the per-customer verdict (it holds no customer identity, and the advisory-vs-authoritative split means checkout is the sole authority — Workshop 003 §3). A per-customer preview is a parked follow-on.

## Capabilities

### Modified Capabilities

- `coupon-promotion`: **Define a coupon** gains the `oneRedemptionPerCustomer` policy field (configuration-as-events). **Release a redemption when its order is cancelled** gains the per-customer-boundary decrement for per-customer coupons. One requirement is **ADDED** — *Enforce one redemption per customer for a per-customer coupon at checkout* — the composite `(coupon × customer)` DCB, the `409 CouponAlreadyRedeemedByCustomer` rejection, the self-race resolving to exactly one, and the released-slot reuse. The global per-coupon cap requirement is unchanged; a non-per-customer coupon behaves exactly as before.
