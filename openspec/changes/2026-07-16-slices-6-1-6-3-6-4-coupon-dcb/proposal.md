## Why

CritterMart's coordination story shows cascading messages, Process Manager via Handlers, and a convention saga — but never a consistency boundary that spans many streams and aligns with **no aggregate**. [ADR 024](../../../docs/decisions/024-dcb-coupon-redemption-in-orders.md) chose to close that gap with a promotions increment built on Marten's **Dynamic Consistency Boundary (DCB)**, and [Workshop 003](../../../docs/workshops/003-promotions-event-model.md) modeled it: coupon **definitions** as configuration-as-events, coupon **redemption** at checkout under a **global per-coupon redemption cap** enforced by DCB inside the Orders store, and **release** of a redemption when the order cancels. This change implements the invariant-bearing core — slices **6.1 (define)**, **6.3 (redeem-with-DCB)**, and **6.4 (release-on-cancel)**. The advisory cart-review query and its storefront UI (slice 6.2, P1) trail in a following PR; the DCB demo — place with a flash coupon, drive the cap to breach, watch a race resolve to exactly one surviving order — stands on this change's API alone.

DCB is verified first-class in the resolved **Marten 9.15.1** (`FetchForWritingByTags`, `EventTagQuery().Or<CouponId>(id)`, `DcbConcurrencyException`, opt-in `EnableDcb()` schema) — ADR 024 verified 9.11.0; the transitively-resolved assembly has since moved to 9.15.1 through WolverineFx 6.19.0, and DCB only gets more first-class in later 9.x. No Polecat, no Wolverine/Marten version bump (the CritterWatch coupling holds — [ADR 017](../../../docs/decisions/017-critterwatch-integrated.md)).

## What Changes

Everything executes in the **Orders** bounded context (Promotions is a lane, not a deployed service — ADR 024). The Orders store **opts into** the DCB schema; no other service is touched.

- **6.1 — Define a coupon (Seller lane; configuration-as-events).** `DefineCoupon { code, discountPercent, cap }` starts a per-coupon stream with `CouponDefined`, projected inline into a `CouponView` (`code → { couponId, discountPercent, cap }`). A `POST /coupons` endpoint drives it so the seeder can realize the demo coupon set the same decoupled way it seeds products (real HTTP, no project reference). Validation rejects `cap < 1` or a discount outside `(0, 100]`; a partial-unique index on `CouponView.Code` is the code-uniqueness backstop (the open-cart precedent; a uniqueness DCB would be over-engineering for seed-issued definitions — Workshop 003 §8 item 3).
- **6.3 — Redeem a coupon at checkout under the cap (DCB).** `PlaceOrder` gains an **optional** `couponCode`. Absent → slice 4.1 behavior is byte-for-byte unchanged. Present → the handler resolves the definition via `CouponView`, opens the DCB boundary `FetchForWritingByTags<CouponUsage>(new EventTagQuery().Or<CouponId>(id))`, and if the net redemption count is below `cap` appends a **tagged** `CouponRedeemed` to the new order stream in the same transaction as `OrderPlaced` (now carrying `subtotal`, `discount`, `total`). Marten's tag-scoped concurrency throws `DcbConcurrencyException` at `SaveChangesAsync` on a losing race; the handler retries once into a fresh boundary read. The **cap-breach** path rejects with **`409 CouponExhausted` and creates no order stream**; an unresolved code rejects with `409 CouponInvalid`. `OrderStatusView` gains `subtotal` / `discount` / `couponCode` (additive, the `placedAt`/`cancelReason` precedent). `CouponUsageView` (advisory, inline; net count per coupon) is registered.
- **6.4 — Release a redemption on cancellation (system).** The `Order` aggregate folds `CouponRedeemed` (carrying `CouponId?`) so each of the three existing cancellation sites (stock-unavailable 4.5, payment-declined 4.6, payment-timeout 4.7) appends a compensating **tagged** `CouponRedemptionReleased` to the same stream **when — and only when — the stream holds a redemption**. The DCB boundary and `CouponUsageView` count redemptions **minus** releases, so a cancelled order returns its flash-sale slot. At-most-one-release is inherited free from Workshop 001's terminal-guard discipline (`OrderCancelled` is terminal and appended once).

### Open-question resolutions (Workshop 003 §8 items 1–5 — settled here)

- **#1 append mechanics:** the tagged `CouponRedeemed` rides the **new order stream** (`StartStream<Order>` in the same session), tagged with the strong-typed `CouponId` via `evt.WithTag(couponId)`; the `CouponUsage` DCB boundary provides the cap assertion at `SaveChangesAsync`. The **retry-on-`DcbConcurrencyException`** seats **handler-local** (one bounded retry into a fresh boundary read → breach rejection), not a Wolverine retry policy — the loser must re-decide against fresh state, not blindly replay. See `design.md`.
- **#2 `CouponUsageView` lifecycle:** **inline** (ADR 008-consistent). Decisive reason: no async daemon runs in this project, so an async advisory view would sit perpetually empty and could not serve the "no longer available" affordance it exists for. Settled with Erik at session start.
- **#3 code uniqueness:** partial-unique index on `CouponView.Code` (open-cart precedent); no uniqueness DCB this round.
- **#4 discount shape:** `discountPercent` only; a flat-amount variant is a payload-level widening the model already permits, deferred.
- **#5 demo coupon set:** the seeder defines `FLASH20` (20% off, **cap 3** — the hand-demonstrable race) plus `WELCOME10` (10% off, high cap) for everyday discounted-order traffic. demo-runbook + `demo-traffic.ps1` updated in the same session.

## Capabilities

### New Capabilities

- `coupon-promotion`: the Coupon-definition aggregate and the coupon-scoped rules of the increment — define a coupon (6.1), enforce the global redemption cap via DCB (6.3), release a redemption on cancellation (6.4), and track advisory per-coupon usage (`CouponUsageView`). One capability per aggregate: the Coupon (definition) stream is the aggregate; the DCB boundary and the tagged redemption/release events are its coupon-scoped invariant even though the events physically ride Order streams (the textbook DCB "boundary aligns with no aggregate" shape).

### Modified Capabilities

- `order-lifecycle`: the Order-stream integration deltas that host coupon redemption. `PlaceOrder` gains an optional `couponCode`; `OrderPlaced` and `OrderStatusView` gain `subtotal` / `discount` / `couponCode` (additive, wire shape preserved as a superset); the checkout transaction composes the tagged `CouponRedeemed` with `OrderPlaced`; and the three cancellation requirements (stock-unavailable, payment-declined, payment-timeout) each additionally append the tagged `CouponRedemptionReleased` when the order carries a redemption. Slice 4.1 and 4.5–4.7 behavior is unchanged when no coupon is present.

## Impact

- **Orders — new (`Ordering/`+`Promotions/`).** New `Promotions/` verb-feature folder: `DefineCoupon` command + `CouponDefined` event + `Coupon` aggregate (or `CouponView`-only) + `CouponView` + `CouponUsageView` + `CouponUsage` (DCB boundary aggregate) + `CouponId` strong-typed tag + `CouponRedeemed` / `CouponRedemptionReleased` events + `POST /coupons` endpoint (`Features/DefineCoupon.cs`). New `CouponExhausted` / `CouponInvalid` rejection shapes.
- **Orders — modified.** `Program.cs`: `opts.Events.EnableDcb()`, `TagEvent<…>`/`WithTag` wiring for the two redemption events, and inline registration of `CouponView` + `CouponUsageView` + `CouponUsage`; the `CouponView.Code` partial-unique index. `Features/PlaceOrder.cs`: optional `couponCode`, the DCB redemption path, priced `OrderPlaced`. `Ordering/Order.cs`, `Ordering/OrderPlaced.cs`, `Ordering/OrderStatusView.cs`: `subtotal`/`discount`/`couponCode` + the `CouponRedeemed`/`CouponRedemptionReleased` folds. `Ordering/StockReservationOutcomeHandlers.cs`, `Ordering/PaymentHandlers.cs`, `Ordering/PaymentTimeoutHandler.cs`: conditional release append at the three cancel sites.
- **Persistence / schema.** The Orders store's `orders.mt_events` gains the opt-in `tags TEXT[]` column + GIN index (`EnableDcb()`). New documents: `mt_doc_couponview`, `mt_doc_couponusageview`. No async daemon, no new stream type beyond the coupon-definition stream. No other service's store changes.
- **Contracts / topology.** **No** new `CritterMart.Contracts` types and **no** cross-BC message — redemption is intra-Orders (ADR 024). No new broker exchanges/queues.
- **Observability.** CritterWatch (trial-blocked, latent) gains the first DCB writes; the `marten.event.append` counter tags `CouponDefined`/`CouponRedeemed`/`CouponRedemptionReleased`.
- **Seeder / demo.** `CritterMart.Seeding` defines the demo coupon set via `POST /coupons`; `docs/demo-runbook.md` gains a coupon-redemption flow (place → discount → drive cap to breach → race → cancel-returns-slot); `docs/demo-traffic.ps1` optionally applies `WELCOME10`.
- **Tests.** Unit: `Order`/`OrderStatusView` coupon folds, the `CouponUsage` boundary fold, `CouponView` fold. Integration (Alba/Marten): 6.1 happy/duplicate/nonsensical; 6.3 happy/cap-breach/concurrent-race/unknown-code/no-coupon; 6.4 release/reused-slot/no-coupon-cancel.
